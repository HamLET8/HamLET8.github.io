SSD / FTL 代码设计学习路径

15 天计划表：从模块边界走到演进路线

这张计划表独立于每日网页报告。每天围绕一个 SSD / FTL 设计判断展开， 四本书作为判断来源分布在完整路径中，目标是逐步覆盖模块边界、领域语言、接口设计、 错误路径、状态机、恢复机制、GC 协作、缓存边界、介质健康与长期演进。

独立计划文件 每日一个 HTML 报告 路径：plan.html + day-01.html ... day-15.html

阶段 1：切边界

Day 1-5 建立模块边界、领域语言、接口深度、Host Write 骨架和 Mapping 所有权。

阶段 2：稳一致性

Day 6-10 处理错误路径、状态机、Journal、Recovery 和 GC 协作边界。

阶段 3：可演进

Day 11-15 进入策略分离、WriteBooster、Read Retry/ECC、Bad Block 和长期路线。

学习路径图

15 天计划表

Day 明确主题 SSD / FTL 设计判断 工程面向 四本书思想分布
Day 1 模块边界先于函数实现 FTL Core 只编排领域动作，不直接理解 NAND 状态位、block table 内部布局或 SLC cache 私有状态。 模块边界、信息隐藏 复杂度隔离、正交性、概念完整性、限界上下文
Day 2 建立 FTL 通用语言 用 LBA、PPA、open block、victim block、mapping commit、relocation 等词汇替代 type、mode、flag。 领域语言、命名、模型一致性 DDD 主导，辅以深模块和 DRY
Day 3 接口深度：把参数列表变成领域意图 Block Manager 应提供按写入意图分配 page 的接口，而不是暴露万能 alloc 函数。 接口设计、封装深度 《软件设计哲学》主导，结合 Pragmatic 正交性
Day 4 Host Write 曳光弹 先打通一条窄但真实的 Host Write 路径：分配、program、journal、mapping update、旧页失效。 主路径、架构骨架 Tracer Bullet、设计探索、领域事件
Day 5 Mapping Table 的所有权 Mapping 模块拥有 L2P/P2L 一致性；GC、Recovery、WriteBooster 只能通过条件更新接口协作。 数据所有权、一致性边界 DRY、聚合边界、复杂度局部化
Day 6 错误路径不是 return 语句 NAND program/read/erase 结果必须先映射为 FTL 决策，再进入 Block、Mapping、Journal 的补偿路径。 错误码、错误矩阵、补偿动作 防软件熵、深模块、概念取舍、领域异常
Day 7 Block 与 Page 生命周期状态机 block/page 状态必须由状态机驱动，禁止在各模块重复判断 valid/free/bad/open。 状态机、不变量 概念完整性、DRY、领域状态建模
Day 8 Metadata Journal 的最小可恢复记录 Journal 记录领域变更意图，而不是转储内存结构；每条记录都应可重放、可幂等、可校验。 日志设计、恢复语义 可逆性、信息隐藏、设计约束、领域事件
Day 9 Power Loss Recovery 的阶段边界 Recovery 应分为扫描、重放、校验、接管四阶段，避免 mount 路径混入正常写路径细节。 恢复机制、启动流程 设计演进、契约、防腐层、正交性
Day 10 GC 是领域服务，不是后台线程 GC 编排空间回收流程；victim 选择、有效页判断、搬移提交、擦除归还分别由对应模块负责。 GC 协作边界 领域服务、深模块、概念完整性
Day 11 Victim Selection 与 Wear Leveling 策略分离 选择策略输出 score 和理由；GC 流程不直接散落 erase count、valid count、hot/cold 数据判断。 策略接口、可演进算法 设计取舍、正交性、领域策略
Day 12 SLC Cache / WriteBooster 的边界 WriteBooster 是写入加速上下文，不应让主 FTL 写路径理解 SLC/TLC 迁移细节。 缓存层、flush 协议、QoS 信息隐藏、原型演进、上下文映射
Day 13 Read Retry、ECC 与读路径决策 ECC/Read Retry 应返回数据可信度和恢复动作建议，而不是把 PHY 细节泄漏给 Mapping 或 Host Read。 读路径、数据完整性 接口契约、错误路径、领域语义
Day 14 Bad Block Management 的隔离边界 坏块隔离、替换、退役、统计应归 Bad Block Management；其他模块只消费可用性决策。 介质健康、隔离策略 防腐层、概念完整性、软件熵控制
Day 15 演进路线：从能跑到能长期维护 用指标、兼容记录格式、模块契约测试和阶段性重构路线，支撑 FTL 从单路径走向产品级。 演进路线、测试、兼容性 设计作为过程、务实反馈、领域模型迭代
