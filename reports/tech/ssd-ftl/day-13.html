Day 13 · SSD / FTL 代码设计判断

Read Retry、ECC 与读路径决策：返回数据可信度和恢复动作建议

今天的核心判断是：ECC / Read Retry 应返回数据可信度和恢复动作建议，而不是把 PHY 细节泄漏给 Mapping 或 Host Read。 读路径不只是“读成功或失败”，它还要判断数据是否可信、是否需要 read reclaim、是否要触发介质风险标记。

一句话判断 介质读结果先转成 FTL 决策，再返回给上层 涉及：Read Path / ECC / Read Retry / Mapping / GC / Bad Block 关键词：data confidence · corrected bits · read reclaim · retry profile · media risk

今日设计判断

Host Read 不应该理解 retry voltage、ECC syndrome、read threshold。它只应收到： 数据是否可信、是否已纠错、是否建议 read reclaim、是否应向介质健康模块报告风险。

今日涉及模块

Read Service、NAND Read Flow、ECC、Read Retry、Mapping Table、Read Reclaim、GC、 Bad Block Management、Media Health、Power Loss Recovery。

今日阅读重点：读结果不是硬件状态码

NAND read 可能 clean、corrected、near-limit、retry recovered、uncorrectable。 这些状态对 Host Read、Mapping、GC 和 Bad Block Management 的含义不同。 设计目标不是把所有 PHY 细节向上传递，而是把介质证据转成稳定的 FTL 决策。

思想来源 今天必须掌握的概念 落到 SSD / FTL 的判断
《软件设计哲学》 深模块应隐藏低层复杂度。ECC syndrome 和 retry profile 是 NAND Read Service 内部知识。 上层接口暴露 read_confidence 与 recovery_action，不暴露 retry voltage 表。
《程序员修炼之道》 契约要明确，失败要可诊断。读路径需要区分可纠错成功、勉强恢复、不可恢复。 使用负 errno + 结构化 evidence，避免所有读问题都变成 -EIO。
《设计的设计》 取舍要服务系统目标。读成功但纠错接近上限时，继续返回数据和立即搬迁是不同设计选择。 通过 action matrix 明确什么时候返回数据、什么时候同步 read reclaim、什么时候后台 scrub。
《领域驱动设计》 领域语言把硬件现象转成业务事实。读结果应表达数据可信度和介质健康，不是寄存器位。 使用 trusted、corrected、retry-recovered、suspect、uncorrectable、media-risk 等领域词汇。

概念融合解释：把“能读出来”拆成可信度和动作

Read Path 的复杂点在于，NAND 返回的数据不是二元状态。ECC correctable 意味着数据可返回，但也可能说明介质正在退化； Read Retry recovered 意味着这次可以读出，但应尽快 read reclaim；Uncorrectable 则要进入副本恢复、RAID/parity 或错误上报。 如果 Host Read 直接处理这些细节，Mapping 和 Bad Block Management 会被迫理解 PHY 语义。

统一判断： Read Service 应作为 NAND/ECC 与 FTL 之间的防腐层。它接收低层 evidence，输出数据可信度、恢复动作、 介质风险信号和上层 errno。这样 Host Read 保持简单，GC 和 Bad Block 也能消费一致的健康语义。

SSD / FTL 设计场景

ECC Correctable

数据可返回，但 corrected bits 进入健康统计；接近阈值时返回 read reclaim 建议。

Read Retry Recovered

通过 retry profile 读出数据时，数据可用但不应长期停留原 PPA；应提交后台迁移或 scrub。

Uncorrectable Read

不可纠错不应直接污染 Mapping；Read Service 先尝试 mirror、parity、journal replay 或返回 -EILSEQ。

Read After Write

Mapping 返回当前 PPA 后，Read Service 只对该 PPA 做介质判定；不在读路径里重新解释 LBA 版本。

Read Reclaim

读路径输出 reclaim action，GC 或 dedicated reclaim worker 负责搬迁；Host Read 不直接写新 page。

Bad Block Risk

多次 retry recovered 或 corrected bits 快速上升时，Media Health 消费风险信号，而不是让 ECC 模块直接 retire block。

反例：读路径泄漏 ECC / Read Retry 细节 C 
int ftl_read_bad(struct ftl_dev *ftl, u64 lba, void *buf, u32 len)
{
	struct ftl_ppa ppa;
	int corrected_bits;
	int retry_level;
	int ret;

	ppa = ftl->l2p[lba];

	ret = nand_read_page(ftl->nand, &ppa, buf, len, &corrected_bits);
	if (!ret) {
		if (corrected_bits > 80) {
			ftl->blocks[ppa.block].weak_reads++;
			gc_move_page_now(ftl, lba, &ppa);
		}
		return 0;
	}

	for (retry_level = 0; retry_level < 7; retry_level++) {
		nand_set_read_retry_voltage(ftl->nand, retry_level);
		ret = nand_read_page(ftl->nand, &ppa, buf, len,
				     &corrected_bits);
		if (!ret) {
			if (retry_level > 3)
				ftl->blocks[ppa.block].bad_candidate = true;
			gc_move_page_now(ftl, lba, &ppa);
			return 0;
		}
	}

	ftl->blocks[ppa.block].read_failures++;
	if (ftl->blocks[ppa.block].read_failures > 4)
		ftl->blocks[ppa.block].bad = true;

	return -EIO;
}
设计问题 代码表现 后果
PHY 细节泄漏 Host Read 调整 retry voltage。 上层读路径被 NAND 参数绑定，后续换 NAND profile 会影响 FTL Core。
错误语义模糊 最终统一返回 -EIO。 调用者无法区分 uncorrectable、retry exhausted、media risk、暂时性读失败。
职责跨界 读路径直接调用 gc_move_page_now()。 Host Read 被前台搬迁延迟污染，也绕过 relocation journal。
坏块误判 ECC 模块间接把 block 标成 bad。 Bad Block Management 的隔离策略和统计口径失效。

改进代码：读服务返回可信度和动作建议

改进版把 NAND/ECC evidence 先转换为 FTL read decision。Host Read 只处理数据和 errno； Read Reclaim、Media Health、Bad Block Management 通过结构化 action 消费介质信号。

read_service.h：读路径领域接口 C 
#ifndef FTL_READ_SERVICE_H
#define FTL_READ_SERVICE_H

#include <linux/types.h>
#include <linux/errno.h>

enum read_confidence {
	READ_CONFIDENCE_TRUSTED = 0,
	READ_CONFIDENCE_CORRECTED,
	READ_CONFIDENCE_RETRY_RECOVERED,
	READ_CONFIDENCE_SUSPECT,
	READ_CONFIDENCE_UNCORRECTABLE,
};

enum read_recovery_action {
	READ_ACTION_NONE = 0,
	READ_ACTION_RECORD_HEALTH,
	READ_ACTION_READ_RECLAIM,
	READ_ACTION_TRY_REDUNDANT_COPY,
	READ_ACTION_REPORT_UNCORRECTABLE,
};

struct nand_read_evidence {
	struct ftl_ppa ppa;
	int nand_status;
	u32 corrected_bits;
	u32 retry_count;
	bool retry_recovered;
	bool uncorrectable;
};

struct ftl_read_decision {
	int errno_value;
	enum read_confidence confidence;
	enum read_recovery_action action;
	u32 health_score;
	const char *reason;
};

	void *buf;
	u32 len;
};

	struct ftl_ppa ppa;
	enum read_confidence confidence;

};

read_service.c：先分类，再执行领域动作 C 
void read_classify_evidence(const struct nand_read_evidence *e,
			    struct ftl_read_decision *d)
{
	memset(d, 0, sizeof(*d));

	if (e->uncorrectable) {
		d->errno_value = -EILSEQ;
		d->confidence = READ_CONFIDENCE_UNCORRECTABLE;
		d->action = READ_ACTION_TRY_REDUNDANT_COPY;
		d->reason = "uncorrectable";
		return;
	}

	if (e->retry_recovered) {
		d->confidence = READ_CONFIDENCE_RETRY_RECOVERED;
		d->action = READ_ACTION_READ_RECLAIM;
		d->health_score = e->retry_count * 16 + e->corrected_bits;
		d->reason = "retry recovered";
		return;
	}

	if (e->corrected_bits > READ_RECLAIM_ECC_THRESHOLD) {
		d->confidence = READ_CONFIDENCE_SUSPECT;
		d->action = READ_ACTION_READ_RECLAIM;
		d->health_score = e->corrected_bits;
		d->reason = "near ecc limit";
		return;
	}

	if (e->corrected_bits) {
		d->confidence = READ_CONFIDENCE_CORRECTED;
		d->action = READ_ACTION_RECORD_HEALTH;
		d->health_score = e->corrected_bits;
		d->reason = "ecc corrected";
		return;
	}

	d->confidence = READ_CONFIDENCE_TRUSTED;
	d->action = READ_ACTION_NONE;
	d->reason = "clean read";
}

int ftl_read_submit(struct read_service *svc, const struct ftl_read_req *req,
		    struct ftl_read_result *result)
{
	struct nand_read_evidence evidence;
	struct ftl_read_decision decision;
	struct ftl_ppa ppa;
	int ret;

	ret = map_lookup_current_ppa(svc->map, req->lba, &ppa);
	if (ret)
		return ret;

	ret = nand_read_with_ecc_retry(svc->nand, &ppa, req->buf, req->len,
				       &evidence);
	if (ret && ret != -EILSEQ)
		return ret;

	read_classify_evidence(&evidence, &decision);

	switch (decision.action) {
	case READ_ACTION_NONE:
		break;
	case READ_ACTION_RECORD_HEALTH:
		media_health_record_read(svc->health, &ppa, &decision);
		break;
	case READ_ACTION_READ_RECLAIM:
		media_health_record_read(svc->health, &ppa, &decision);
		read_reclaim_queue(svc->reclaim, req->lba, &ppa);
		result->reclaim_queued = true;
		break;
	case READ_ACTION_TRY_REDUNDANT_COPY:
		ret = read_recover_from_redundancy(svc, req, &ppa);
		if (ret)
			return -EILSEQ;
		decision.errno_value = 0;
		decision.confidence = READ_CONFIDENCE_CORRECTED;
		break;
	case READ_ACTION_REPORT_UNCORRECTABLE:
		return -EILSEQ;
	}

	result->ppa = ppa;
	result->confidence = decision.confidence;
	return decision.errno_value;
}

决策表与思想落点表

ECC / Read Retry 到 FTL 决策矩阵

低层结果 FTL 可信度 返回给 Host Read 后续动作 所有者
Clean read TRUSTED 返回数据。 无。 Read Service
ECC corrected bits below threshold CORRECTED 返回数据和 corrected 标记。 记录健康统计。 ECC / Media Health
ECC near limit SUSPECT 返回数据，但标记风险。 排队 read reclaim。 Read Reclaim / GC
Read Retry recovered RETRY_RECOVERED 返回数据，但提高风险权重。 尽快搬迁，更新介质风险。 Read Service / Media Health
Uncorrectable UNCORRECTABLE 尝试冗余恢复；失败返回 -EILSEQ。 镜像、parity、RAID、坏块风险上报。 Recovery / Bad Block

读路径责任表

模块 负责什么 不负责什么 接口输出
Mapping Table 根据 LBA 返回当前 PPA。 不解释 ECC、retry、介质健康。 current PPA、mapping seq。
NAND / ECC 执行 read、ECC decode、retry profile。 不决定 Mapping 是否更新，不直接 retire block。 nand_read_evidence。
Read Service 把 evidence 转成可信度和动作建议。 不执行 GC 策略，不维护坏块表。 ftl_read_decision。
Read Reclaim 搬迁 suspect / retry recovered 数据。 不重新分类 ECC 结果。 relocation request。
Bad Block Management 消费风险信号，决定隔离、观察或退役。 不被单次 read retry 直接驱动退役。 media availability decision。
