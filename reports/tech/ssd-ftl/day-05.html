Day 5 · SSD / FTL 代码设计判断

Mapping Table 的所有权：所有 L2P/P2L 一致性必须经过它

今天的核心判断是：Mapping 模块拥有 LBA 到 PPA 的当前事实。 GC、Recovery、WriteBooster、Host Write 都可以提出变更意图，但不能直接改 L2P/P2L。 它们必须通过条件提交接口，让 Mapping 保持一致性、不变量和恢复边界。

一句话判断 Mapping 拥有地址关系事实 涉及：Mapping / GC / Recovery / Journal / WriteBooster 关键词：所有权 · 条件提交 · L2P/P2L 一致性 · 防腐边界

今日设计判断

任何模块都不应直接写 l2p[lba]。 Mapping Table 提供 lookup、reverse lookup、prepare delta、commit delta if current、replay commit 等接口。

今日涉及模块

Mapping Table、GC Relocation、Metadata Journal、FTL Recovery、WriteBooster Flush、 Host Write、Victim Block Selection、Power Loss Recovery。

今日阅读重点：数据所有权比访问便利更重要

FTL 里最容易被破坏的数据边界是 Mapping。因为所有路径都“需要更新映射”，所以最诱人的做法就是让每个模块都能写 L2P。 这会迅速摧毁恢复语义和 GC 正确性。

思想来源 今天必须掌握的概念 落到 SSD / FTL 的判断
《软件设计哲学》 深模块隐藏关键数据结构，调用者不应依赖内部布局。 Mapping 内部的 cache、page-level table、translation page、dirty bitmap 不向 GC 和 Recovery 暴露。
《程序员修炼之道》 DRY 约束知识重复。当前映射事实只能有一个权威来源。 L2P/P2L 当前关系由 Mapping 维护，Journal 记录变更，Recovery 重放变更但不拥有当前事实。
《设计的设计》 概念完整性要求把“地址关系”作为稳定主概念，而不是散落到每条路径。 Host Write、GC relocation、SLC flush 都提交 mapping delta，而不是各自实现映射更新。
《领域驱动设计》 聚合边界保护不变量。Mapping 是地址关系聚合，外部模块通过领域服务协作。 GC 可以询问 old PPA 是否仍当前有效，但不能绕过 Mapping 直接改 L2P。

概念融合解释：Mapping 是事实边界，不是数组工具

Mapping Table 不是一个方便大家读写的数组。它是 FTL 中“逻辑地址当前指向哪里”的事实边界。 一旦 GC、Recovery、WriteBooster 都能直接写 L2P，系统会出现多个事实来源：GC 认为搬移成功， Recovery 认为 journal 尚未提交，Host Write 认为用户写入覆盖了旧页。

统一判断： Mapping 模块应该接受“条件变更”，而不是暴露“直接赋值”。关键接口不是 set_l2p(lba, ppa)，而是 map_commit_delta_if_current()。

SSD / FTL 设计场景

Host Write

Host Write 通过 mapping delta 提交 old PPA 到 new PPA 的变化，Mapping 负责更新 L2P/P2L 和 dirty 状态。

GC Relocation

GC 必须使用条件提交：只有 old PPA 仍是当前映射时，才允许搬移结果成为 new PPA。

WriteBooster Flush

SLC 到 TLC 的 flush 不是普通覆盖；它必须确认 LBA 尚未被更新写覆盖，避免 stale flush 回滚用户数据。

Metadata Journal

Journal 记录 mapping delta，不拥有当前 mapping；Recovery 重放时仍通过 Mapping 的 replay 接口提交。

Power Loss Recovery

Recovery 不直接修 L2P 数组，而是扫描 journal 和 NAND metadata，然后调用 Mapping 的幂等 replay。

Victim Block Selection

GC 需要知道 PPA 是否仍有效，应通过 reverse lookup 或 iterator，而不是读取 Mapping 内部表结构。

反例代码：每个模块都能写 L2P

坏味道： GC、Recovery、WriteBooster 都直接改 L2P。短期少写接口，长期会出现 stale relocation、flush 覆盖新写入、replay 非幂等。
反例：Mapping 所有权被多个模块破坏 C 
int gc_move_valid_page(struct ftl_dev *ftl, u64 lba,
		       int old_ppn, int new_ppn)
{
	nand_copy_page(ftl->nand, old_ppn, new_ppn);
	ftl->l2p[lba] = new_ppn;
	ftl->p2l[new_ppn] = lba;
	ftl->p2l[old_ppn] = P2L_INVALID;
	return 0;
}

int wb_flush_entry(struct ftl_dev *ftl, u64 lba, int slc_ppn,
		   int tlc_ppn)
{
	nand_copy_page(ftl->nand, slc_ppn, tlc_ppn);
	ftl->l2p[lba] = tlc_ppn;
	ftl->p2l[tlc_ppn] = lba;
	return 0;
}

int recovery_replay_record(struct ftl_dev *ftl, struct raw_record *r)
{
	if (r->type == REC_MAPPING) {
		ftl->l2p[r->lba] = r->new_ppn;
		ftl->p2l[r->new_ppn] = r->lba;
	}

	return 0;
}
问题 代码表现 后果
所有权分裂 三个模块都直接写 l2p 和 p2l。 Mapping 无法维护 L2P/P2L 互逆不变量。
缺少条件提交 GC 和 flush 不检查 old PPA 是否仍当前。 旧搬移结果可能覆盖更新的 host write。
Recovery 非幂等 replay 直接赋值，没有版本、原因和 old PPA 校验。 重复 replay 或部分 replay 后状态不可解释。
统计失真 旧 PPA 的失效和 valid count 更新不在同一提交边界。 GC victim selection 依赖的统计会漂移。

改进代码：Mapping 提供条件提交接口

改进版的关键是把“当前事实”和“变更请求”分开。外部模块提交 delta； Mapping 模块决定这个 delta 是否仍然有效、如何更新 L2P/P2L、如何标记 dirty 和统计。

mapping.h：Mapping 拥有提交语义 C 
#ifndef FTL_MAPPING_H
#define FTL_MAPPING_H

#include <linux/types.h>
#include <linux/errno.h>

enum map_commit_reason {
	MAP_COMMIT_HOST_WRITE = 0,
	MAP_COMMIT_GC_RELOCATION,
	MAP_COMMIT_SLC_FLUSH,
	MAP_COMMIT_RECOVERY_REPLAY,
};

struct map_delta {
	u64 lba;
	struct ftl_ppa old_ppa;
	struct ftl_ppa new_ppa;
	enum map_commit_reason reason;
	u64 seq;
};

int map_lookup_l2p(struct map_table *map, u64 lba,
		   struct ftl_ppa *ppa);
int map_reverse_lookup(struct map_table *map, const struct ftl_ppa *ppa,
		       u64 *lba);
int map_commit_delta_if_current(struct map_table *map,
				const struct map_delta *delta);
int map_replay_delta(struct map_table *map,
		     const struct map_delta *delta);

#endif
mapping.c：条件提交保护当前事实 C 
int map_commit_delta_if_current(struct map_table *map,
				const struct map_delta *delta)
{
	struct ftl_ppa current;
	int ret;

	if (!map || !delta || !ftl_ppa_valid(&delta->new_ppa))
		return -EINVAL;

	ret = map_lookup_l2p(map, delta->lba, &current);
	if (ret && ret != -ENOENT)
		return ret;

	if (ret == -ENOENT && ftl_ppa_valid(&delta->old_ppa))
		return -ESTALE;

	if (!ret && !ftl_ppa_equal(&current, &delta->old_ppa))
		return -ESTALE;

	map_set_l2p_internal(map, delta->lba, &delta->new_ppa);
	map_set_p2l_internal(map, &delta->new_ppa, delta->lba);

	if (ftl_ppa_valid(&delta->old_ppa))
		map_clear_p2l_internal(map, &delta->old_ppa);

	map_mark_dirty(map, delta->lba);
	return 0;
}
gc.c：GC 只提交 relocation delta C 
int gc_commit_relocation(struct ftl_dev *ftl, u64 lba,
			 const struct ftl_ppa *old_ppa,
			 const struct ftl_ppa *new_ppa)
{
	struct map_delta delta = {
		.lba = lba,
		.old_ppa = *old_ppa,
		.new_ppa = *new_ppa,
		.reason = MAP_COMMIT_GC_RELOCATION,
		.seq = journal_next_seq(ftl->journal),
	};
	int ret;

	ret = journal_append_map_delta(ftl->journal, &delta);
	if (ret)
		return ret;

	ret = map_commit_delta_if_current(ftl->map, &delta);
	if (ret == -ESTALE)
		return 0;

	return ret;
}

责任与落点表

Mapping 协作边界表

调用方 允许动作 禁止动作 Mapping 返回语义
Host Write 提交 host write delta 直接写 L2P/P2L 成功、旧映射不存在、内部错误
GC Relocation 条件提交 relocation delta 覆盖当前 LBA 映射 -ESTALE 表示跳过 stale page
WriteBooster Flush 条件提交 SLC flush delta 把 flush 结果强制设为当前 -ESTALE 表示 host 已更新
Recovery 幂等 replay journal delta 绕过 Mapping 重建数组 重复 replay 应返回成功或已应用
Victim Selection 查询 PPA 当前有效性 读取 Mapping 内部缓存布局 valid、stale、unknown

四本书思想落点表

书 思想 SSD / FTL 设计位置 约束了什么 帮助避免什么问题
《软件设计哲学》 信息隐藏：深模块保护内部数据结构。 Mapping Table 内部 L2P/P2L/cache。 外部不得依赖数组布局。 避免 Mapping 结构变化牵动 GC/Recovery。
《程序员修炼之道》 DRY：当前映射事实只能有一个权威来源。 Mapping commit 和 reverse lookup。 L2P/P2L 更新规则集中在 Mapping。 避免多模块重复维护地址关系。
《设计的设计》 概念完整性：地址关系是独立主概念。 map_delta、conditional commit、replay。 所有变更都表达为同一种可审查模型。 避免 Host/GC/Recovery 各走一套更新语义。
《领域驱动设计》 聚合边界维护不变量。 Mapping 聚合与 GC/Recovery 领域服务协作。 外部只能提交领域变更，不能改内部状态。 避免跨上下文污染和 stale update。
