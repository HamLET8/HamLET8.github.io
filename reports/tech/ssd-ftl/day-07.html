Day 7 · SSD / FTL 代码设计判断

Block 与 Page 生命周期状态机：不要让 valid/free/bad/open 变成散落的判断

今天的核心判断是：block/page 状态必须由状态机驱动，禁止 GC、Mapping、Recovery、Write Path 各自重复判断 valid、free、 bad、open。状态不是一组 bool， 而是 FTL 对介质生命周期、写入事务和恢复边界的统一语言。

一句话判断 状态只能通过 Block Manager 的领域事件转换 涉及：Block Manager / Free Block Pool / GC / Bad Block / Recovery 关键词：状态机 · 不变量 · DRY · 聚合边界 · 恢复语义

今日设计判断

不要让调用者读写 is_free、 is_open、valid_pages 这类内部事实。调用者应该提交领域事件：open block、allocate page、commit page、 invalidate page、select victim、finish erase。

今日涉及模块

Block Manager、Free Block Pool、Victim Block Selection、Wear Leveling、 Bad Block Management、NAND Program / Erase / Read Flow、Metadata Journal、Power Loss Recovery。

今日阅读重点：状态机是复杂度压缩器

Block/Page 生命周期看起来只是“几个标志位”，但在 FTL 中它承载了空间管理、写入提交、GC 搬迁、 坏块隔离和掉电恢复的共同事实。今天需要掌握的不是状态机语法，而是为什么状态机能把散落知识收回到一个模块里。

思想来源 今天必须掌握的概念 如果理解不足，会在 FTL 中变成什么问题
《软件设计哲学》 复杂度来自依赖和晦涩。状态判断如果散在调用点，就会把内部不变量暴露给每个模块。 GC、Recovery、Write Path 都要理解 block 是否 open、page 是否 committed、valid count 是否可信，修改一个状态含义会放大到全系统。
《程序员修炼之道》 DRY 不是不重复代码，而是不重复知识。valid/free/bad/open 的合法组合是一种关键知识。 多个模块各自写 if (!bad && free)，最终出现“看起来 free 但不允许分配”的幽灵状态。
《设计的设计》 设计取舍要保持概念完整性。少量清晰状态比大量局部标志更能表达设计意图。 为了快速修 bug 增加 need_gc、was_open、half_bad，系统逐渐失去统一模型。
《领域驱动设计》 状态变化应是领域事件。Block Manager 是 block/page 生命周期的聚合边界。 高层 FTL 直接改 NAND metadata 字段，Recovery 只能猜“这个 page 到底是写入中、已提交还是孤儿页”。

概念融合解释：状态是领域事实，不是字段快照

四本书在这里形成一个共同判断：状态机不是为了形式化而形式化，而是为了让不变量有唯一归属。 《软件设计哲学》要求把复杂状态隐藏在深模块内部；《程序员修炼之道》要求不要重复状态知识； 《设计的设计》提醒我们用少量一致概念抵抗补丁式标志；《领域驱动设计》则把状态变化命名为领域事件， 让接口说出真实业务含义。

统一判断： Block Manager 不只是“存 block 信息的结构体集合”。它应该是生命周期状态机的所有者。 GC、Mapping、Recovery、WriteBooster 不能直接设置状态字段，只能提交事件并接收明确结果。 这样才能让模块边界、接口语义、错误路径和恢复机制共享同一个事实来源。

SSD / FTL 设计场景

Host Write

写路径只请求 block_alloc_page() 和 block_commit_page()，不判断 page 是否 free，也不直接增加 valid count。

GC Relocation

GC 只能把 CLOSED block 请求转为 VICTIM，并通过 relocation delta 提交新旧 PPA 关系，不能自己把 page 改成 invalid。

Bad Block Management

Program/erase fail 不是局部标志。Block Manager 根据错误决策把 OPEN 或 ERASING block 转到 RETIRED，并从 free pool 中移除。

Metadata Journal

Journal 失败后，PROGRAMMED page 不能直接回到 FREE，而应进入 ORPHAN，保证 Recovery 能解释“写了但没有提交”。

Power Loss Recovery

恢复逻辑读取状态和 journal，不重跑写路径。状态机提供合法收敛方向：COMMITTED 保留，ORPHAN 回收，ERASING 重新校验。

Free Block Pool

Free pool 只接受 FREE block。CLOSED、VICTIM、ERASING、RETIRED 不能因为计数看似可用而被重新分配。

反例代码：状态判断散落，合法组合没人负责

坏味道： 这段代码把 block/page 状态拆成多个 bool 和计数器。Write Path、GC、Recovery 都直接修改字段， 导致状态组合没有唯一解释，错误路径和恢复路径只能靠猜。
反例：多个模块直接读写状态字段 C 
struct ftl_page_bad {
	bool free;
	bool valid;
	bool programmed;
	bool journaled;
	u64 lba;
};

struct ftl_block_bad {
	bool free;
	bool open;
	bool victim;
	bool bad;
	int valid_pages;
	int free_pages;
	struct ftl_page_bad pages[256];
};

int ftl_write_page_bad(struct ftl_dev *ftl, u64 lba, const void *buf)
{
	struct ftl_block_bad *blk = ftl->open_blk;
	struct ftl_page_bad *page;
	int ret;

	if (!blk || blk->bad || !blk->open || blk->victim)
		return -ENOSPC;

	if (blk->free_pages == 0) {
		blk->open = false;
		blk->free = false;
		return -ENOSPC;
	}

	page = find_first_free_page(blk);
	page->free = false;
	page->lba = lba;

	ret = nand_program_page(ftl->nand, blk, page, buf);
	if (ret) {
		blk->bad = true;
		page->free = false;
		page->valid = false;
		return ret;
	}

	page->programmed = true;

	ret = journal_append_mapping(ftl->journal, lba, blk, page);
	if (ret)
		return ret;

	page->journaled = true;
	page->valid = true;
	blk->valid_pages++;
	blk->free_pages--;
	return map_update_l2p(ftl->map, lba, blk, page);
}

void gc_mark_victim_bad(struct ftl_block_bad *blk)
{
	if (!blk->bad && !blk->open && !blk->free)
		blk->victim = true;
}

void recovery_fix_page_bad(struct ftl_page_bad *page)
{
	if (page->programmed && !page->journaled) {
		page->free = true;
		page->valid = false;
	}
}
设计问题 代码表现 真实后果
状态组合无约束 free、open、victim、bad 可被任意组合。 可能出现 open 且 victim、bad 仍在 free pool、programmed 但 free 的非法状态。
模块边界泄漏 Write Path、GC、Recovery 都直接改 block/page 字段。 Block Manager 不再拥有生命周期事实，状态修复需要跨模块搜索。
错误路径不可恢复 journal 失败后直接返回，没有把 page 标为 ORPHAN。 掉电后 Recovery 不知道该 page 是旧数据、新数据还是未提交垃圾。
计数漂移 valid/free count 在多个路径手动调整。 GC victim selection、free block pressure、wear leveling 的输入不可信。

改进代码：Block Manager 拥有状态机

改进版把状态字段收回到 Block Manager 内部。外部模块不能写状态，只能提交领域事件。 接口返回负 errno，同时保留可恢复语义：非法转换是 -EINVAL， 并发或陈旧请求是 -ESTALE，介质失败进入退役路径。

#ifndef FTL_BLOCK_STATE_H
#define FTL_BLOCK_STATE_H

#include <linux/types.h>
#include <linux/errno.h>

enum ftl_block_state {
	FTL_BLOCK_FREE = 0,
	FTL_BLOCK_OPEN,
	FTL_BLOCK_CLOSED,
	FTL_BLOCK_VICTIM,
	FTL_BLOCK_ERASING,
	FTL_BLOCK_RETIRED,
};

enum ftl_page_state {
	FTL_PAGE_FREE = 0,
	FTL_PAGE_ALLOCATED,
	FTL_PAGE_PROGRAMMED,
	FTL_PAGE_COMMITTED,
	FTL_PAGE_INVALID,
	FTL_PAGE_ORPHAN,
};

enum ftl_block_event {
	FTL_BLOCK_EV_OPEN = 0,
	FTL_BLOCK_EV_CLOSE,

	FTL_BLOCK_EV_BEGIN_ERASE,
	FTL_BLOCK_EV_FINISH_ERASE,
	FTL_BLOCK_EV_RETIRE,
};

struct ftl_ppa {
	u32 ch;
	u32 lun;
	u32 block;
	u32 page;
};

struct ftl_page_commit {
	struct ftl_ppa ppa;
	u64 lba;
	u64 mapping_seq;
};

struct ftl_block_manager;

int block_open_free_block(struct ftl_block_manager *bm,

block_state.c：转换函数集中维护不变量 C 
static int block_next_state(enum ftl_block_state from,
			    enum ftl_block_event event,
			    enum ftl_block_state *to)
{
	switch (from) {
	case FTL_BLOCK_FREE:
		if (event == FTL_BLOCK_EV_OPEN) {
			*to = FTL_BLOCK_OPEN;
			return 0;
		}
		break;
	case FTL_BLOCK_OPEN:
		if (event == FTL_BLOCK_EV_CLOSE) {
			*to = FTL_BLOCK_CLOSED;
			return 0;
		}
		if (event == FTL_BLOCK_EV_RETIRE) {
			*to = FTL_BLOCK_RETIRED;
			return 0;
		}
		break;
	case FTL_BLOCK_CLOSED:
		if (event == FTL_BLOCK_EV_SELECT_VICTIM) {
			*to = FTL_BLOCK_VICTIM;
			return 0;
		}
		break;
	case FTL_BLOCK_VICTIM:
		if (event == FTL_BLOCK_EV_BEGIN_ERASE) {
			*to = FTL_BLOCK_ERASING;
			return 0;
		}
		break;
	case FTL_BLOCK_ERASING:
		if (event == FTL_BLOCK_EV_FINISH_ERASE) {
			*to = FTL_BLOCK_FREE;
			return 0;
		}
		if (event == FTL_BLOCK_EV_RETIRE) {
			*to = FTL_BLOCK_RETIRED;
			return 0;
		}
		break;
	case FTL_BLOCK_RETIRED:
		break;
	}

	return -EINVAL;
}

int block_apply_event(struct ftl_block_manager *bm,
		      const struct ftl_ppa *ppa,
		      enum ftl_block_event event)
{
	struct ftl_block *blk;
	enum ftl_block_state next;
	int ret;

	blk = block_lookup_mutable(bm, ppa);
	if (!blk)
		return -ENOENT;

	ret = block_next_state(blk->state, event, &next);
	if (ret)
		return ret;

	blk->state = next;
	block_refresh_pool_membership(bm, blk);
	return 0;
}

int block_mark_programmed(struct ftl_block_manager *bm,
			  const struct ftl_ppa *ppa)
{
	struct ftl_page *page;

	page = block_lookup_page_mutable(bm, ppa);
	if (!page)
		return -ENOENT;
	if (page->state != FTL_PAGE_ALLOCATED)
		return -EINVAL;

	page->state = FTL_PAGE_PROGRAMMED;
	return 0;
}

int block_commit_page(struct ftl_block_manager *bm,
		      const struct ftl_page_commit *commit)
{
	struct ftl_block *blk;
	struct ftl_page *page;

	blk = block_lookup_mutable(bm, &commit->ppa);
	if (!blk)
		return -ENOENT;
	if (blk->state != FTL_BLOCK_OPEN)
		return -ESTALE;

	page = block_page_at(blk, commit->ppa.page);
	if (page->state != FTL_PAGE_PROGRAMMED)
		return -EINVAL;

	page->state = FTL_PAGE_COMMITTED;
	page->lba = commit->lba;
	page->mapping_seq = commit->mapping_seq;
	blk->valid_pages++;
	return 0;
}

int block_mark_orphan(struct ftl_block_manager *bm, const struct ftl_ppa *ppa)
{
	struct ftl_page *page;

	page = block_lookup_page_mutable(bm, ppa);
	if (!page)
		return -ENOENT;
	if (page->state != FTL_PAGE_PROGRAMMED)
		return -EINVAL;

	page->state = FTL_PAGE_ORPHAN;
	return 0;
}

状态表与思想落点表

Block 生命周期状态转换表

当前状态 允许事件 下一个状态 触发者 不变量
FREE OPEN OPEN Write Path / Free Block Pool 只允许 free pool 中的 block 被打开；打开后立即从 free pool 移除。
OPEN CLOSE CLOSED Block Manager 不能再分配新 page；valid/free/page cursor 必须一致。
OPEN RETIRE RETIRED Error Handling / Bad Block program failure 后禁止继续写入；后续由恢复或搬迁处理可读页。
CLOSED SELECT_VICTIM VICTIM GC victim 不能被写路径重新打开；只允许 relocation 读取有效页。
VICTIM BEGIN_ERASE ERASING GC / Block Manager 所有有效页必须已搬迁或被确认无效；不能进入 free pool。
ERASING FINISH_ERASE FREE NAND Erase Flow erase 成功且 metadata 重置后才可回到 free pool。
ERASING RETIRE RETIRED Bad Block Management erase failure 永远不能回到 free pool。

Page 生命周期状态转换表

当前状态 事件 下一个状态 设计含义
PAGE_FREE ALLOCATE PAGE_ALLOCATED 只表示已预留物理位置，不代表 NAND 写入成功。
PAGE_ALLOCATED PROGRAM_OK PAGE_PROGRAMMED NAND 已写入，但 mapping 还不可见，journal 仍是关键边界。
PAGE_PROGRAMMED MAPPING_COMMIT PAGE_COMMITTED 该 PPA 成为 LBA 当前版本，valid count 可以增加。
PAGE_PROGRAMMED JOURNAL_FAIL PAGE_ORPHAN 介质上存在数据，但没有可恢复提交记录，不能对外可见。
PAGE_COMMITTED SUPERSEDE PAGE_INVALID 新映射提交后旧页失效，GC 可在 victim 阶段回收。

四本书思想落点表

书 思想 SSD / FTL 设计位置 它约束了什么 它帮助避免什么问题
《软件设计哲学》 用深模块隐藏复杂状态和不变量。 Block Manager 状态转换接口。 外部模块不能直接读写 block/page 内部字段。 避免状态知识散落导致变更放大和调试困难。
《程序员修炼之道》 DRY 表示知识唯一化，契约要可执行。 状态表、负 errno、非法转换检查。 所有合法状态组合只定义一次。 避免重复 if 判断产生互相矛盾的状态解释。
《设计的设计》 概念完整性优先于局部补丁。 FREE、OPEN、CLOSED、VICTIM、ERASING、RETIRED 的有限状态集合。 新增状态必须能解释生命周期，不是为某个 bug 增加临时 flag。 避免系统被 need_fix、half_valid 这类旗标污染。
《领域驱动设计》 领域状态变化应由聚合边界维护，并使用业务语言命名事件。 open block、commit page、mark orphan、select victim、finish erase。 GC、Recovery、Mapping 只能提交领域事件，不能跨边界改状态。 避免恢复逻辑无法解释写入事务停在哪个阶段。
