Day 10 · SSD / FTL 代码设计判断

GC 是领域服务，不是后台线程：让它编排流程，而不是拥有全部知识

今天的核心判断是：GC 负责协调空间回收流程，但不应该直接拥有 victim 选择、有效页判断、 relocation commit、block 擦除和 free pool 归还的全部内部知识。GC 是领域服务，不是一个什么都能改的后台线程。

一句话判断 GC 编排回收，不侵入 Mapping、Block、Journal 的事实所有权 涉及：GC / Mapping / Block Manager / Journal / Victim Selection 关键词：领域服务 · 协作边界 · relocation delta · valid check · erase handoff

今日设计判断

GC 的接口应该像 gc_reclaim_once() 一样表达编排意图， 内部通过 Mapping 查询当前性、通过 Journal 提交 relocation、通过 Block Manager 推进状态。

今日涉及模块

GC、Victim Block Selection、Mapping Table、Block Manager、Metadata Journal、 Free Block Pool、Wear Leveling、Bad Block Management、NAND Erase Flow。

今日阅读重点：把协调和所有权分开

GC 很容易变成系统里权限最大的函数，因为它要读 mapping、搬 page、擦 block、补 free pool。 但这正是边界最需要收紧的地方。GC 的价值在于编排，而不是成为第二套 Mapping、第二套 Block Manager。

思想来源 今天必须掌握的概念 落到 SSD / FTL 的判断
《软件设计哲学》 深模块把复杂度藏在稳定接口后面。GC 不应绕过深模块直接修改内部状态。 GC 不直接改 L2P、valid count、block state；它调用 Mapping、Block Manager、Journal 的领域接口。
《程序员修炼之道》 正交性要求一个知识只有一个所有者。空间压力、有效性、状态推进不能散在一个大线程里。 GC 只保留流程知识；victim 分数、page 当前性、擦除结果处理分别由策略和模块负责。
《设计的设计》 好的设计不是把所有需求放到一个中心，而是选择清晰协作形状。 GC 的设计取舍是“编排器 + 协作接口”，不是“超级后台任务”。
《领域驱动设计》 领域服务用于表达跨聚合流程，但不接管聚合内部不变量。 GC 是跨 Mapping、Block、Journal 的领域服务；Mapping 和 Block 仍拥有各自事实。

概念融合解释：GC 应该是一条可审计的协作链

GC 的复杂度来自跨模块：它要基于空间压力触发、选择 victim、识别有效页、搬迁数据、提交映射、 擦除 block、归还 free pool。把这些都写进一个线程会让每个模块的事实所有权失效。 更稳的设计是让 GC 只掌握流程阶段，每个阶段通过领域接口向真正所有者发请求。

统一判断： GC 是领域服务，它的职责是把“空间回收”这个业务过程串起来；它不应成为 Mapping、Block Manager、 Journal、Wear Leveling 的替身。只要 GC 直接写它们的内部字段，恢复语义和长期演进都会被破坏。

SSD / FTL 设计场景

Free Block Pressure

GC 根据 free block 水位和写入预算启动，但水位计算来自 Block Manager，GC 不直接维护 free pool。

Victim Block Selection

GC 请求策略模块给出 victim candidate、score 和 reason；它不在流程中散落 erase count、valid count、冷热判断。

Valid Page Relocation

GC 搬迁前调用 Mapping 做 current check，只有当前 LBA 版本才构造 relocation delta。

Metadata Journal

每个 relocation delta 先 journal append，再 commit mapping；掉电后 Recovery 能重复 replay。

NAND Erase Flow

GC 只请求 Block Manager 从 VICTIM 进入 ERASING；erase 结果由 Block Manager 决定 FREE 或 RETIRED。

Bad Block Management

erase fail 不是 GC 私有错误，必须走坏块退役路径，避免失败 block 被重新加入 free pool。

坏味道： 这段代码把 victim 选择、mapping 判断、page 搬迁、valid count、erase、free pool 更新全部写在 GC 线程里。 它看似集中，实际是在复制多个模块的知识。
反例：GC 直接修改 Mapping 和 Block 内部状态 C 
int gc_thread_bad(struct ftl_dev *ftl)
{
	struct ftl_block *victim = NULL;
	struct ftl_page *page;
	struct ftl_ppa new_ppa;
	int i;
	int ret;

	for (i = 0; i < ftl->nr_blocks; i++) {
		struct ftl_block *blk = &ftl->blocks[i];

		if (blk->bad || blk->open || blk->free)
			continue;
		if (!victim || blk->valid_pages < victim->valid_pages)
			victim = blk;
	}

	if (!victim)
		return -ENOSPC;

	victim->victim = true;

	for_each_page(victim, page) {
		if (!page->valid)
			continue;
		if (ftl->l2p[page->lba].block != victim->id)
			continue;

		ret = block_alloc_page(ftl->bm, &new_ppa);
		if (ret)
			return ret;

		ret = nand_program_page(ftl->nand, &new_ppa, page->data);
		if (ret)
			return ret;

		ftl->l2p[page->lba] = new_ppa;
		page->valid = false;
		victim->valid_pages--;
	}

	ret = nand_erase_block(ftl->nand, victim->id);
	if (ret)
		victim->bad = true;
	else
		free_pool_push(ftl->free_pool, victim);

	return ret;
}
设计问题 代码表现 后果
策略和流程耦合 victim 选择直接写在 GC 循环里。 Wear Leveling、冷热数据、QoS 策略无法独立演进。
Mapping 所有权泄漏 GC 直接读写 l2p。 无法做条件提交、journal replay 和并发当前性校验。
Block 状态污染 直接改 victim、valid_pages、bad。 状态机被绕过，Recovery 无法解释 block 停在哪个阶段。
错误路径无补偿 program fail 后直接返回。 新分配 page、journal、mapping、old page 状态没有统一收敛。

改进代码：GC 只编排领域协作

改进版把 GC 定义为领域服务。GC 生成 reclaim plan，向策略模块拿 candidate， 向 Mapping 提交 relocation delta，向 Block Manager 推进 victim 和 erase 状态。

gc_service.h：GC 服务接口 C 
#ifndef FTL_GC_SERVICE_H
#define FTL_GC_SERVICE_H

#include <linux/types.h>
#include <linux/errno.h>

enum gc_reason {
	GC_REASON_LOW_FREE_BLOCKS = 0,
	GC_REASON_WRITE_BUDGET,
	GC_REASON_WEAR_LEVELING,
};

struct gc_request {
	enum gc_reason reason;
	u32 target_free_blocks;
	u32 max_pages_to_move;
	bool allow_foreground_gc;
};

struct gc_reclaim_result {
	u32 moved_pages;
	u32 reclaimed_blocks;
	u32 retired_blocks;
	int last_errno;
};

struct gc_service;

int gc_reclaim_once(struct gc_service *gc, const struct gc_request *req,
		    struct gc_reclaim_result *result);

#endif
gc_service.c：流程编排，不直接拥有事实 C 
static int gc_move_valid_page(struct gc_service *gc,
			      const struct ftl_ppa *old_ppa,
			      struct gc_reclaim_result *result)
{
	struct mapping_relocation_delta delta;
	struct ftl_ppa new_ppa;
	int ret;

	ret = map_build_relocation_delta(gc->map, old_ppa, &delta);
	if (ret == -ENOENT)
		return 0;
	if (ret)
		return ret;

	ret = block_alloc_page(gc->bm, &new_ppa);
	if (ret)
		return ret;

	delta.new_ppa = new_ppa;

	ret = nand_copy_page(gc->nand, old_ppa, &new_ppa);
	if (ret)
		goto err_copy;

	ret = journal_append_relocation(gc->journal, &delta);
	if (ret)
		goto err_journal;

	ret = map_commit_relocation_if_current(gc->map, &delta);
	if (ret)
		goto err_mapping;

	ret = block_commit_relocated_page(gc->bm, &delta);
	if (ret)
		return ret;

	result->moved_pages++;
	return 0;

err_mapping:
	gc_mark_recovery_required(gc, GC_RECOVERY_REPLAY_RELOCATION);
	return ret;

err_journal:
	block_mark_orphan(gc->bm, &new_ppa);
	return ret;

err_copy:
	block_apply_program_failure(gc->bm, &new_ppa, ret);
	return ret;
}

int gc_reclaim_once(struct gc_service *gc, const struct gc_request *req,
		    struct gc_reclaim_result *result)
{
	struct victim_candidate victim;
	struct ftl_ppa page_ppa;
	int ret;

	memset(result, 0, sizeof(*result));

	ret = victim_select(gc->victim_policy, req, &victim);
	if (ret)
		return ret;

	ret = block_begin_gc_victim(gc->bm, &victim.block_ppa);
	if (ret)
		return ret;

	while (block_next_valid_page(gc->bm, &victim.block_ppa, &page_ppa) == 0) {
		ret = gc_move_valid_page(gc, &page_ppa, result);
		if (ret)
			return ret;
		if (result->moved_pages >= req->max_pages_to_move)
			return -EAGAIN;
	}

	ret = block_erase_victim(gc->bm, &victim.block_ppa);
	if (ret == -EIO)
		result->retired_blocks++;
	else if (!ret)
		result->reclaimed_blocks++;

	return ret;
}

责任表与思想落点表

GC 协作责任矩阵

协作点 GC 负责 其他模块负责 禁止做法
触发条件 根据空间压力和预算发起 reclaim request。 Block Manager 提供 free block 水位和状态统计。 GC 自己维护一份 free pool 影子状态。
Victim 选择 请求 candidate，记录 reason。 策略模块计算 score，Wear Leveling 提供磨损压力。 GC 流程里硬编码 score 公式。
有效页判断 按页请求 relocation delta。 Mapping 判断 old PPA 是否仍是当前版本。 GC 直接读取并改写 L2P。
提交搬迁 编排 copy、journal、mapping commit、block commit。 Journal 保证可恢复记录，Mapping 条件提交，Block 推进 page 状态。 program 后直接替换 mapping。
擦除归还 请求 block erase victim。 Block Manager 根据 erase 结果转 FREE 或 RETIRED。 GC 直接把 block push 到 free pool。

GC 阶段错误路径矩阵

阶段 失败 领域解释 补偿所有者 返回语义
Victim begin 状态不允许进入 VICTIM。 candidate 已过期或 block 正被其他流程接管。 Block Manager -ESTALE
Copy page program fail。 new PPA 未提交，可能需要退役 open block。 Block Manager / Bad Block -EIO
Journal append relocation delta 未持久化。 new page 不能成为当前 mapping。 Journal / Block Manager -EIO
Mapping commit old PPA 已非当前。 该页已被前台写替换，GC 应跳过或标记旧页无效。 Mapping Table -ESTALE 可收敛
Erase victim erase fail。 block 不能回 free pool。 Bad Block Management -EIO

四本书思想落点表

书 思想 SSD / FTL 设计位置 它约束了什么 它帮助避免什么问题
《软件设计哲学》 深模块和信息隐藏。 Mapping、Block Manager、Journal 的 GC-facing 接口。 GC 不能绕过模块直接改内部状态。 避免 GC 成为复杂度黑洞。
《程序员修炼之道》 正交性和知识唯一化。 GC request、victim candidate、relocation delta。 策略知识、状态知识、映射知识分别只有一个所有者。 避免 valid count、erase count、冷热判断散落。
《设计的设计》 概念完整性优先于局部方便。 GC 编排器与协作接口的形状。 新增功能必须落在合适协作点，而不是继续扩大 GC 线程。 避免后台任务吞掉所有设计边界。
《领域驱动设计》 领域服务协调跨聚合流程，但不拥有聚合内部不变量。 GC Service、Mapping Aggregate、Block Aggregate、Journal Service。 GC 只表达 reclaim 过程，不拥有 L2P 或 block 生命周期事实。 避免 Recovery、GC、Mapping 的语言混乱。
