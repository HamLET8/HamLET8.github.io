Day 8 · SSD / FTL 代码设计判断

Metadata Journal 的最小可恢复记录：记录领域变更，而不是转储内存结构

今天的核心判断是：Journal 应记录 FTL 领域变更意图，而不是把运行时 struct 原样写到 NAND。 每条记录都必须可重放、可幂等、可校验，并能让 Power Loss Recovery 判断事务停在哪个边界。

一句话判断 Journal 是恢复契约，不是调试转储 涉及：Metadata Journal / Mapping / Block Manager / GC / Recovery 关键词：领域事件 · redo log · 幂等 · checksum · seq · commit boundary

今日设计判断

Journal record 应表达“把 LBA 从 old PPA 切到 new PPA”“把 block 从 VICTIM 推进到 ERASING” 这类领域事件。它不应该依赖内存布局、指针、缓存字段或某个版本的 struct padding。

今日涉及模块

Metadata Journal、Mapping Table、Block Manager、GC Relocation、Victim Block Selection、 Free Block Pool、Power Loss Recovery、NAND Program / Erase Flow。

今日阅读重点：Journal 是写给未来系统看的接口

Metadata Journal 的读者不是当前函数，而是掉电后的 Recovery。它必须在没有内存上下文、 没有锁状态、没有临时缓存的情况下解释系统事实。最小可恢复记录的目标，是用尽量少的稳定字段表达足够恢复的领域语义。

思想来源 今天必须掌握的概念 落到 SSD / FTL 的判断
《软件设计哲学》 信息隐藏不等于少暴露函数，而是隐藏易变表示。Journal 格式一旦落盘，就是长期接口。 Journal 不写 struct map_entry 的内存镜像；只写恢复需要的 LBA、old PPA、new PPA、mapping seq、校验信息。
《程序员修炼之道》 可逆性和正交性要求关键决策可撤退、可重放。日志记录必须让系统从中间失败点回到一致状态。 每条记录要有 magic、version、type、seq、length、crc，Recovery 能跳过 torn record，也能重复 replay 不破坏状态。
《设计的设计》 设计不是追求信息最多，而是选择恰好足够的约束。多写字段会把当前实现锁进未来。 把 record 设计成领域事件集合，而不是“一条万能 metadata blob”；不同事件有自己的必要字段和校验边界。
《领域驱动设计》 领域事件描述已经发生、需要被系统承认的事实。Journal 是领域事件的持久化边界。 使用 JNL_MAP_DELTA、JNL_BLOCK_STATE、JNL_GC_RELOCATION 表达业务含义，而不是 mode/flag。

概念融合解释：最小记录要同时满足三件事

Metadata Journal 的记录太少，Recovery 无法判断该不该 replay；记录太多，落盘格式会绑死内存实现。 合理边界是：记录领域事件、携带前置条件、保留校验信息。这样接口稳定、错误路径可解释、状态机可推进， 并且未来可以演进 record version，而不是迁移整个内存结构。

统一判断： Journal 记录不是“把当前状态保存下来”，而是“保存一个可验证的领域变更”。它必须能回答： 这条记录描述什么事件、事件的前置事实是什么、重复应用是否安全、记录本身是否完整。

SSD / FTL 设计场景

Host Write Mapping Commit

Program 成功后，Journal 记录 LBA: old PPA -> new PPA 和 mapping seq。掉电后 Recovery 可判断是否已经提交。

GC Relocation

GC 搬迁不能只写“某个 page 有效”。它要记录 relocation delta，说明从 victim PPA 搬到 new PPA，并允许重复 replay。

Block State Event

Victim block 进入 ERASING 或 erase 成功回 FREE，应作为 block state event 记录，避免 Recovery 重新把不可用 block 放回 free pool。

Power Loss Recovery

Recovery 按 seq 扫描 journal：校验 header，识别 record type，执行幂等 replay，遇到 torn record 停止而不是猜测。

Bad Block Management

Program/erase fail 后的 retire block 需要 journal 化，否则掉电后坏块可能被重新分配，形成数据破坏。

Mapping Snapshot

Checkpoint 只截断已持久化前缀。Journal record 的 seq 与 checkpoint seq 共同定义恢复起点。

反例代码：把内存结构当 Journal 格式

坏味道： 这段代码把 Mapping、Block、GC 的运行时结构直接写入 journal。它依赖 struct 布局， 没有 record type、version、seq、crc，也没有前置条件；重放时会重复增加计数，无法幂等。
反例：Journal 变成内存转储 C 
struct map_entry_bad {
	u64 lba;
	struct ftl_ppa ppa;
	struct ftl_block *owner;
	u32 flags;
	u32 valid_pages_snapshot;
};

struct journal_blob_bad {
	u32 mode;
	u32 size;
	u8 data[512];
};

int journal_write_mapping_bad(struct ftl_dev *ftl,
			      struct map_entry_bad *entry)
{
	struct journal_blob_bad blob;

	blob.mode = 3;
	blob.size = sizeof(*entry);
	memcpy(blob.data, entry, sizeof(*entry));

	return nand_program_page(ftl->nand, ftl->journal_cursor, &blob,
				 sizeof(blob));
}

int recovery_replay_blob_bad(struct ftl_dev *ftl,
			     const struct journal_blob_bad *blob)
{
	struct map_entry_bad entry;
	struct ftl_block *blk;

	if (blob->mode != 3)
		return -EINVAL;

	memcpy(&entry, blob->data, sizeof(entry));
	blk = entry.owner;
	ftl->l2p[entry.lba] = entry.ppa;

	if (entry.flags & 0x1)
		blk->valid_pages++;

	return 0;
}
设计问题 代码表现 恢复后果
落盘格式依赖内存布局 直接 memcpy() struct，包含指针和 padding。 版本升级或重启后指针无意义，字段变化会破坏旧 journal。
领域语义丢失 mode = 3 和 flags 表达不出 mapping delta、relocation、retire block 的区别。 Recovery 无法判断记录应该调用 Mapping、Block Manager 还是 GC 补偿。
不可校验 没有 magic、version、length、crc、seq。 torn record 可能被当成有效记录重放，造成 silent corruption。
不可幂等 重放时无条件增加 valid_pages。 重复 replay 会让计数漂移，GC victim selection 和 free block pressure 失真。

改进代码：版本化、可校验、可幂等的领域记录

改进版把 journal 格式设计成公开契约：header 负责完整性和顺序，payload 只表达某类领域事件的必要字段。 Recovery 不复制内存，而是根据 record type 调用对应模块的幂等提交接口。

journal.h：最小可恢复记录接口 C 
#ifndef FTL_JOURNAL_H
#define FTL_JOURNAL_H

#include <linux/types.h>
#include <linux/errno.h>

#define FTL_JOURNAL_MAGIC	0x4a4e4c31u
#define FTL_JOURNAL_VERSION	1u

enum journal_record_type {
	JNL_REC_MAP_DELTA = 1,
	JNL_REC_GC_RELOCATION,
	JNL_REC_BLOCK_STATE,
	JNL_REC_CHECKPOINT,
};

struct journal_record_header {
	u32 magic;
	u16 version;
	u16 type;
	u64 seq;
	u32 payload_len;
	u32 payload_crc;
};

struct journal_map_delta {
	u64 lba;
	struct ftl_ppa old_ppa;
	struct ftl_ppa new_ppa;
	u64 expected_mapping_seq;
	u64 new_mapping_seq;
};

struct journal_block_state {
	struct ftl_ppa block_ppa;
	enum ftl_block_state from;
	enum ftl_block_state to;
	u64 reason_seq;
};

struct ftl_journal;
struct ftl_recovery_ctx;

int journal_append_mapping_delta(struct ftl_journal *jnl,
				 const struct journal_map_delta *delta);
int journal_append_block_state(struct ftl_journal *jnl,
			       const struct journal_block_state *state);
int journal_replay_record(struct ftl_recovery_ctx *ctx,
			  const struct journal_record_header *hdr,
			  const void *payload);

#endif
journal.c：append 和 replay 都围绕领域事件 C 
static int journal_validate_header(const struct journal_record_header *hdr,
				   u32 payload_crc)
{
	if (hdr->magic != FTL_JOURNAL_MAGIC)
		return -EINVAL;
	if (hdr->version != FTL_JOURNAL_VERSION)
		return -EOPNOTSUPP;
	if (hdr->payload_crc != payload_crc)
		return -EILSEQ;
	if (!hdr->payload_len)
		return -EINVAL;

	return 0;
}

int journal_append_mapping_delta(struct ftl_journal *jnl,
				 const struct journal_map_delta *delta)
{
	struct journal_record_header hdr;
	u32 crc;

	if (!ftl_ppa_valid(&delta->new_ppa))
		return -EINVAL;
	if (delta->new_mapping_seq <= delta->expected_mapping_seq)
		return -EINVAL;

	crc = ftl_crc32(delta, sizeof(*delta));
	hdr.magic = FTL_JOURNAL_MAGIC;
	hdr.version = FTL_JOURNAL_VERSION;
	hdr.type = JNL_REC_MAP_DELTA;
	hdr.seq = journal_next_seq(jnl);
	hdr.payload_len = sizeof(*delta);
	hdr.payload_crc = crc;

	return journal_append_raw(jnl, &hdr, delta, sizeof(*delta));
}

static int replay_mapping_delta(struct ftl_recovery_ctx *ctx,
				const struct journal_map_delta *delta)
{
	int ret;

	if (map_delta_already_applied(ctx->map, delta))
		return 0;

	ret = map_commit_delta_if_current(ctx->map, delta->lba,
					  &delta->old_ppa,
					  &delta->new_ppa,
					  delta->expected_mapping_seq,
					  delta->new_mapping_seq);
	if (ret == -ESTALE && map_delta_already_applied(ctx->map, delta))
		return 0;

	return ret;
}

int journal_replay_record(struct ftl_recovery_ctx *ctx,
			  const struct journal_record_header *hdr,
			  const void *payload)
{
	u32 crc;
	int ret;

	crc = ftl_crc32(payload, hdr->payload_len);
	ret = journal_validate_header(hdr, crc);
	if (ret)
		return ret;

	switch (hdr->type) {
	case JNL_REC_MAP_DELTA:
		if (hdr->payload_len != sizeof(struct journal_map_delta))
			return -EINVAL;
		return replay_mapping_delta(ctx, payload);
	case JNL_REC_BLOCK_STATE:
		if (hdr->payload_len != sizeof(struct journal_block_state))
			return -EINVAL;
		return block_replay_state_event(ctx->bm, payload);
	case JNL_REC_GC_RELOCATION:
		return gc_replay_relocation(ctx, payload, hdr->payload_len);
	case JNL_REC_CHECKPOINT:
		return recovery_accept_checkpoint(ctx, payload, hdr->seq);
	default:
		return -EOPNOTSUPP;
	}
}

记录表与思想落点表

最小可恢复记录字段表

字段 必须性 解决的问题 设计约束
magic 识别 journal record 起点。 避免把随机 NAND 数据或 torn page 当成有效记录。 固定值，跨版本保持可识别。
version 区分 record 格式。 允许未来扩展 payload，而不破坏旧 Recovery。 新增字段必须有兼容策略或迁移边界。
type 表达领域事件类型。 Recovery 能分派到 Mapping、Block Manager、GC 等所有者。 禁止用 mode/flag 表达多个概念混合状态。
seq 定义重放顺序和 checkpoint 边界。 检测 gap、重复记录、旧记录覆盖新事实。 单调递增；checkpoint 只截断已确认前缀。
payload_len 定义校验和扫描边界。 避免越界读取或误读下一个 record。 必须和 type 的 payload schema 匹配。
payload_crc 判断 record 是否完整。 识别 torn write、bit flip 和错误扫描位置。 校验失败时停止或进入恢复策略，不能盲目 replay。
precondition 让 replay 幂等。 重复应用不会二次增加计数或覆盖更新版本。 如 old PPA、expected seq、from state 必须是领域前置事实。

掉电位置到恢复动作矩阵

掉电位置 Journal 状态 Recovery 判断 动作 所有者
Program 成功前 无记录 没有可见变更。 保留旧 mapping；可扫描未提交 page。 Recovery / Block Manager
Program 成功，Journal 前 无记录 新 page 是 orphan 候选。 不提交 mapping；标记 ORPHAN 或等待 GC 回收。 Block Manager
Journal append 中断 torn record crc 或 length 不通过。 停止扫描到最后完整 seq；不 replay 破损记录。 Journal / Recovery
Journal 完整，Mapping 前 完整 map delta 领域事件已持久化但未必可见。 按前置条件幂等 replay 到 Mapping。 Mapping Table
Mapping 后，Block commit 前 完整 map delta mapping 可能已应用，page 状态可能未推进。 检测 mapping seq；已应用则补齐 page COMMITTED。 Mapping / Block Manager
Erase 完成前 block state event 可能完整 不能假设 block 可用。 重新校验 erase 结果；失败转 RETIRED，成功转 FREE。 Block Manager / Bad Block

四本书思想落点表

书 思想 SSD / FTL 设计位置 它约束了什么 它帮助避免什么问题
《软件设计哲学》 信息隐藏和深模块：隐藏易变表示，暴露稳定语义。 Journal record schema、append/replay API。 不能把内存 struct、指针、缓存字段写成落盘接口。 避免格式和实现强绑定，导致演进困难。
《程序员修炼之道》 可逆性、正交性、契约清晰。 seq、crc、version、幂等 replay。 每条记录必须能验证完整性，并能安全重复应用。 避免 torn record、重复 replay、silent corruption。
《设计的设计》 设计取舍是选择恰好足够的约束，而不是保存一切。 最小 payload、record type 分解、checkpoint 边界。 字段必须服务恢复语义，不能因为“以后可能有用”无限膨胀。 避免 journal 变成不可维护的 metadata 垃圾场。
《领域驱动设计》 领域事件表达系统已经承认的事实。 JNL_REC_MAP_DELTA、JNL_REC_BLOCK_STATE、JNL_REC_GC_RELOCATION。 Record type 必须映射到明确领域所有者和恢复动作。 避免 mode/flag 让 Recovery 无法解释事件含义。
