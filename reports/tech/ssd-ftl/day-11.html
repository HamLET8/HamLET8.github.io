Day 11 · SSD / FTL 代码设计判断

Victim Selection 与 Wear Leveling 策略分离：策略输出 score 和理由

今天的核心判断是：选择策略应输出 score、reason 和约束说明；GC 流程不直接散落 erase count、 valid count、hot/cold 数据判断。策略可替换，流程保持稳定，长期磨损控制才有演进空间。

一句话判断 策略计算分数，GC 只消费决策 涉及：Victim Selection / Wear Leveling / GC / Mapping / Block Manager 关键词：score · reason · policy ops · hot/cold · wear pressure · algorithm evolution

今日设计判断

不要在 gc_reclaim_once() 里写复杂打分公式。 用 victim_policy_select() 返回 candidate、score、reason 和风险标记。

今日涉及模块

Victim Selection、Wear Leveling、GC、Block Manager、Mapping Table、Hot/Cold Data Classifier、 Free Block Pool、Bad Block Management。

今日阅读重点：策略要能替换，理由要能审计

Victim 选择不是一个 if 条件，而是长期产品策略：空间回收效率、写放大、磨损均衡、前台延迟都在这里冲突。 策略分离的目的不是抽象漂亮，而是让这些取舍可以被替换、观测和回放。

思想来源 今天必须掌握的概念 落到 SSD / FTL 的判断
《软件设计哲学》 接口要隐藏复杂实现，但暴露有用信息。策略算法可复杂，输出接口必须简单稳定。 Victim policy 返回 score、reason、candidate，不暴露内部公式给 GC。
《程序员修炼之道》 正交性和可配置性要求策略和机制分离。算法变化不应改动 GC 主流程。 GC 是机制，Victim Selection 和 Wear Leveling 是策略；二者通过 policy ops 连接。
《设计的设计》 设计取舍需要可解释。没有 reason 的 score 只是魔法数字，无法评估副作用。 输出 reason：低有效页、磨损压力、冷热迁移、QoS 预算，而不是只返回 block id。
《领域驱动设计》 策略是领域知识，不是流程细节。它应使用领域语言表达候选理由。 使用 wear pressure、valid ratio、cold data、relocation cost 等领域词汇建模。

概念融合解释：score 是机器选择，reason 是工程判断

如果 victim selection 只返回一个 block，GC 无法知道这次选择是为了快速回收空间，还是为了降低磨损偏差。 如果策略代码散在 GC 中，算法变化会改动回收流程。更稳的设计是让策略模块输出结构化决策： candidate、score、reason、relocation cost、wear delta、QoS 风险。

统一判断： 策略分离不是为了“多一层抽象”，而是为了把会频繁调整的取舍从稳定流程中移出。 GC 主流程保持可预测，策略模块承担算法演进，Block Manager 和 Mapping 继续保有事实所有权。

SSD / FTL 设计场景

Low Free Block GC

空间压力高时，策略优先选择 valid ratio 低、搬迁成本小的 block，以快速回收 free block。

Dynamic Wear Leveling

前台写入集中在少量 block 时，策略提高磨损低 block 的选择概率，避免热点区域过早老化。

Static Wear Leveling

冷数据长期占用低 erase count block 时，策略可选择高价值冷 block 做迁移，平衡全盘磨损。

Hot / Cold Data

策略读取数据温度信号，但 GC 流程不解释温度算法；温度分类器可以单独演进。

QoS Budget

前台延迟敏感时，策略降低搬迁成本高的 candidate 权重，返回 reason 说明选择受 QoS 限制。

Bad Block Risk

接近坏块阈值的 block 可被策略标记 risk，交给 GC 以更保守的 erase / retire 路径处理。

反例代码：评分公式散落在 GC 流程里

坏味道： 这段代码把 victim selection、wear leveling、冷热判断和 QoS 都硬编码在 GC 中。 算法变化会改动流程，且没有 reason 可供调试和回放。
反例：GC 内部硬编码策略 C 
struct ftl_block *gc_pick_victim_bad(struct ftl_dev *ftl, int foreground)
{
	struct ftl_block *best = NULL;
	int best_score = -1;
	int i;

	for (i = 0; i < ftl->nr_blocks; i++) {
		struct ftl_block *blk = &ftl->blocks[i];
		int score;

		if (blk->bad || blk->open || blk->free)
			continue;

		score = (blk->pages_per_block - blk->valid_pages) * 100;

		if (blk->erase_count < ftl->avg_erase_count)
			score += 30;
		if (blk->cold_data)
			score -= 20;
		if (foreground && blk->valid_pages > 32)
			score -= 200;
		if (blk->last_read_retry)
			score += 10;

		if (score > best_score) {
			best_score = score;
			best = blk;
		}
	}

	return best;
}
设计问题 代码表现 后果
算法和流程耦合 GC 直接计算 score。 替换策略必须修改 GC 主流程，回归风险高。
reason 缺失 只返回 best block。 无法解释某次选择是为了空间、磨损、冷热还是 QoS。
输入所有权混乱 直接读 cold_data、last_read_retry。 温度分类、介质健康和磨损统计都被 GC 流程吸收。
不可演进 魔法数散落。 调参无法记录版本，也无法在仿真和线上共享同一策略。

改进代码：策略接口输出结构化决策

改进版把策略定义为可替换接口。GC 只消费候选结果，策略模块负责读取特征、计算 score、 输出 reason 和风险标记。这样可以在不改 GC 流程的前提下演进算法。

victim_policy.h：策略输出 score 和 reason C 
#ifndef FTL_VICTIM_POLICY_H
#define FTL_VICTIM_POLICY_H

#include <linux/types.h>
#include <linux/errno.h>

enum victim_reason {
	VICTIM_REASON_RECLAIM_EFFICIENCY = 0,
	VICTIM_REASON_DYNAMIC_WL,
	VICTIM_REASON_STATIC_WL,
	VICTIM_REASON_QOS_BUDGET,
	VICTIM_REASON_MEDIA_RISK,
};

struct victim_features {
	u32 valid_pages;
	u32 free_pages;
	u32 erase_count;
	u32 avg_erase_count;
	u32 hotness;
	u32 corrected_bits;
	bool foreground_gc;
};

struct victim_candidate {
	struct ftl_ppa block_ppa;
	s32 score;
	enum victim_reason reason;
	u32 relocation_cost;
	u32 wear_delta;
	u32 risk_flags;
};

struct block_manager;

struct victim_policy_request {
	struct block_manager *bm;
	bool foreground_gc;
	u32 max_relocation_cost;
};

struct victim_policy_ops {
	int (*score)(const struct victim_features *features,
		     struct victim_candidate *candidate);
};

struct victim_policy {
	const struct victim_policy_ops *ops;
	void *private;
};

int victim_policy_select(struct victim_policy *policy,
			 const struct victim_policy_request *req,
			 struct victim_candidate *candidate);

#endif
victim_policy.c：策略独立演进 C 
static int balanced_policy_score(const struct victim_features *f,
				 struct victim_candidate *candidate)
{
	s32 reclaim_score;
	s32 wear_score;
	s32 qos_penalty;

	reclaim_score = (s32)(f->free_pages * 100) - (s32)(f->valid_pages * 4);
	wear_score = (s32)f->avg_erase_count - (s32)f->erase_count;
	qos_penalty = f->foreground_gc ? (s32)f->valid_pages * 8 : 0;

	candidate->score = reclaim_score + wear_score - qos_penalty;
	candidate->relocation_cost = f->valid_pages;
	candidate->wear_delta = abs(wear_score);

	if (f->corrected_bits > MEDIA_RISK_CORRECTED_BITS) {
		candidate->reason = VICTIM_REASON_MEDIA_RISK;
		candidate->risk_flags |= VICTIM_RISK_READ_RECLAIM;
	} else if (wear_score > WEAR_LEVELING_THRESHOLD) {
		candidate->reason = VICTIM_REASON_DYNAMIC_WL;
	} else if (f->foreground_gc && qos_penalty) {
		candidate->reason = VICTIM_REASON_QOS_BUDGET;
	} else {
		candidate->reason = VICTIM_REASON_RECLAIM_EFFICIENCY;
	}

	return 0;
}

int victim_policy_select(struct victim_policy *policy,
			 const struct victim_policy_request *req,
			 struct victim_candidate *candidate)
{
	struct victim_features features;
	struct victim_candidate best = { 0 };
	bool found = false;
	int ret;

	block_iter_begin(req->bm);
	while (block_iter_next_candidate(req->bm, &features) == 0) {
		struct victim_candidate current = { 0 };

		current.block_ppa = features_to_block_ppa(&features);
		ret = policy->ops->score(&features, &current);
		if (ret)
			return ret;

		if (!found || current.score > best.score) {
			best = current;
			found = true;
		}
	}

	if (!found)
		return -ENOSPC;

	*candidate = best;
	return 0;
}

策略表与思想落点表

策略输入与所有权表

输入特征 所有者 策略用途 接口约束
valid / free page count Block Manager 估算 relocation cost 和回收收益。 策略只读快照，不直接修正计数。
erase count / avg erase count Wear Leveling 统计 估算 wear delta，判断是否需要动态均衡。 统计版本要可追踪，不能散落魔法阈值。
hotness / coldness Hot/Cold Classifier 判断静态 wear leveling 是否值得迁移冷数据。 分类算法可变，策略接口保持稳定。
corrected bits / read retry ECC / Read Path 把介质健康纳入 read reclaim 或 media risk。 策略输出 risk flag，不直接退役 block。
foreground GC / QoS budget Scheduler / GC Request 限制高 relocation cost candidate。 策略解释选择原因，GC 执行预算。

score 到 GC 决策映射

reason score 主要来源 GC 应如何消费 不能做什么
RECLAIM_EFFICIENCY 低 valid ratio，高 free gain。 按普通 relocation 流程搬迁有效页并擦除 victim。 GC 不重新计算收益公式。
DYNAMIC_WL erase count 与平均值差距。 记录 wear-leveling 原因，限制对前台延迟的影响。 GC 不直接修改 wear 统计。
STATIC_WL 冷数据长期占用低磨损 block。 以低频后台任务执行，避免扰动 hot write。 不能在前台 GC 中强行搬大量冷数据。
QOS_BUDGET foreground request 和 relocation cost。 选择可在预算内完成的 victim。 不能忽略 score 直接选最低 valid count。
MEDIA_RISK ECC corrected bits、read retry 次数。 触发 read reclaim 或更保守迁移。 策略不能直接把 block 标成 bad。

书 思想 SSD / FTL 设计位置 它约束了什么 它帮助避免什么问题
《软件设计哲学》 深模块：复杂算法隐藏在简单稳定接口后。 victim_policy_select() 和 victim_candidate。 GC 只接收 score 和 reason，不理解内部公式。 避免策略细节泄漏到回收流程。
《程序员修炼之道》 正交性：策略和机制分离。 policy ops、features snapshot、GC request。 算法替换不改 GC 主流程。 避免每次调参都影响 relocation 和 erase 路径。
《设计的设计》 取舍要可解释。 reason、risk_flags、relocation_cost。 每次选择必须能说明为什么选这个 block。 避免 score 成为不可审计魔法数字。
《领域驱动设计》 策略是领域知识，必须使用领域语言表达。 wear pressure、hotness、cold data、media risk。 策略输入来自领域事实所有者，输出领域决策。 避免 erase count、hot/cold、ECC 信号在 GC 中混杂。
