Day 6 · SSD / FTL 代码设计判断

错误路径不是 return 语句：先把 NAND 结果映射成 FTL 决策

今天的核心判断是：NAND program/read/erase 的失败不能被主路径简单透传。 FTL 必须先把介质结果转成领域决策，再进入 Block Manager、Mapping、Journal、Recovery 的补偿路径。 否则错误处理会散落，掉电恢复也无法解释系统停在哪个阶段。

一句话判断 介质错误先领域化，再补偿 涉及：NAND / Block Manager / Mapping / Journal / Recovery 关键词：错误矩阵 · 补偿路径 · orphan page · recovery boundary

今日设计判断

主路径不应该写成 if (ret) return ret;。 它应该识别失败发生在哪个事务阶段，并调用对应模块完成 discard、retire、retry、replay 标记或映射回滚。

今日涉及模块

NAND Program / Erase / Read Flow、ECC、Read Retry、Bad Block Management、 Metadata Journal、Mapping Table、Power Loss Recovery、Free Block Pool。

今日阅读重点：错误是设计的一部分

在 FTL 里，错误不是边角 case。NAND 介质会失败，写入事务会卡在中间阶段，journal 可能写不进去， erase 可能退役 block，read 可能依赖 ECC 或 read retry。错误路径就是设计路径。

思想来源 今天必须掌握的概念 落到 SSD / FTL 的判断
《软件设计哲学》 复杂度来自特殊情况扩散。深模块应隐藏错误细节，向上暴露稳定决策。 NAND 层返回介质结果，FTL 错误转换层把它映射为 retry、retire、discard、recovery-required。
《程序员修炼之道》 务实工程要求失败早暴露、契约明确。不要让调用方猜错误码背后的状态。 -EIO 不够；需要附带阶段、PPA、介质动作和可执行补偿建议。
《设计的设计》 设计取舍要承认现实约束。错误路径必须服务概念完整性，而不是散在每个 if 分支。 program 成功但 journal 失败时，概念上是“已写未提交页”，不是普通写失败。
《领域驱动设计》 领域异常要用领域语言表达。错误不是硬件状态码，而是 FTL 状态转换事件。 使用 orphan page、bad block candidate、recovery-required、stale mapping 等领域词汇建模失败。

概念融合解释：错误路径必须绑定事务阶段

同一个 -EIO 在不同阶段含义完全不同。 Program 前失败，可能只是没有空间；program 失败，可能要隔离 block；program 成功但 journal 失败， 会产生已写但不可见的 orphan page；journal 成功但 mapping commit 失败，则 Recovery 必须能重放。

统一判断： 错误处理的第一步不是返回错误码，而是定位事务阶段。阶段决定补偿所有者：NAND 负责介质结果， Block Manager 负责 page/block 生命周期，Journal 负责可恢复记录，Mapping 负责逻辑可见性，Recovery 负责跨重启收敛。

SSD / FTL 设计场景

NAND Program Failure

Program fail 发生在 mapping 可见前。FTL 应报告 Block Manager，隔离候选坏块，并重新分配或返回可解释错误。

Journal Append Failure

Program 已成功但 journal 失败，新页不能成为当前映射；它应被标为 uncommitted/orphan，等待 GC 回收。

Mapping Commit Failure

Journal 已成功但 Mapping commit 失败，不能丢弃记录；系统应标记 recovery required，允许重放收敛。

Read Retry / ECC

Read correctable 不是普通成功，应该反馈介质健康；uncorrectable 进入 read retry、镜像副本或恢复路径。

Erase Failure

Erase fail 表示 block 不能回到 free pool。Block Manager 应退役或隔离 block，更新坏块统计。

Power Loss Recovery

恢复时不能重走普通错误路径，而应根据 journal 与 NAND metadata 判断事务停在哪个阶段。

反例代码：错误码透传导致补偿丢失

坏味道： 这段代码把所有失败都当作 return ret。 它没有记录事务阶段，也没有让拥有者模块执行补偿动作。
反例：错误路径散落且不可恢复 C 
int ftl_write_bad(struct ftl_dev *ftl, u64 lba, const void *buf)
{
	struct ftl_ppa old_ppa;
	struct ftl_ppa new_ppa;
	int ret;

	ret = map_lookup_l2p(ftl->map, lba, &old_ppa);
	if (ret && ret != -ENOENT)
		return ret;

	ret = block_alloc_page(ftl->bm, &new_ppa);
	if (ret)
		return ret;

	ret = nand_program_page(ftl->nand, &new_ppa, buf);
	if (ret)
		return ret;

	ret = journal_append(ftl->journal, lba, &old_ppa, &new_ppa);
	if (ret)
		return ret;

	ret = map_set_l2p(ftl->map, lba, &new_ppa);
	if (ret)
		return ret;

	block_invalidate_page(ftl->bm, &old_ppa);
	return 0;
}
问题 代码表现 后果
Program fail 无补偿 直接返回 NAND 错误。 Block Manager 不知道 PPA 分配失败、block 可能需要隔离。
Journal fail 产生 orphan program 成功后 journal 失败直接返回。 已写 page 没有 mapping，也没有被标记为 uncommitted。
Mapping fail 不可恢复 journal 已存在但 mapping 失败直接返回。 下次启动可能能重放，但系统没有标记 recovery required。
Invalidate fail 被忽略 旧页失效返回值未处理。 valid count 可能漂移，GC victim selection 失真。

改进代码：错误转换层与阶段化补偿

改进版把 NAND 结果先转换成 FTL 决策，并在每个事务阶段收敛到明确补偿点。 主路径仍然简洁，但错误路径不再隐形。

ftl_error.h：介质结果到 FTL 决策 C 
#ifndef FTL_ERROR_H
#define FTL_ERROR_H

#include <linux/types.h>
#include <linux/errno.h>

enum nand_op {
	NAND_OP_READ = 0,
	NAND_OP_PROGRAM,
	NAND_OP_ERASE,
};

enum ftl_error_action {
	FTL_ERR_NONE = 0,
	FTL_ERR_RETRY,
	FTL_ERR_RETIRE_BLOCK,
	FTL_ERR_DISCARD_PAGE,
	FTL_ERR_MARK_RECOVERY_REQUIRED,
	FTL_ERR_READ_RECLAIM,
};

struct nand_op_result {
	enum nand_op op;
	struct ftl_ppa ppa;
	int status;
	u32 ecc_corrected_bits;
	bool timeout;
	bool program_fail;
	bool erase_fail;
	bool uncorrectable;
};

struct ftl_error_decision {
	int errno_value;
	enum ftl_error_action action;
	const char *reason;
};

void ftl_classify_nand_result(const struct nand_op_result *result,
			      struct ftl_error_decision *decision);

#endif
ftl_write.c：按事务阶段补偿 C 
int ftl_submit_write(struct ftl_dev *ftl, const struct ftl_write_req *req)
{
	struct block_alloc_result alloc;
	struct ftl_mapping_delta delta;
	struct nand_op_result nand_result;
	struct ftl_error_decision decision;
	int ret;

	ret = ftl_prepare_write_delta(ftl, req, &delta, &alloc);
	if (ret)
		return ret;

	ret = nand_program_page(ftl->nand, &alloc.ppa, req->buf,
				req->len, &nand_result);
	if (ret)
		goto err_program;

	delta.new_ppa = alloc.ppa;

	ret = journal_append_mapping_delta(ftl->journal, &delta);
	if (ret)
		goto err_journal;

	ret = map_commit_delta_if_current(ftl->map, &delta);
	if (ret)
		goto err_mapping;

	ret = block_commit_programmed_page(ftl->bm, &alloc.ppa);
	if (ret)
		goto err_block_commit;

	if (ftl_ppa_valid(&delta.old_ppa))
		block_invalidate_page(ftl->bm, &delta.old_ppa);

	return 0;

err_block_commit:
	ftl_mark_recovery_required(ftl, FTL_RECOVERY_MAPPING_COMMITTED);
	return ret;

err_mapping:
	ftl_mark_recovery_required(ftl, FTL_RECOVERY_JOURNAL_REPLAY);
	return ret;

err_journal:
	block_discard_uncommitted_page(ftl->bm, &alloc.ppa);
	return ret;

err_program:
	ftl_classify_nand_result(&nand_result, &decision);
	block_apply_program_failure(ftl->bm, &alloc.ppa, &decision);
	return decision.errno_value;
}

错误矩阵与落点表

NAND 结果到 FTL 决策矩阵

NAND 结果 FTL 领域解释 负责模块 补偿动作 返回语义
Program fail new PPA 未提交，不可逻辑可见 Block Manager / Bad Block 标记 program failure，隔离候选 block，重新分配或失败返回 -EIO 或 -EAGAIN
Journal append fail page 已写但不可恢复 Journal / Block Manager 标记 uncommitted page，等待 GC 回收 -EIO、-ENOSPC
Mapping commit fail journal 已记录但内存状态未收敛 Mapping / Recovery 标记 recovery required，后续 replay delta -EIO
Read correctable 数据可用但介质健康下降 ECC / Read Path 记录 corrected bits，必要时 read reclaim 0
Read uncorrectable 当前副本不可用 Read Retry / Recovery 尝试 read retry、镜像副本、恢复扫描 -EILSEQ
Erase fail block 不能回 free pool Block Manager / Bad Block retire block，更新坏块统计和 free pool -EIO

四本书思想落点表

书 思想 SSD / FTL 设计位置 约束了什么 帮助避免什么问题
《软件设计哲学》 特殊情况不能扩散；深模块应隐藏错误细节。 NAND result classifier、Block Manager 补偿接口。 主路径不解释硬件状态位。 避免每个调用点写一套错误判断。
《程序员修炼之道》 契约明确，失败可诊断。 负 errno + decision/action 结构。 错误码必须附带可执行补偿语义。 避免 -EIO 成为语义黑洞。
《设计的设计》 概念完整性要求错误路径遵守同一事务模型。 program、journal、mapping、invalidate 的阶段边界。 每个阶段失败都有唯一补偿所有者。 避免补丁式 if/else 破坏恢复语义。
《领域驱动设计》 领域异常应使用领域语言表达。 orphan page、bad block candidate、recovery required。 错误路径要改变领域状态，而不仅是返回数字。 避免硬件错误码泄漏到上层业务判断。
