Day 1 · SSD / FTL 代码设计判断

模块边界先于函数实现：把 FTL Core 从 NAND 细节中切开

今天的核心不是“怎么写一个可运行的 ftl_write()”， 而是先判断哪些知识应该留在 FTL Core，哪些知识必须被 Block Manager、NAND、Mapping、 Journal、WriteBooster 吸收。这个判断会决定后续接口、错误路径、状态机和恢复机制是否能长期演进。

一句话判断 FTL Core 只编排领域动作 涉及：Block / NAND / Mapping / Journal / WriteBooster 关键词：深模块 · 正交性 · 概念完整性 · 限界上下文

今日设计判断

FTL Core 应该编排“写入、分配、program、journal、mapping commit、旧页失效、恢复标记”， 不应该直接读取 NAND 状态位、改 block table、遍历 mapping 数组或塞入 SLC cache 私有状态。

知识充分性判断

只知道“深模块、DRY、DDD 名词”不够。Day 1 需要理解四个最小概念： 复杂度如何泄漏、正交性如何被破坏、概念完整性如何约束取舍、领域语言如何进入接口。

今日阅读重点：够用但不空泛的四本书概念

Day 1 不需要把四本书读成四份摘要，但必须读到足以支撑一个工程判断： FTL Core 不拥有介质细节和空间生命周期细节。 下面这些概念，是今天做边界判断的最低知识量。

思想来源 今天必须掌握的概念 如果理解不足，会在 FTL 中变成什么问题
《软件设计哲学》 复杂度主要来自依赖、晦涩、变更放大。深模块不是函数长短，而是简单接口背后隐藏足够多的设计知识。 ftl_write() 需要理解 NAND status、open block cursor、valid bitmap、journal record 格式，调用者被迫吸收所有复杂度。
《程序员修炼之道》 DRY 指“知识不要重复”，不是“代码行不要重复”；正交性要求一个设计决策变化时，影响范围可预测。 GC、Recovery、WriteBooster、Host Write 各自维护 free block 低水位和坏块判断，最后同一规则出现多个版本。
《设计的设计》 设计是在约束下维护概念完整性。好设计不等于一次覆盖所有情况，而是先选出稳定主概念。 今天为 SLC cache 加一个 flag，明天为 GC relocation 加一个 flag，后天为 metadata write 再加一个 flag，主路径失去统一概念。
《领域驱动设计》 限界上下文决定模型在哪里有效；通用语言必须进入接口、结构体和错误语义，而不是停在文档里。 代码里只有 type、mode、flag， 设计讨论中的 host write、mapping commit、victim block、relocation 无法被代码审查。
对上一版内容的判断： 只有少量概念性描述是不够的。读者需要知道这些概念如何变成“谁拥有知识、谁暴露接口、谁处理错误、谁承担恢复语义”的代码判断。 因此本版把书本概念压缩成可操作的设计约束，并在后面的图、表和 C 代码里反复落地。

概念融合解释：边界是知识所有权，不是目录名

四本书在这里汇成一个判断：模块边界的本质是知识所有权。FTL Core 拥有流程编排知识； Block Manager 拥有空间和生命周期知识；NAND 层拥有介质操作知识；Mapping 拥有地址关系知识； Journal 拥有可恢复提交知识；WriteBooster 拥有 SLC cache 吸收和 flush 策略知识。

这个边界图同时体现四种约束：深模块要求每个模块隐藏自己的复杂度；正交性要求改动不会横向扩散； 概念完整性要求主路径始终是“写入事务编排”，不是“到处补介质特判”；DDD 要求每条边上的接口用 FTL 领域语言命名。

SSD / FTL 设计场景

SLC Cache manager / WriteBooster

FTL Core 不直接判断 SLC cache 是否满，也不维护 flush 游标。它提交写入意图； WriteBooster 返回“吸收到 SLC”“需要 flush”“降级到 TLC”的领域结果。

Free Block Pool 低水位

写路径不遍历 block table 找空闲块。Block Manager 返回 -ENOSPC 或低水位原因，FTL Core 再请求 foreground GC。

NAND Program / Erase / Read Flow

NAND 层保留寄存器、retry、timing 等细节；向上返回 program result、read reliability、erase outcome， 由 FTL 决定补偿动作。

GC 有效页搬移

GC 不直接相信 victim block 中的旧 PPA。它通过 Mapping 条件接口确认 old PPA 仍是当前映射， 再提交 relocation journal。

Bad Block Management

program fail 不是在写路径里直接 is_bad = 1。 坏块隔离、退役、替换和统计应由 Bad Block Management 或 Block Manager 的坏块子域处理。

Power Loss Recovery

Recovery 不依赖 FTL Core 的临时变量，而依赖 Metadata Journal 中可重放、可校验、可幂等的领域记录。

反例代码：FTL Core 吸收了所有知识

坏味道： 这个函数把 NAND 状态、SLC cache、free block、mapping、journal、GC、bad block 处理都塞进一个写路径。 它不是“快”，而是把未来所有变化都抵押给同一个函数。
反例：边界混乱的 ftl_write() C 
int ftl_write(struct ftl_dev *ftl, u64 lba, const void *buf, int len,
	      int use_slc, int sync)
{
	int blk;
	int page;
	int old_ppn;
	int new_ppn;
	int ret;

	if (!ftl || !buf || len & 511)
		return -1;

	if (ftl->nand->status & NAND_STATUS_BUSY)
		return -2;

	if (use_slc) {
		if (ftl->slc_cache.full)
			ftl->slc_cache.flush_needed = 1;

		blk = ftl->slc_cache.open_blk;
	} else {
		blk = ftl->cur_open_blk;
	}

	if (ftl->free_blk_cnt < 2)
		gc_run(ftl, 1, 0, 0);

	if (ftl->blk_tbl[blk].is_bad) {
		ftl->blk_tbl[blk].state = 9;
		ftl->cur_open_blk++;
		blk = ftl->cur_open_blk;
	}

	page = ftl->blk_tbl[blk].write_ptr++;
	new_ppn = blk * ftl->pages_per_blk + page;

	ret = nand_program(ftl->nand, blk, page, buf, len);
	if (ret) {
		if (ret == NAND_PROG_FAIL) {
			ftl->blk_tbl[blk].is_bad = 1;
			ftl->bad_blk_cnt++;
		}
		return ret;
	}

	old_ppn = ftl->l2p[lba];
	ftl->l2p[lba] = new_ppn;

	if (old_ppn >= 0) {
		int old_blk = old_ppn / ftl->pages_per_blk;

		ftl->blk_tbl[old_blk].valid_cnt--;
		ftl->blk_tbl[old_blk].invalid_cnt++;
	}

	if (sync)
		journal_append_raw(ftl->journal, lba, old_ppn, new_ppn,
				   ftl->blk_tbl[blk].erase_cnt);

	return 0;
}
真实工程坏味道 代码表现 后续代价
模块边界混乱 FTL Core 读写 nand->status、blk_tbl、slc_cache。 低层结构变化会强迫主路径改动，变更放大。
领域语言不清 use_slc、sync、state = 9 缺少领域含义。 新增 GC relocation、metadata write、flush policy 时继续堆 flag。
错误路径散落 program fail 在主路径里直接标坏块。 Bad Block、Recovery、Block 生命周期无法形成统一不变量。
提交顺序不可靠 先更新 L2P，再写 journal。 掉电后可能出现内存映射已变但 journal 不可重放的窗口。

改进代码：让接口表达领域边界

改进版不试图“一步完成产品级写路径”。它只确立 Day 1 的边界原则： FTL Core 编排事务，其他模块通过领域接口承接自己的知识。错误码使用接近 Linux kernel 的负 errno， 错误路径通过 goto 收敛到可演进的补偿点。

ftl_write.h：头文件暴露领域意图，而不是内部结构 C 
#ifndef FTL_WRITE_H
#define FTL_WRITE_H

#include <linux/types.h>
#include <linux/errno.h>

enum ftl_write_origin {
	FTL_WRITE_HOST = 0,
	FTL_WRITE_SLC_FLUSH,
	FTL_WRITE_GC_RELOCATE,
	FTL_WRITE_METADATA,
};

enum ftl_media_class {
	FTL_MEDIA_ANY = 0,
	FTL_MEDIA_SLC,
	FTL_MEDIA_TLC,
};

struct ppa {
	u32 ch;
	u32 lun;
	u32 blk;
	u32 page;
};

struct ftl_write_req {
	u64 lba;
	const void *buf;
	u32 len;
	enum ftl_write_origin origin;
	enum ftl_media_class preferred_media;
};

struct ftl_mapping_commit {
	u64 lba;
	struct ppa old_ppa;
	struct ppa new_ppa;
	enum ftl_write_origin origin;
};

int ftl_submit_write(struct ftl_dev *ftl, const struct ftl_write_req *req);

#endif
ftl_write.c：FTL Core 只编排领域动作 C 
static int ftl_validate_write_req(const struct ftl_write_req *req)
{
	if (!req || !req->buf)
		return -EINVAL;

	if (!req->len || req->len & 511)
		return -EINVAL;

	return 0;
}

int ftl_submit_write(struct ftl_dev *ftl, const struct ftl_write_req *req)
{
	struct block_alloc_req alloc_req;
	struct ftl_mapping_commit commit;
	struct nand_program_result prog;
	int ret;

	ret = ftl_validate_write_req(req);
	if (ret)
		return ret;

	ret = map_lookup_l2p(ftl->map, req->lba, &commit.old_ppa);
	if (ret && ret != -ENOENT)
		return ret;

	alloc_req.origin = req->origin;
	alloc_req.preferred_media = req->preferred_media;
	alloc_req.lba = req->lba;

	ret = block_alloc_page_for_write(ftl->bm, &alloc_req, &commit.new_ppa);
	if (ret == -ENOSPC) {
		ret = gc_request_foreground(ftl->gc, GC_CAUSE_FREE_POOL_LOW);
		if (ret)
			return ret;

		ret = block_alloc_page_for_write(ftl->bm, &alloc_req,
						 &commit.new_ppa);
	}
	if (ret)
		return ret;

	ret = nand_program_page(ftl->nand, &commit.new_ppa, req->buf,
				req->len, &prog);
	if (ret)
		goto err_program;

	commit.lba = req->lba;
	commit.origin = req->origin;

	ret = journal_append_mapping_commit(ftl->journal, &commit);
	if (ret)
		goto err_journal;

	ret = map_commit_l2p(ftl->map, req->lba, &commit.old_ppa,
			     &commit.new_ppa);
	if (ret)
		goto err_mapping;

	ret = block_commit_programmed_page(ftl->bm, &commit.new_ppa);
	if (ret)
		goto err_block_commit;

	if (ppa_is_valid(&commit.old_ppa))
		block_invalidate_page(ftl->bm, &commit.old_ppa);

	return 0;

err_block_commit:
	ftl_mark_recovery_required(ftl, FTL_RECOVERY_MAPPING_COMMITTED);
	return ret;

err_mapping:
	ftl_mark_recovery_required(ftl, FTL_RECOVERY_JOURNAL_REPLAY);
	return ret;

err_journal:
	block_discard_uncommitted_page(ftl->bm, &commit.new_ppa);
	return ret;

err_program:
	block_handle_program_result(ftl->bm, &commit.new_ppa, &prog);
	return ret;
}

FTL Core 写入事务顺序、模块编排、主路径语义。 submit write、request GC、mark recovery required。 NAND status bit、block table 布局、journal 编码。 -EINVAL、上游透传的负 errno。
Block Manager free block pool、open block、page/block 生命周期、valid/invalid 计数。 按写入意图分配、提交 programmed page、失效旧页。 Mapping、GC、WriteBooster 的内部策略。 -ENOSPC、-EUCLEAN、-EIO。
NAND Layer program/read/erase 时序、介质状态、retry 原始结果。 介质操作结果和可执行建议。 FTL Core、Mapping、Journal。 -EIO、-ETIMEDOUT、-EAGAIN。
Mapping Table LBA 到 PPA 的当前关系、条件提交、反查能力。 lookup、commit、compare-and-swap style update。 GC victim 策略、NAND 读写细节。 -ENOENT、-ESTALE。
Metadata Journal 可恢复记录、顺序、校验、重放幂等性。 append mapping commit、append relocation、replay。 FTL Core 的临时变量、Block Manager 内存布局。 -ENOSPC、-EIO。
WriteBooster SLC cache 吸收策略、flush 边界、降级条件。 选择写入介质、触发 flush、返回降级理由。 Host Write 主路径和 GC 内部流程。 -EAGAIN、-ENOSPC。

NAND 操作结果到 FTL 决策的映射

NAND 层结果 FTL 看到的领域结果 负责模块 后续动作
Program success new PPA 可进入 mapping commit FTL Core + Journal + Mapping 先写 journal，再提交 L2P，再提交 page 状态。
Program fail 候选 PPA 不可提交 Block Manager / Bad Block Management 退役或隔离 block，重新分配 page，必要时返回 -EIO。
Read correctable ECC 数据可信但需要健康反馈 Read Path / ECC / Wear Leveling 记录介质健康，必要时触发 read reclaim。
Read uncorrectable 当前副本不可用 Read Retry / Recovery 尝试 retry、镜像、日志恢复或上报不可恢复错误。
Erase fail block 不可回到 free pool Block Manager / Bad Block Management 隔离 block，更新 free pool 与坏块统计。

四本书思想落点表

书 思想 SSD / FTL 设计位置 约束了什么 帮助避免什么问题
《软件设计哲学》 复杂度应被深模块吸收，而不是由调用方理解。 FTL Core 与 Block / NAND / Mapping / Journal 的接口。 禁止高层直接读写低层内部结构。 避免主路径变成跨模块细节仓库。
《程序员修炼之道》 正交性与 DRY 要保护知识的单一归属。 free pool、valid count、journal commit、bad block 标记。 同一规则只能由一个模块拥有。 避免 GC、Recovery、WriteBooster 重复实现空间规则。
《设计的设计》 概念完整性比临时功能拼接更重要。 写入主路径的提交顺序与模块协作概念。 接口必须服务于稳定主概念，而不是临时 flag。 避免每个新需求都破坏原有设计语言。
《领域驱动设计》 领域语言应该进入代码边界。 LBA、PPA、mapping commit、relocation、victim block、program result。 接口命名必须表达业务意图。 避免 type/mode/flag 代码让领域判断不可审查。
