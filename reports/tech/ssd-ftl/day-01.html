Day 1 · Reading First

为什么可工作的代码还不够：从“能跑”进入“能解释、能修改、能演进”

今天不是把四本书各抽一句口号，而是建立整个 15 天课程的判断基准：代码能通过测试，只说明它在当前场景下能运行； 设计是否合格，要看后来的人能否理解它、定位变化影响、解释模型假设，并在约束变化时安全修改。

今日主问题：可工作为什么不等于设计好 阅读主轴：四本书导论与问题定义 练习场：Host Write 主路径

今日阅读范围

Day 1 只做一件事：读懂四本书各自为什么要谈“设计”。不要急着把所有概念套到 SSD / FTL。 今天要先分清四本书的入口问题。

书 今天阅读的主题块 读完要能回答的问题
《软件设计哲学》 导论、复杂度的性质、为什么“工作代码”不够。 复杂度从哪里来？为什么 tactical programming 会让系统越来越难改？
《程序员修炼之道》 务实程序员的责任、软件熵、足够好、知识组合、沟通。 工程师为什么要主动管理反馈、质量边界和自己的知识库存？
《设计的设计》 设计问题是什么、理性模型为何不足、设计如何在约束中探索。 为什么设计不是按步骤推导出来的，而是不断发现约束、修正方案的过程？
《领域驱动设计》 知识消化、模型与沟通、模型为什么必须能进入实现。 模型为什么不是图上的名词，而是团队理解问题并写出代码的共同语言？

今日新增概念：四本书不是同一句话

复杂度是认知成本

《软件设计哲学》关注的是维护者要理解多少依赖、例外、隐藏状态和不明显决策。复杂度不是代码行数，而是修改时必须装进脑子的事实数量。

务实是持续反馈

《程序员修炼之道》不是教人写聪明代码，而是要求工程师对结果负责，用反馈、沟通和知识更新降低长期失控。

设计是约束探索

《设计的设计》提醒我们，设计不是一次性从需求推导实现。真实设计会不断暴露新约束，因此需要保留比较、复盘和修正空间。

模型是工作假设

《领域驱动设计》把模型看成沟通和实现之间的桥。模型如果不能帮助讨论、命名、编码和验证，就只是装饰。

今天的递进： 旧版页面直接从“模块边界”开始，这是跳步。新版本先建立判断标准：我们为什么要读设计书？ 答案不是为了收集原则，而是为了在真实代码中识别不可见的认知成本、反馈缺失、约束变化和模型贫血。

概念慢读：今天到底在读什么

如果 Day 1 只记住“复杂度不好”“要有模型”，那它和旧版没有差别。今天真正要读的是四本书对“好代码”的不同入口定义。 它们并不是在重复同一条原则，而是在回答四个层级的问题：代码结构为什么变难、工程师如何对长期结果负责、 设计过程为什么不能假装全知，以及模型如何把问题理解压进实现。

1. 《软件设计哲学》：复杂度不是抽象道德词

这本书的起点不是“代码要优雅”，而是维护者的认知负担。一个系统变难，不一定因为代码很多， 而是因为一个小改动要求你同时理解太多事实：某个标志位在三个路径里被解释、某个返回码在正常路径和恢复路径里含义不同、 某个结构体字段看似公开但其实只有一个模块可以安全修改。复杂度的危险在于它常常不可见。 代码第一次写出来时，作者知道全部上下文；半年后，维护者只看到字段、分支、异常路径和测试失败。

所以 Day 1 读这本书时，不要急着寻找“深模块”结论。先学会识别复杂度的三种早期信号： 第一，修改前必须读很多无关文件；第二，调用一个函数前必须知道它内部的状态假设；第三，出现问题时无法从错误返回和日志判断系统停在哪一步。 这些信号比“代码是否好看”更可靠。

2. 《程序员修炼之道》：务实不是短平快

这本书的入口是工程师的责任感。务实不是“先跑起来再说”，而是知道当前决策会对未来反馈产生什么影响。 当你把所有错误都返回 -EIO，你不是简化了错误处理，而是破坏了反馈； 当你把多个语义塞进一个 flags，你不是提高了灵活性，而是让未来每次修改都要重新推理组合规则。

Day 1 只要求建立一个判断：务实程序员不是反设计，而是反空想。真正务实的设计必须能产生反馈。 测试能反馈行为是否破坏；日志能反馈系统停在哪个阶段；接口能反馈调用者是否正确表达意图；代码审查能反馈模型是否被团队共同理解。 如果一个设计让反馈渠道消失，它即使短期能跑，也不是务实。

3. 《设计的设计》：设计不是事后包装的完美路线

这本书提醒我们，真实设计不是先拥有完整需求，再推导出最优结构。设计经常是在不知道全部约束时开始的。 你先做一个方案，才发现性能约束；你加入恢复逻辑，才发现之前的状态模型不够；你接入 GC，才发现 mapping 接口没有表达旧页失效。 如果报告把设计讲成“从原则直接推出代码”，读者会误以为设计是一次性正确，而不是持续暴露约束、比较方案、修正模型。

因此 Day 1 需要读出“设计记录”的重要性。一个教学报告不能只给最后答案，还要告诉读者为什么旧方案不够、 新方案解决了哪个约束、还留下哪些问题到后续章节处理。这样读者才会看到概念递进，而不是每天看到同一组漂亮结论。

4. 《领域驱动设计》：模型不是名词表

DDD 的入口是知识消化。很多代码里有 LBA、PPA、block、page 这些名词，但这不代表有领域模型。 真正的模型会改变系统的说话方式和编码方式：团队讨论时用同一个词表达同一件事，函数参数能体现这个词的边界， 错误码能体现领域状态，测试能描述领域场景，日志能帮助恢复领域事实。

所以 Day 1 的模型问题不是“我们能不能画出 FTL 的对象图”，而是“代码是否帮助团队理解写入事务”。 如果程序里只有 mode、type、 flags，而没有 write intent、programmed page、commit record、orphan page 这样的概念， 那么模型还没有进入实现。

四书差异：它们从四个方向逼近同一个问题

如果只记成口号 今天真正要学到
“复杂度不好。” 要能指出复杂度具体藏在哪：依赖、晦涩、状态、例外路径、跨模块知识。
“要务实。” 要能解释当前设计有什么反馈渠道：测试、日志、代码审查、用户结果、维护者理解。
“设计要完整。” 要能记录备选方案、约束和被放弃的路径，而不是把最后结果包装成唯一正确。
“要有领域模型。” 要能判断模型是否真的改变了命名、接口、数据结构和讨论方式。

误区辨析：这些说法看起来对，但教学价值很低

常见说法 为什么不够 应该推进到哪里
“代码要降低复杂度。” 这是结论，不是教学。读者不知道复杂度在当前代码中具体是什么。 指出维护者必须同时知道哪些事实，并说明哪些事实可以被模块收口。
“要有清晰模型。” 这容易退化成画图和命名，不能保证模型真的改变实现。 检查模型是否进入类型、接口、状态、错误、日志和测试。
“先写能跑的版本。” 如果没有记录阶段和约束，能跑版本会变成后续所有复杂度的来源。 保留可解释的主路径骨架，并明确哪些问题留给 Day 2-4 继续处理。
“设计是为了以后扩展。” 这太模糊。扩展什么、因为什么变化、由谁修改，都没有定义。 把变化来源写清楚：介质错误、恢复要求、GC 介入、映射一致性、性能策略。

SSD / FTL 练习：用 Host Write 检验“可工作还不够”

今天不要求你完成 FTL 架构，只要求你看一条 Host Write 主路径，并问四个问题： 维护者要知道多少隐藏事实？错误反馈在哪里？哪些设计约束没有被记录？代码里的名词是否真的对应领域模型？

检查角度 在 Host Write 中要看什么 不合格信号
复杂度 写路径是否必须知道 NAND 状态位、block 表内部字段、journal 格式和 mapping 更新细节。 一个函数同时分配 page、program NAND、写 journal、改 mapping、处理坏块。
反馈 失败后能否知道是空间不足、介质失败、journal 失败还是 mapping 竞争。 所有失败都返回 -EIO 或 false。
约束 代码是否记录了“program 成功但 mapping 未提交”这类中间事实。 只有 success/fail 两态，掉电后无法解释写入停在哪一步。
模型 LBA、PPA、open block、write intent、commit record 是否成为代码中的一等概念。 接口使用 type、mode、flags 传递所有语义。

案例推导：Host Write 为什么会从主路径变成复杂度源头

假设你现在只想实现一个最小 Host Write：给定 LBA 和数据，找一个物理 page，写入 NAND，更新 mapping。 第一眼看，这是一条线性流程。很多坏设计正是从这个“第一眼”开始的，因为线性流程掩盖了三个事实： NAND program 和 mapping commit 不是同一件事；旧页失效和新页可见不是同一时刻；journal 记录不是附属动作，而是掉电后解释系统状态的证据。

阶段 1：能跑版本

最初的能跑版本通常只有一个函数：分配、写入、更新、返回。这个版本可以通过简单测试，因为测试只验证写完以后读回数据。 但它没有回答“中途失败怎么办”。如果 program 成功但 journal 失败，物理介质上已经有一份数据，但 mapping 还没承认它。 如果此时掉电，恢复流程看到这个 page 时，不能把它当成 free，也不能直接当成有效数据。能跑版本没有保存这个中间事实， 于是复杂度被推迟到 recovery，后者只能猜。

阶段 2：补丁版本

下一步常见补丁是加几个字段：programmed、 journaled、valid。 这看似解决问题，实际上把模型散开了。Write Path 负责设置 programmed，Journal Path 负责设置 journaled， Mapping Path 负责设置 valid，Recovery 再组合这些 bool 推断状态。每个模块都知道一点真相，但没有一个模块拥有完整生命周期。

阶段 3：可解释版本

Day 1 不要求最终状态机，但要求你看见可解释版本的方向：把主路径拆成能够命名的阶段。 Allocated 表示已经占用物理位置；Programmed 表示 NAND 写入成功但未对外可见；Journaled 表示恢复证据已经持久化； Committed 表示 mapping 已经承认新 PPA。这样每个阶段都不是装饰性名字，而是恢复、错误处理和后续模块协作的边界。

阶段 系统已经知道什么 系统还不能假设什么 如果失败，谁应该解释
Allocated 已经选出一个 PPA，空间被预留。 不能假设 NAND 已写入，也不能让 mapping 可见。 Block Manager 解释空间和分配状态。
Programmed NAND program 成功，介质上可能存在新数据。 不能假设主机读应该看到这份数据。 Media / Recovery 共同解释未提交数据。
Journaled 恢复流程有证据重放或回滚该写入。 不能假设 mapping 已完成条件更新。 Journal / Recovery 解释重放边界。
Committed LBA 的当前版本已经指向新 PPA。 旧页失效、valid count、GC 可见性仍需对应模块维护。 Mapping / Block Manager 解释一致性。

这一段推导就是 Day 1 和旧版报告的关键差异：不是把“模型”“复杂度”“反馈”贴到 FTL 上， 而是从一个能跑函数出发，展示它为什么不足、补丁为什么仍不足、可解释结构为什么是后续学习的入口。

反例与改写：先看复杂度如何泄漏

反例目标： 这段代码不是因为短而坏，而是因为它把太多事实压进同一个调用点。它可以工作，但维护者必须同时理解空间管理、介质错误、日志、映射和旧页失效。
反例：可工作但不可解释的写路径 C 
int ftl_write_bad(struct ftl_dev *ftl, u64 lba, const void *buf)
{
	struct ftl_block *blk;
	struct ftl_page *page;
	int ret;

	blk = ftl->open_blk;
	if (!blk || blk->bad || blk->free_pages == 0)
		return -EIO;

	page = &blk->pages[blk->write_pos++];
	page->lba = lba;
	page->valid = false;

	ret = nand_program(ftl->nand, blk->id, page->id, buf);
	if (ret) {
		blk->bad = true;
		return -EIO;
	}

	ret = journal_write(ftl->journal, lba, blk->id, page->id);
	if (ret)
		return -EIO;

	map_set(ftl->map, lba, blk->id, page->id);
	page->valid = true;
	blk->valid_pages++;
	blk->free_pages--;
	return 0;
}
今天的书本概念 反例暴露的问题
复杂度是依赖和晦涩 调用点必须理解五个模块的内部顺序和失败含义。
务实要求反馈 所有失败都压成 -EIO，后续无法知道要重试、退役、恢复还是上报。
设计需要记录约束 没有表达“已 program 但未 commit”的中间事实。
模型必须进入实现 代码没有 write intent、program result、commit record 这些领域名词。
改写：Day 1 只建立可解释的主路径骨架 C 
enum ftl_write_stage {
	FTL_WRITE_ALLOCATED = 0,
	FTL_WRITE_PROGRAMMED,
	FTL_WRITE_JOURNALED,
	FTL_WRITE_COMMITTED,
};

struct ftl_write_intent {
	u64 lba;
	const void *buf;
	size_t len;
};

struct ftl_write_ctx {
	struct ftl_ppa ppa;
	enum ftl_write_stage stage;
	int media_status;
};

int ftl_submit_host_write(struct ftl_dev *ftl,
			  const struct ftl_write_intent *intent)
{
	struct ftl_write_ctx ctx = { };
	int ret;

	ret = block_alloc_for_host_write(ftl->blocks, intent, &ctx.ppa);
	if (ret)
		return ret;
	ctx.stage = FTL_WRITE_ALLOCATED;

	ret = media_program_user_page(ftl->media, &ctx.ppa, intent->buf,
				      intent->len, &ctx.media_status);
	if (ret)
		return block_handle_program_failure(ftl->blocks, &ctx);
	ctx.stage = FTL_WRITE_PROGRAMMED;

	ret = journal_record_host_write(ftl->journal, intent, &ctx.ppa);
	if (ret)
		return recovery_mark_uncommitted_write(ftl->recovery, &ctx);
	ctx.stage = FTL_WRITE_JOURNALED;

	ret = mapping_commit_host_write(ftl->map, intent->lba, &ctx.ppa);
	if (ret)
		return ret;
	ctx.stage = FTL_WRITE_COMMITTED;

	return block_commit_host_page(ftl->blocks, &ctx.ppa);
}

这还不是最终设计。Day 1 的目的只是让主路径能说清楚每一步的领域含义。 后续几天会继续推进：Day 2 处理语言和命名，Day 3 处理接口深度，Day 4 处理错误边界。

代码走读：不是“改成多个函数”这么简单

很多教学材料会在这里犯一个错误：展示一段坏代码，再展示一段“看起来更架构化”的好代码，但不解释中间的判断过程。 这样读者只会模仿形式，比如把一个函数拆成五个函数，却不知道每个函数应该拥有哪类知识。 下面把 Day 1 的反例逐段拆开，说明每一步对应四本书中的哪个入口问题。

反例前半段：分配和介质写入混在一起

反例中的 blk = ftl->open_blk 看起来只是取一个当前 block， 但它把一个重要假设藏进调用点：系统永远只有一个适合 host write 的 open block。 一旦后续加入冷热数据、SLC cache、GC relocation 或 wear leveling，这个假设就会破裂。 《软件设计哲学》会把它视为信息泄漏，因为调用点知道了 block 选择策略； 《程序员修炼之道》会把它视为反馈风险，因为策略变化后很难定位哪些调用点依赖了这个假设； 《设计的设计》会追问有没有比较过“单 open block”和“按写入意图选择 block”的方案； DDD 会追问这里是否缺少“write intent”这个领域概念。

再看 blk->bad || blk->free_pages == 0。 这不是普通防御式检查，而是把 block 可用性的判断散到了写路径。 如果 bad block 处理策略以后变成“隔离、重试、迁移可读页、更新统计”，这个 if 分支就无法表达新语义。 Day 1 不要求你立刻设计完整 Bad Block Management，但要求你看见：一个 bool 检查可能已经把未来设计空间压窄了。

反例中段：program 成功后缺少可解释阶段

nand_program() 成功之后，系统进入了一个很危险的中间状态： 物理介质上已经有新数据，但 mapping 还没承认它，journal 也未必记录了它。 如果教学报告只说“这里需要 journal”，还不够。真正要讲清楚的是：为什么这个中间状态必须被命名。 没有名字，错误路径只能写成“如果失败就返回”；有名字，恢复流程才能说“这是一个 programmed but uncommitted page”。

这就是模型进入实现的第一步。模型不是在图上画一个 Page 节点，而是让代码能表达“Page 现在处于哪个事务阶段”。 当 FTL_WRITE_PROGRAMMED 出现时，它不是为了漂亮，而是为了让 recovery、journal、mapping 和 block manager 围绕同一个事实说话。

反例后半段：mapping 可见性与旧页失效被压扁

反例最后直接调用 map_set()，然后把 page 标成 valid。 这会掩盖两个问题。第一，mapping 更新通常不是简单赋值，它可能需要条件更新：只有当旧映射仍然指向预期版本时，新 PPA 才能成为当前版本。 第二，新页 valid 后，旧页如何 invalid、valid count 如何变化、GC 何时能看到旧页，都不是写路径可以随意决定的内部细节。

因此改写版本中的 mapping_commit_host_write() 和 block_commit_host_page() 是两个不同动作。 前者表达 LBA 当前版本的可见性，后者表达物理页生命周期的提交。 这一区分为后续 Day 5 的 Mapping 所有权和 Day 7 的 Page 状态机埋下基础。

选择 open_blk 找一个可写 block。 block 选择策略泄漏到写路径。 Day 2-3 讲语言和深接口，Day 11 讲策略分离。
检查 bad/free_pages 判断是否能继续写。 可用性规则被简化成局部 bool。 Day 4 讲错误边界，Day 14 讲 Bad Block Management。
nand_program() 写入介质。 program 成功但未提交的状态没有命名。 Day 6-9 讲错误路径、状态机、journal、recovery。
map_set() 更新 L2P。 mapping 可见性和物理页生命周期混在一起。 Day 5 讲 Mapping 所有权，Day 7 讲 Page 生命周期。

读完这一节，你应该能看出 Day 1 的核心并不是“把代码拆函数”，而是建立一种阅读能力： 看到一行短代码时，能追问它背后压缩了哪些未命名事实。后续 14 天会逐步给这些事实找到合适的模块、语言、状态和契约。

阅读笔记任务：把四本书读成可检查的问题

今天读完后，不要写“我学到复杂度很重要”。这样的笔记无法指导代码审查。你应该写出可检查的问题， 并且每个问题都能拿到 FTL 代码里试用。

书 今天的笔记格式 示例
《软件设计哲学》 写一个“维护者必须知道”的清单。 调用 ftl_write() 前是否必须知道 open block 的内部状态？
《程序员修炼之道》 写一个“反馈是否足够”的清单。 journal 失败后，日志和错误码能否让恢复流程知道写入停在 Programmed 阶段？
《设计的设计》 写一个“当前方案暴露了什么新约束”的清单。 加入 power loss recovery 后，原本的 success/fail 两态为什么不够？
《领域驱动设计》 写一个“模型是否进入实现”的清单。 代码里是否有 commit record、orphan page、write intent，还是只有 flags？

自测题：确认你不是只记住了口号

  • 问题 1： 如果一段代码可以通过所有当前测试，为什么仍可能是坏设计？
    参考要点： 测试只能证明当前行为；坏设计的风险在未来修改时暴露，表现为依赖太多、语义不清、错误不可解释、模型不能指导变化。
  • 问题 2： 《软件设计哲学》和《程序员修炼之道》今天的关注点有什么不同？
    参考要点： 前者分析复杂度如何进入软件结构，后者强调工程师用责任、反馈、沟通和知识更新管理长期风险。
  • 问题 3： 为什么 DDD 中的模型不是“画出几个名词”？
    参考要点： 模型必须改变沟通和实现：命名、接口、数据结构、边界和测试都应该能看到模型。
  • 问题 4： 在 Host Write 反例中，哪个问题最能说明“能跑不等于能解释”？
    参考要点： journal 失败和 program 成功之间缺少可解释状态；掉电恢复无法知道写入停在哪一步。
