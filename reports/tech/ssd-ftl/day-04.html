Day 4 · SSD / FTL 代码设计判断

Host Write 曳光弹：用一条真实主路径逼设计落地

今天的核心判断是：Host Write 的第一版不应该是 demo，也不应该是万能大函数。 它应该是一条窄但真实的主路径，穿过请求校验、Mapping lookup、Block allocation、NAND program、 Metadata Journal、Mapping commit、旧页失效和错误补偿。

一句话判断 曳光弹必须穿过真实架构层次 涉及：FTL Core / Mapping / Block Manager / NAND / Journal 关键词：Tracer Bullet · 深模块 · 可恢复提交 · 主路径骨架

今日设计判断

Host Write 主路径只编排领域动作，不直接操作 block table、L2P 数组、NAND 寄存器或 journal buffer。 第一条真实路径要为 GC、Recovery、Bad Block、WriteBooster 留出接入点。

今日涉及模块

FTL Core、Mapping Table、Metadata Journal、Block Manager、NAND Program Flow、 Bad Block Management、Power Loss Recovery、Free Block Pool。

今日阅读重点：曳光弹不是降低设计要求

Host Write 是最适合作为 FTL 曳光弹的路径，因为它天然穿过多个关键上下文。 这条路径越早真实，设计越早暴露约束；但它也越容易腐化成“先跑通再说”的上帝函数。

思想来源 今天必须掌握的概念 落到 SSD / FTL 的判断
《软件设计哲学》 真实主路径仍然要通过深模块协作，不能把所有细节塞进一个函数。 FTL Core 编排流程，Block Manager、Mapping、NAND、Journal 各自隐藏复杂度。
《程序员修炼之道》 Tracer Bullet 要穿过真实系统，而不是 mock。它可以窄，但不能假。 第一条 Host Write 要真实执行 program、journal 和 mapping commit，而不是只更新内存数组。
《设计的设计》 设计是在真实约束中探索。主路径会暴露提交顺序、掉电窗口和补偿动作。 journal 应在 mapping commit 前，program 成功但 journal 失败时必须处理 orphan page。
《领域驱动设计》 主路径必须使用领域语言表达：host write、mapping delta、program commit、old page invalidation。 接口名称和结构体要描述领域动作，而不是抽象成 step1() 或 flags。

概念融合解释：真实路径承载未来复杂度

一条合格的 Host Write 曳光弹，不是“少做功能”，而是用最小范围承载真实架构。 它应该明确每个阶段的所有者和失败后果。主路径越清晰，后续 GC、Recovery、Bad Block 和 WriteBooster 越容易接入。

统一判断： Host Write 的价值不只是写入成功，而是把 FTL 的核心事务边界固定下来： NAND program 只是介质动作，journal 才提供可恢复承诺，mapping commit 才改变逻辑可见状态。

SSD / FTL 设计场景

SLC Cache / WriteBooster

Host Write 可能先落入 SLC，但主路径仍应提交 mapping delta；后续 flush 是另一个领域动作，不应改写 host write 语义。

Free Block Pool 低水位

分配失败时 Host Write 不直接扫描 block table，而是接受 Block Manager 的 -ENOSPC 和 GC 建议。

NAND Program Failure

Program fail 发生在 mapping commit 前，新 PPA 不可见；错误补偿应报告 Block Manager，而不是直接改 L2P。

Metadata Journal Failure

Program 成功但 journal 失败时，新页已写但未提交，必须被标为 uncommitted/orphan，等待 GC 回收。

Power Loss Recovery

掉电后只重放已写入 journal 的 mapping delta；没有 journal 的 programmed page 不应成为当前映射。

GC 协作

Host Write 产生旧页失效，Block Manager 更新 valid/invalid 计数，为后续 victim selection 提供真实统计。

反例代码：跑通了，但事务边界消失了

坏味道： 这个写路径直接操作 mapping、block table 和 NAND，提交顺序错误，错误路径没有可恢复语义。
反例：浅模块式 Host Write C 
int ftl_write(struct ftl_dev *ftl, u64 lba, const void *buf, u32 len)
{
	int blk = ftl->open_blk;
	int page = ftl->blk_tbl[blk].write_ptr++;
	int new_ppn = blk * ftl->pages_per_blk + page;
	int old_ppn;
	int ret;

	if (!buf || len != FTL_PAGE_SIZE)
		return -EINVAL;

	if (ftl->blk_tbl[blk].is_bad)
		return -EIO;

	old_ppn = ftl->l2p[lba];
	ftl->l2p[lba] = new_ppn;

	ret = nand_program(ftl->nand, blk, page, buf, len);
	if (ret) {
		ftl->l2p[lba] = old_ppn;
		ftl->blk_tbl[blk].is_bad = 1;
		return ret;
	}

	if (old_ppn >= 0) {
		ftl->blk_tbl[old_ppn / ftl->pages_per_blk].valid_cnt--;
		ftl->blk_tbl[old_ppn / ftl->pages_per_blk].invalid_cnt++;
	}

	journal_write(ftl->journal, lba, old_ppn, new_ppn);
	return 0;
}
问题 代码表现 后果
提交顺序错误 先改 L2P，再 program，再写 journal。 掉电后内存状态和 journal 状态无法对应。
模块边界混乱 写路径直接操作 open block、write pointer、valid count。 Block Manager 不再拥有 block 生命周期不变量。
错误路径粗糙 program fail 直接标坏块。 没有区分可重试、需要 retire、需要 read reclaim 的介质结果。
Journal 语义太晚 journal 在 mapping 可见后写入。 Power Loss Recovery 不能以 journal 作为权威提交记录。

改进代码：主路径只编排事务阶段

改进版的重点不是代码更多，而是每个阶段都有清晰所有者。FTL Core 编排顺序； Mapping、Block Manager、NAND 和 Journal 分别承担自己的契约。

ftl_write.h：Host Write 请求与提交结构 C 
#ifndef FTL_WRITE_H
#define FTL_WRITE_H

#include <linux/types.h>
#include <linux/errno.h>

struct ftl_write_req {
	u64 lba;
	const void *buf;
	u32 len;
};

struct ftl_mapping_delta {
	u64 lba;
	struct ftl_ppa old_ppa;
	struct ftl_ppa new_ppa;
	enum ftl_commit_reason reason;
};

int ftl_submit_host_write(struct ftl_dev *ftl,
			  const struct ftl_write_req *req);

#endif
ftl_write.c：真实但窄的 Host Write 曳光弹 C 
int ftl_submit_host_write(struct ftl_dev *ftl,
			  const struct ftl_write_req *req)
{
	struct block_alloc_req alloc_req;
	struct block_alloc_result alloc;
	struct ftl_mapping_delta delta;
	struct nand_program_result prog;
	int ret;

	ret = ftl_validate_write_req(req);
	if (ret)
		return ret;

	ret = map_lookup_l2p(ftl->map, req->lba, &delta.old_ppa);
	if (ret && ret != -ENOENT)
		return ret;

	alloc_req.lba = req->lba;
	alloc_req.origin = FTL_WRITE_HOST;
	alloc_req.page_role = FTL_PAGE_USER_DATA;
	alloc_req.preferred_media = FTL_MEDIA_ANY;
	alloc_req.policy = BLOCK_ALLOC_NORMAL;

	ret = block_alloc_page_for_write(ftl->bm, &alloc_req, &alloc);
	if (ret)
		return ret;

	ret = nand_program_page(ftl->nand, &alloc.ppa, req->buf,
				req->len, &prog);
	if (ret)
		goto err_program;

	delta.lba = req->lba;
	delta.new_ppa = alloc.ppa;
	delta.reason = FTL_COMMIT_HOST_WRITE;

	ret = journal_append_mapping_delta(ftl->journal, &delta);
	if (ret)
		goto err_journal;

	ret = map_commit_delta(ftl->map, &delta);
	if (ret)
		goto err_mapping;

	block_commit_programmed_page(ftl->bm, &alloc.ppa);
	if (ftl_ppa_valid(&delta.old_ppa))
		block_invalidate_page(ftl->bm, &delta.old_ppa);

	return 0;

err_mapping:
	ftl_mark_recovery_required(ftl, FTL_RECOVERY_JOURNAL_REPLAY);
	return ret;

err_journal:
	block_discard_uncommitted_page(ftl->bm, &alloc.ppa);
	return ret;

err_program:
	block_handle_program_result(ftl->bm, &alloc.ppa, &prog);
	return ret;
}

Host Write 事务阶段表

阶段 拥有模块 成功后状态 失败补偿
Lookup old mapping Mapping Table 获得 old PPA 或无旧映射 非 -ENOENT 直接返回
Allocate new page Block Manager new PPA reserved for write -ENOSPC 交给 GC 策略

Journal append Metadata Journal mapping delta 可重放 丢弃 uncommitted page
Mapping commit Mapping Table LBA 指向 new PPA 标记 recovery required

四本书思想落点表

书 思想 SSD / FTL 设计位置 约束了什么 帮助避免什么问题
《软件设计哲学》 深模块协作，主路径不吸收内部细节。 FTL Core 到 Mapping/Block/NAND/Journal 的接口。 Host Write 不直接修改内部表。 避免主路径变成复杂度聚集点。
《程序员修炼之道》 Tracer Bullet 穿过真实架构层。 Host Write 第一条真实路径。 不能只 mock mapping 或跳过 journal。 避免后期才发现恢复和错误路径不可接入。
《设计的设计》 用真实约束探索设计取舍。 program、journal、mapping commit 的顺序。 逻辑可见性必须晚于可恢复记录。 避免掉电窗口无法解释。
《领域驱动设计》 主路径以领域语言表达。 mapping delta、program result、uncommitted page。 接口名必须表达领域动作。 避免 step/flag 代码不可审查。
