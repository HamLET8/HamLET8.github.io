Day 2 · SSD / FTL 代码设计判断

建立 FTL 通用语言：不要让 type、mode、flag 偷走设计判断

今天的核心判断是：FTL 的领域词汇必须进入代码边界。 lba、ppa、open block、 victim block、mapping commit、 relocation 不是注释里的术语，而应该成为 struct、enum、接口名、journal record 和错误语义。

一句话判断 用领域词汇替代 type / mode / flag 涉及：Mapping / GC / Journal / Block Manager / WriteBooster 关键词：通用语言 · 限界上下文 · DRY · 概念完整性

今日设计判断

如果一个接口需要调用者传 type = 2、 mode = 1、flags = 0x10， 它已经把领域判断压扁成参数猜谜。FTL 代码应该把 host write、GC relocation、SLC flush、metadata commit 明确建模。

今日涉及模块

Mapping Table、Block Manager、GC、Metadata Journal、SLC Cache manager / WriteBooster、 FTL Recovery、NAND Program / Erase / Read Flow。

今日阅读重点：通用语言不是命名美化

Day 2 要掌握的知识，不是“变量名起得好一点”。在 SSD / FTL 里，语言直接决定模块能不能保持边界： 同一个词在不同模块里含义不一致，错误路径、恢复语义和 GC 协作都会变得不可审查。

思想来源 今天必须掌握的概念 落到 SSD / FTL 的判断
《软件设计哲学》 晦涩是复杂度的一种。调用者必须记住 magic number、flag 组合和隐含前置条件时，复杂度已经从模块内部泄漏到接口。 用 struct ftl_mapping_delta 表达 mapping 变更， 比 update_map(lba, old, new, 3) 更能隔离复杂度。
《程序员修炼之道》 DRY 约束的是知识重复。通用语言可以让“什么是 relocation”“什么是 committed mapping”只出现一个定义。 GC、Recovery、Journal 如果各自解释 relocation，就会出现三套恢复边界；应由共享领域结构和接口统一。
《设计的设计》 概念完整性要求设计有少数稳定主概念。名词太少会退化成 flag，名词太多会变成伪模型。 Day 2 只固定核心词汇：LBA、PPA、write origin、page role、mapping delta、journal record、relocation intent。
《领域驱动设计》 通用语言必须在限界上下文内有效。Host 视角的 LBA、NAND 视角的 PPA、GC 视角的 victim block 不能混成一个泛化 ID。 接口和结构体要体现上下文：Mapping 负责 LBA/PPA 关系，Block Manager 负责 block 生命周期，Journal 负责可恢复事件。

概念融合解释：词汇必须绑定责任

通用语言不是把缩写写进代码，而是把领域判断压进类型、接口和记录格式。 FTL 代码里最危险的语言坏味道，是一个词没有稳定边界：page 有时表示 NAND page，有时表示 logical page，有时表示 cache entry；block 有时表示 open block，有时表示 victim block，有时表示 bad block candidate。

统一判断： 每个核心词汇都要回答三个问题：它在哪个上下文有效，谁拥有它的不变量，哪个接口可以改变它。 回答不了这三个问题的词汇，不应该进入公共头文件。

这张图的重点是：语言不是文档层装饰，而是工程路径。讨论中的词汇如果没有进入 enum、struct、接口和 journal record， 它就不能约束实现；实现里如果只剩 flag，设计讨论也会失去抓手。

SSD / FTL 设计场景

Mapping Table

不要用 update(lba, ppa, type)。 用 mapping_delta 表达 old PPA、new PPA、origin 和提交语义。

GC Relocation

GC 搬移有效页不是普通 write。它应被命名为 relocation，并要求 Mapping 做条件提交， 避免覆盖 host write 的更新。

Metadata Journal

Journal 记录的不是“写了几个数字”，而是 mapping commit、relocation commit、block erase commit 这些可重放事件。

WriteBooster

SLC cache 写入不应只是 mode = FAST。 它是 host write 的吸收路径，后续 flush 到 TLC 时语义要保持可追踪。

Bad Block Management

program fail candidate、confirmed bad block、retired block 不是同一个状态。 语言不分清，错误路径就会过早退役或重复分配。

Power Loss Recovery

Recovery 只能重放稳定语言。若 journal 里只有 type 和 flag，恢复代码就必须猜测掉电前的领域意图。

反例代码：参数看似通用，实际让语义消失

坏味道： 这个例子把 host write、GC relocation、SLC flush、metadata write 全部压进 type/mode/flags。 调用点短，系统语义却散了；错误路径和 Recovery 都只能反向猜测参数组合的含义。
反例：领域语言被 magic 参数吞掉 C 
#define WRITE_TYPE_HOST		0
#define WRITE_TYPE_GC		1
#define WRITE_TYPE_META		2
#define WRITE_MODE_FAST		1
#define WRITE_MODE_SAFE		2
#define WRITE_FLAG_SYNC		0x01
#define WRITE_FLAG_MOVE		0x02
#define WRITE_FLAG_CACHE	0x04

int ftl_do_write(struct ftl_dev *ftl, u64 a, u64 b, void *buf,
		 int type, int mode, unsigned int flags)
{
	int old;
	int new;
	int ret;

	if (type == WRITE_TYPE_GC && (flags & WRITE_FLAG_MOVE))
		old = (int)a;
	else
		old = ftl->l2p[a];

	if (mode == WRITE_MODE_FAST || (flags & WRITE_FLAG_CACHE))
		new = alloc_page(ftl, 1);
	else
		new = alloc_page(ftl, 0);

	ret = nand_program(ftl->nand, new, buf);
	if (ret)
		return ret;

	if (flags & WRITE_FLAG_SYNC)
		journal_add(ftl->journal, type, a, old, new, flags);

	if (type == WRITE_TYPE_GC)
		ftl->l2p[b] = new;
	else
		ftl->l2p[a] = new;

	return 0;
}
坏味道 具体表现 FTL 后果
领域语言丢失 a、b 在不同 type 下含义不同。 调用者必须查实现才能知道传 LBA、old PPA 还是 new LBA。
限界上下文混合 Mapping、GC、WriteBooster、Journal 语义塞进一个函数。 任一上下文变化都会修改同一段代码。
恢复语义不可读 Journal 记录 type 和 flags，而不是领域事件。 Power Loss Recovery 必须复制写路径里的 if/else 才能重放。
错误路径不可演进 NAND fail 没有区分 host write、relocation、metadata write。 Bad Block、orphan page、mapping rollback 无法按意图补偿。

改进代码：用领域结构承载接口语义

改进版不是把名字写长，而是让每个公开结构都带有明确上下文： 写入来源、目标介质、mapping delta、relocation intent 和 journal record 分别表达不同领域判断。

ftl_domain.h：把核心语言放进公共边界 C 
#ifndef FTL_DOMAIN_H
#define FTL_DOMAIN_H

#include <linux/types.h>
#include <linux/errno.h>

enum ftl_write_origin {
	FTL_WRITE_HOST = 0,
	FTL_WRITE_SLC_FLUSH,
	FTL_WRITE_GC_RELOCATE,
	FTL_WRITE_METADATA,
};

enum ftl_page_role {
	FTL_PAGE_USER_DATA = 0,
	FTL_PAGE_MAPPING_META,
	FTL_PAGE_JOURNAL_META,
};

enum ftl_commit_reason {
	FTL_COMMIT_HOST_WRITE = 0,
	FTL_COMMIT_RELOCATION,
	FTL_COMMIT_RECOVERY_REPLAY,
};

struct ftl_ppa {
	u32 ch;
	u32 lun;
	u32 blk;
	u32 page;
};

struct ftl_mapping_delta {
	u64 lba;
	struct ftl_ppa old_ppa;
	struct ftl_ppa new_ppa;
	enum ftl_commit_reason reason;
};

struct ftl_relocation_intent {
	u64 lba;
	struct ftl_ppa victim_ppa;
	enum ftl_page_role page_role;
};

int ftl_ppa_valid(const struct ftl_ppa *ppa);

#endif
gc_relocate.c：GC 用 relocation 语言，而不是普通 write flag C 
static int gc_build_relocation_delta(struct ftl_dev *ftl,
				     const struct ftl_relocation_intent *intent,
				     struct ftl_mapping_delta *delta)
{
	struct block_alloc_req alloc_req;
	int ret;

	if (!intent || !delta)
		return -EINVAL;

	ret = map_lookup_l2p(ftl->map, intent->lba, &delta->old_ppa);
	if (ret)
		return ret;

	if (!ftl_ppa_equal(&delta->old_ppa, &intent->victim_ppa))
		return -ESTALE;

	alloc_req.origin = FTL_WRITE_GC_RELOCATE;
	alloc_req.page_role = intent->page_role;
	alloc_req.lba = intent->lba;

	ret = block_alloc_page_for_write(ftl->bm, &alloc_req,
					 &delta->new_ppa);
	if (ret)
		return ret;

	delta->lba = intent->lba;
	delta->reason = FTL_COMMIT_RELOCATION;

	return 0;
}

int gc_relocate_valid_page(struct ftl_dev *ftl,
			   const struct ftl_relocation_intent *intent)
{
	struct ftl_mapping_delta delta;
	int ret;

	ret = gc_build_relocation_delta(ftl, intent, &delta);
	if (ret)
		return ret;

	ret = nand_copy_page(ftl->nand, &intent->victim_ppa,
			     &delta.new_ppa);
	if (ret)
		goto err_copy;

	ret = journal_append_mapping_delta(ftl->journal, &delta);
	if (ret)
		goto err_journal;

	ret = map_commit_delta(ftl->map, &delta);
	if (ret)
		goto err_mapping;

	block_invalidate_page(ftl->bm, &intent->victim_ppa);
	return 0;

err_mapping:
	ftl_mark_recovery_required(ftl, FTL_RECOVERY_JOURNAL_REPLAY);
	return ret;

err_journal:
	block_discard_uncommitted_page(ftl->bm, &delta.new_ppa);
	return ret;

err_copy:
	block_report_relocation_failure(ftl->bm, &intent->victim_ppa);
	return ret;
}

状态表与责任表

FTL 领域语言责任表

词汇 所属上下文 谁拥有不变量 推荐代码形态 避免的问题
LBA Host / Mapping Mapping Table u64 lba，只在 Mapping API 中解释。 避免把 host 地址和 NAND 物理地址混用。
PPA NAND / Block Manager Block Manager struct ftl_ppa。 避免用 int ppn 泄漏 geometry 编码。
Open Block Block Manager Block Manager block_alloc_page_for_write()。 避免 FTL Core 直接推进 write pointer。
Victim Block GC GC policy + Block Manager struct gc_victim。 避免把可回收块和坏块候选混为一谈。
Mapping Commit Mapping / Journal Mapping Table struct ftl_mapping_delta。 避免 journal、mapping、recovery 各自解释更新语义。
Relocation GC / Recovery GC service struct ftl_relocation_intent。 避免把 GC 搬移误建模为普通 host write。

命名到错误路径的映射

领域动作 典型失败 推荐错误语义 补偿归属
Host write mapping commit journal append fail -EIO 或 -ENOSPC Block Manager 丢弃未提交 page，FTL Core 返回 host 写失败。
GC relocation old PPA 已被 host write 替换 -ESTALE GC 跳过该 page，不更新 Mapping。
SLC flush TLC 分配失败 -ENOSPC WriteBooster 保留 SLC 数据并请求 GC。
Metadata journal replay record checksum mismatch -EILSEQ Recovery 停止重放该 segment，进入扫描校验。

四本书思想落点表

书 思想 SSD / FTL 设计位置 约束了什么 帮助避免什么问题
《软件设计哲学》 消除晦涩，接口应该暴露意图而非实现猜谜。 Mapping、GC、Journal 的公共头文件。 禁止 magic type/mode/flag 作为主要语义载体。 避免调用者必须阅读实现才能正确调用。
《程序员修炼之道》 DRY 是知识单一来源，正交性要求词汇变化影响可控。 Relocation、mapping commit、bad block candidate 的统一定义。 同一领域词只能有一个主定义。 避免 GC、Recovery、Journal 三套解释互相漂移。
《设计的设计》 概念完整性来自稳定、有限、可组合的设计概念。 FTL 写入、搬移、提交、恢复的核心词汇集合。 不要为每个临时分支发明新名词，也不要把所有意图压成 flag。 避免领域模型过瘦或过度膨胀。
《领域驱动设计》 通用语言必须进入代码，并受限界上下文约束。 LBA、PPA、open block、victim block、journal record。 每个词必须说明上下文、所有者和可变更接口。 避免同名不同义和跨上下文误用。
