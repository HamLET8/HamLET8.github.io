Day 2 · Reading First

模块要深，语言要准确：把理解变成接口、名称和所有权

Day 1 解决“为什么可工作还不够”。Day 2 开始回答“怎么把理解落实到代码结构”。 今天的关键词不是简单的“起好名字”，而是让模块接口隐藏复杂度，让语言承载不变量，让同一份知识只归一个地方所有。

今日主问题：语言如何承载设计知识 阅读主轴：深模块、DRY、正交性、通用语言 练习场：FTL 术语与接口改写

今日阅读范围

今天四本书都在推进同一件事：设计知识不能只留在脑子里或文档里，它必须进入模块边界、参数类型、函数名和团队语言。

书 今天阅读的主题块 读完要能回答的问题
《软件设计哲学》 深模块、信息隐藏、信息泄漏，以及更通用的模块为什么通常更深。 一个接口是把复杂度藏起来了，还是把内部知识推给调用者了？
《程序员修炼之道》 DRY、正交性、可逆性。 同一份知识是否被重复表达？一个决策变化会不会污染多个方向？
《设计的设计》 概念完整性、设计约束与早期设计决策如何限制后续空间。 哪些核心概念必须保持稳定，否则系统会变成补丁集合？
《领域驱动设计》 通用语言、模型与实现绑定。 团队说出的词，是否能在代码结构中找到对应物？

今日新增概念：语言不是标签，而是知识所有权

深模块

深模块不是“函数很多”或“代码藏起来”，而是用小接口承担大功能。调用者不需要知道内部算法、状态组合和例外处理。

知识单一来源

DRY 要保护的是知识，不是文本。一个状态含义、映射规则或错误分类，如果散在多个模块，就已经重复。

概念完整性

概念完整性不是口号，而是限制新增概念的纪律：每个新状态、新类型、新接口都必须能解释系统主模型。

通用语言

DDD 的语言不是术语表。只有当词汇进入代码、测试、日志、讨论和文档，它才真的成为模型的一部分。

今天的教学判断： 如果一个 FTL 接口叫 do_update(type, flags)， 问题不只是名字丑，而是调用者必须知道 type 和 flags 背后的隐含知识。语言没有承载设计，复杂度就会回到调用点。

概念慢读：从“看懂代码”推进到“接口替你守住知识”

Day 1 的重点是识别“可工作但不可解释”的风险。Day 2 不再停留在风险识别，而是进入具体设计手段： 哪些知识应该被藏进模块内部，哪些语言必须暴露在接口上，哪些变化轴不能混在一起。 如果 Day 2 读完后只得到“命名要清楚”，那仍然太浅。今天要学的是：语言、接口、类型和模块边界共同决定知识归属。

1. 深模块：小接口不是少写参数，而是少泄漏知识

一个浅接口的问题不一定是参数多，而是参数背后有很多隐含规则。比如 alloc_page(type, flags, ctx) 参数看似少，但调用者必须知道： 哪种 type 可以配哪种 flag，GC 分配是否允许跳过 journal，hot data 是否应该进入独立 block， recovery replay 能否复用普通 host write 分配路径。这些知识本应属于 Block Manager 或策略模块， 却被接口推给了调用者。

深模块的目标不是把接口做成黑箱，而是让调用者只表达领域意图。调用者说“我要为 host write 分配一个 page”， Block Manager 再决定 open block、冷热分区、free pool、write pointer 和磨损约束。这样调用者不必理解内部策略， 也不会在多个地方重复策略判断。

2. 信息隐藏：隐藏的是变化原因，不是字段本身

信息隐藏常被误读成“把结构体字段设成 private”。在 C 风格系统里，即使没有语言级 private，也可以通过接口和文件边界隐藏变化原因。 例如 block 的 free_pages 计数可能因为 host write、GC relocation、erase finish、bad block retire 而变化。 如果外部模块都能直接改这个计数，那么真正泄漏的不是字段，而是“什么事件会影响空间状态”这份知识。

因此今天要把信息隐藏理解成：一个模块拥有一类变化原因，外部模块只提交事件或请求，不直接修改内部事实。 这为后续 Day 7 的状态机打基础，但 Day 2 还不讲完整状态机，只讲接口如何先把知识归属划出来。

3. DRY 与正交性：避免同一规则以不同形状出现

DRY 不是看到重复代码就抽函数。真正危险的是同一条规则在不同地方以不同形状出现。 例如“GC relocation 不能把数据写进普通 host write 的 open block”这条规则， 如果同时出现在调用点的 flag 组合、Block Manager 的 if 分支、GC 的预检查、Recovery 的补偿逻辑里， 那么以后规则变化时必然漏改。

正交性则要求变化轴不要互相污染。数据冷热是一条变化轴，写入来源是一条变化轴，是否需要稳定提交是另一条变化轴。 把它们全部塞进 flags 后，一个策略变化会牵动所有调用点。 这就是“看似灵活，实际耦合”。

4. 通用语言：不是把中文词翻成英文函数名

DDD 的通用语言要求团队讨论、代码结构和测试场景使用同一套词。但这不等于把 “写入意图”翻译成 write_intent 就完成了。 要检查这个词是否真的承载了规则：它是否区分 host write 与 GC relocation？是否说明 mapping 何时可见？ 是否影响 journal 记录？是否让错误路径能说清楚写入停在哪个阶段？

如果一个词只出现在注释里，不出现在类型、接口、测试和日志中，它还不是模型语言。 Day 2 的核心任务就是把“语言”从说明文字推进到可执行边界。

四书差异：今天它们各自约束什么

书 今天关注的对象 它会如何审查一个 FTL 接口
《软件设计哲学》 接口是否浅，是否泄漏内部复杂度。 问调用者是否必须理解 block 状态、page cursor、bad block 策略和 journal 顺序。
《程序员修炼之道》 知识是否重复，变化轴是否正交。 问 allocation policy、write intent、temperature、GC pressure 是否被混进同一个 flag。
《设计的设计》 概念集合是否一致，是否能长期承受约束变化。 问新概念是稳定抽象，还是为某个 bug 临时添加的例外通道。
《领域驱动设计》 语言是否连接领域讨论和代码实现。 问 open block、host write、relocation、mapping commit 是否在代码里是明确类型或事件。

术语重建：哪些词必须变成代码里的概念

旧版报告经常把“领域语言”当成泛泛的命名建议。Day 2 要更具体：下面这些词如果只存在于文章中， 但没有进入接口和类型，系统仍然会退回 type、 mode、flags 的世界。

领域词 它承载的知识 应该进入哪里 不应该退化成什么
Host Write 来自主机的新写入，要求最终成为 LBA 当前版本。 struct ftl_host_write_alloc、mapping commit 接口、journal record。 type = 0
Relocation GC 搬迁有效页，不代表主机提交了新数据。 struct ftl_relocation_alloc、victim block 上下文、old PPA 证据。 FTL_ALLOC_GC flag
Replay 恢复流程按 journal 证据重建事实，不能等同普通写路径。 struct ftl_replay_alloc、journal seq、幂等检查。 ctx 指针
Stable Commit 写入是否需要在对外可见前留下可恢复证据。 commit policy 字段、journal 决策、错误返回。 FTL_ALLOC_SYNC 与 NO_JOURNAL 混用
Temperature 数据冷热影响放置策略，但不等于写入来源。 enum ftl_data_temperature 或独立 placement hint。 和 GC / sync 混在同一个 flags 中。

这个表的教学意义在于：每个词都要说明“它承载什么知识”。如果一个词不能改变接口、类型、日志或测试， 就还只是描述性文字。真正的通用语言必须能减少调用者需要记住的隐含规则。

SSD / FTL 练习：把模糊接口改成领域语言

今天的练习不是重新设计整个 FTL，而是把“万能函数 + flags”的接口改成能表达意图和不变量的接口。 改写时要问：这个名字是否说明调用者想做什么？参数是否携带必要语义？哪些知识被模块拥有？

旧接口信号 说明的问题 Day 2 改写方向
int type 领域意图被压成数字，调用者和被调用者共享隐含知识。 改成 enum ftl_write_intent_kind 或更具体的请求结构。
unsigned long flags 多个变化轴混在一起，破坏正交性。 拆成 temperature、durability、source、journal policy 等独立字段。
void *ctx 调用契约无法从类型看出，维护者必须读实现。 为 host write、GC relocation、recovery replay 分别建上下文类型。
update() 函数名没有表达领域动作，也无法说明不变量由谁维护。 改成 commit mapping、reserve page、mark orphan、finish erase 等动作。

接口推导：为什么不能只把函数改个名字

很多“改进版”只会把 ftl_alloc_page() 改名成 ftl_alloc_for_host_write()，但参数仍然是 type、flags、ctx。 这不是模型进步，只是把旧问题包了一层更好看的名字。真正的推导要从调用者意图开始。

第一步：列出调用者真正想表达的意图

Host Write 想表达的是“主机写入一个 LBA，希望最终成为当前版本”；GC Relocation 想表达的是“从 victim block 搬迁一个仍然有效的旧 PPA”； Recovery Replay 想表达的是“根据 journal 证据重放一个之前已经承认过的事实”。这三件事都需要物理 page， 但它们的前置条件、错误语义、journal 要求和统计归属不同。如果共享一个万能接口，调用者就必须知道这些区别。

第二步：把不同前置条件放进不同请求类型

请求类型不是为了形式感。它应该让非法组合无法自然出现。Host Write 需要 LBA、temperature、stable commit 需求； Relocation 需要 old PPA、victim block、原 LBA；Replay 需要 journal sequence 和幂等识别。 如果一个字段对某类请求没有意义，它就不应该出现在这类请求结构里。

第三步：让模块内部拥有策略变化

拆出三个接口后，并不表示内部要复制三套分配算法。Block Manager 仍然可以调用同一个内部 helper， 但 helper 接收的是已经归一化后的 placement decision，而不是让外部调用者传入随意组合的 flags。 外部接口表达意图，内部策略计算结果，这就是深模块和 DRY 同时成立的方式。

设计步骤 如果没做到 做到以后有什么变化
识别不同意图 所有调用点都传 type。 调用者不再共享隐含数字约定。
拆分请求类型 无意义字段到处存在，非法组合靠 if 检查。 类型本身表达可用字段和前置条件。
策略留在模块内部 冷热、GC、journal、sync 规则散在调用点。 规则变化时主要修改 Block Manager 或策略模块。
错误语义按意图返回 所有失败都变成 -EINVAL 或 -EIO。 host write 可返回空间压力，relocation 可返回 victim stale，replay 可返回 journal conflict。

反例与改写：从万能分配函数到深接口

反例目标： 这段代码看似灵活，实际把多个知识轴混进 type 和 flags。它违反的不是“命名风格”，而是深模块、DRY、正交性和通用语言。
反例：万能接口让调用者背负内部知识 C 
#define FTL_ALLOC_HOT		BIT(0)
#define FTL_ALLOC_GC		BIT(1)
#define FTL_ALLOC_SYNC		BIT(2)
#define FTL_ALLOC_NO_JOURNAL	BIT(3)

int ftl_alloc_page_bad(struct ftl_dev *ftl, int type, unsigned long flags,
		       u64 lba, void *ctx, struct ftl_ppa *ppa)
{
	if (type == 0 && (flags & FTL_ALLOC_GC))
		return -EINVAL;
	if ((flags & FTL_ALLOC_NO_JOURNAL) && !(flags & FTL_ALLOC_SYNC))
		return -EINVAL;

	if (flags & FTL_ALLOC_HOT)
		return alloc_hot_page(ftl, lba, ppa);
	if (flags & FTL_ALLOC_GC)
		return alloc_gc_page(ftl, ctx, ppa);

	return alloc_normal_page(ftl, lba, ppa);
}
问题 为什么是 Day 2 的问题
接口浅 调用者必须知道 hot、GC、sync、journal 之间的合法组合。
知识重复 合法 flag 组合会在调用点、分配函数、测试和文档中重复。
变化轴混合 写入来源、冷热、持久化策略、同步语义被塞进一个 flags。
语言贫血 代码没有表达 host write、GC relocation、recovery replay 这些真实领域动作。
改写：接口按领域意图拆开，策略知识归模块所有 C 
enum ftl_data_temperature {
	FTL_DATA_COLD = 0,
	FTL_DATA_WARM,
	FTL_DATA_HOT,
};

struct ftl_host_write_alloc {
	u64 lba;
	enum ftl_data_temperature temperature;
	bool needs_stable_commit;
};

struct ftl_relocation_alloc {
	struct ftl_ppa old_ppa;
	u64 lba;
	u32 victim_block;
};

struct ftl_replay_alloc {
	u64 lba;
	u64 journal_seq;
};

int block_alloc_for_host_write(struct ftl_block_manager *bm,
			       const struct ftl_host_write_alloc *req,
			       struct ftl_ppa *ppa);
int block_alloc_for_relocation(struct ftl_block_manager *bm,
			       const struct ftl_relocation_alloc *req,
			       struct ftl_ppa *ppa);
int block_alloc_for_replay(struct ftl_block_manager *bm,
			   const struct ftl_replay_alloc *req,
			   struct ftl_ppa *ppa);

int ftl_prepare_host_write(struct ftl_dev *ftl, u64 lba,
			   enum ftl_data_temperature temperature,
			   struct ftl_ppa *ppa)
{
	struct ftl_host_write_alloc req = {
		.lba = lba,
		.temperature = temperature,
		.needs_stable_commit = true,
	};

	return block_alloc_for_host_write(ftl->blocks, &req, ppa);
}

这个改写不是为了多写几个结构体，而是让语言变成约束。Host write、relocation、replay 的合法组合不同， 所以它们不应该共享一个万能 flag 接口。Block Manager 内部仍可复用分配算法，但外部接口必须表达调用意图。

重构走读：从 flags 接口走到深模块接口

Day 2 的改写容易被误解成“多建几个结构体”。为了避免这个误解，需要把重构过程拆成更细的判断。 下面不是新的最终方案，而是一条审查路径：每一步都要说明删掉了哪类隐含知识，又把哪类知识交给了更合适的所有者。

第一步：不要先问函数应该叫什么，先问调用者为什么调用它

坏接口的第一个问题是 ftl_alloc_page_bad() 这个名字太宽。 它没有说明调用者要为哪种写入分配 page。Host write、GC relocation、Recovery replay 都需要 page， 但它们不是同一个领域动作。Host write 面向主机可见性；relocation 面向空间回收；replay 面向恢复证据。 如果函数名只说“分配 page”，调用者就必须用额外参数补充真正意图，于是 type 和 flags 开始膨胀。

正确的第一步不是立即写三个函数，而是列出三类调用者的前置条件和失败后果。 如果两类调用者的失败后果不同，就不应该强行共用同一个外部接口。 例如 host write 分配失败可能要向上返回空间压力；relocation 分配失败可能意味着 GC 需要换 victim 或触发紧急回收； replay 分配失败则可能说明 journal 证据和当前介质状态冲突。这些不是同一个 -ENOSPC 能表达的。

第二步：把 flags 拆成独立变化轴

FTL_ALLOC_HOT、FTL_ALLOC_GC、 FTL_ALLOC_SYNC、FTL_ALLOC_NO_JOURNAL 看起来只是四个 bit，实际上代表四类完全不同的问题：数据温度、写入来源、提交持久性、journal 策略。 它们放在同一个 flags 里后，调用者可以组合出很多没有领域意义的状态，例如 GC relocation + no journal + sync。 然后实现不得不用 if 去排除非法组合，测试也必须覆盖这些组合，文档还要解释这些组合。 这就是知识重复。

拆分变化轴后，数据温度进入 host write 请求；GC 来源进入 relocation 请求；journal sequence 进入 replay 请求； commit policy 成为单独策略。这样每个字段都有明确上下文。更重要的是，非法组合不再是运行时才发现的神秘错误， 而是在接口层面就不自然。

第三步：区分外部深接口和内部复用

很多人担心拆接口会导致代码重复。这里要区分外部接口和内部实现。外部接口应该按领域意图拆开， 因为调用者需要表达不同意图；内部实现可以共享算法，因为 Block Manager 最终都要选择一个物理 page。 一个常见结构是：外部接口分别解析 host write、relocation、replay 请求，然后转换成内部 placement_request，再由统一的 placement engine 处理。

这样既不是万能函数，也不是复制三套算法。它满足《软件设计哲学》的深模块，因为调用者不再理解内部策略； 满足《程序员修炼之道》的 DRY，因为放置规则只在内部策略中定义； 满足《设计的设计》的概念完整性，因为外部概念集合稳定； 也满足 DDD 的通用语言，因为代码接口使用真实领域动作。

重构前问题 错误修法 正确修法 检查结果
函数名过宽。 只改名成 alloc_for_write()。 按 host write、relocation、replay 拆分外部意图。 调用点不再用数字补充语义。
flags 混合变化轴。 增加更多 flag 或宏注释。 把温度、来源、commit policy、journal 证据拆到各自请求类型。 非法组合减少，规则归属更清晰。
调用者知道内部策略。 把策略判断移动到另一个 helper，但参数不变。 调用者只提交意图，Block Manager 内部计算 placement decision。 策略变化不污染上层调用点。
错误语义太粗。 继续返回 -EINVAL。 按意图区分 no space、stale victim、journal conflict、unsupported policy。 失败能指导恢复、重试或上报。

第四步：检查语言是否真的进入测试

最后一步看测试名字。如果测试仍然叫 test_alloc_flags()， 说明语言没有真正改变。更好的测试应该像： host_write_allocation_keeps_gc_blocks_separate、 relocation_allocation_rejects_stale_victim、 replay_allocation_requires_matching_journal_seq。 这些名字不是装饰，它们让测试成为通用语言的一部分。

到这里，Day 2 和 Day 1 的差异就很清楚了：Day 1 让你看见“可工作但不可解释”的问题； Day 2 要求你把解释能力落实到外部接口、请求类型、错误语义和测试语言中。

代码审查演练：看到一个接口时先问五个问题

Day 2 的知识应该能用于代码审查。以后看到任何 FTL 接口，不要先评价名字好不好听， 而是用下面五个问题判断它是否真的承载了设计知识。

问题 为什么重要 在分配接口中的例子
调用者表达的是意图，还是内部策略？ 深模块要求调用者少知道内部决策。 表达 host write，而不是传 hot + sync + no journal。
同一条规则是否只出现一次？ DRY 保护规则的单一归属。 GC relocation 的合法目标 block 不应在多个调用点重复判断。
变化轴是否被分开？ 正交性决定未来修改的影响范围。 temperature 不应和 write source 混在同一个 flags。
领域词是否进入类型和测试？ 通用语言必须可执行。 测试名应能出现 host write allocation、relocation allocation、replay allocation。
错误返回是否能说明哪条规则失败？ 反馈决定系统是否可维护。 区分 no free block、victim stale、journal conflict，而不是统一返回 -EINVAL。

自测题：确认你能区分命名和模型

  • 问题 1： 深模块和“大模块”有什么区别？
    参考要点： 深模块用小接口承担大量内部复杂度；大模块只是代码多。深度看调用者少知道了多少内部事实。
  • 问题 2： 为什么 DRY 不是“不要复制代码”？
    参考要点： DRY 保护知识单一来源。两段代码不同，但如果表达同一条规则，也可能违反 DRY。
  • 问题 3： 一个函数名从 update() 改成 mapping_commit_host_write()， 什么时候只是改名，什么时候是模型进步？
    参考要点： 如果只是换字符串，不算进步；如果接口参数、错误语义、测试和调用边界都围绕 commit host write 改变，才是模型进入实现。
  • 问题 4： 为什么万能 flags 会破坏正交性？
    参考要点： flags 常把来源、策略、持久化、冷热、错误处理混在一起，一个变化会影响多个方向，调用者也必须理解组合规则。
