Day 3 · SSD / FTL 代码设计判断

接口深度：把参数列表变成领域意图

今天的核心判断是：Block Manager 不应该暴露一个万能 alloc_page(type, mode, flags)。 它应该接收明确的写入意图，例如 host write、GC relocation、SLC flush、metadata journal write， 并在模块内部处理 free pool、open block、reserve、磨损和坏块约束。

一句话判断 接口表达意图，模块吸收策略 涉及：Block Manager / GC / WriteBooster / Journal / Mapping 关键词：深模块 · 正交性 · 契约 · 领域意图

今日设计判断

深接口不是参数少，而是调用者不需要知道内部策略。调用者只说明“为什么要分配”， Block Manager 决定“从哪里分配、是否触发 GC、是否使用 reserve、是否拒绝该请求”。

今日涉及模块

Block Manager、Free Block Pool、GC、Victim Block Selection、WriteBooster、 Metadata Journal、Bad Block Management、NAND Program Flow。

今日阅读重点：接口是复杂度的边境

Day 1 划了模块边界，Day 2 建了领域语言。Day 3 要把这两件事落实到接口： 如果接口仍然要求调用者传 magic flag，边界和语言都会退化成装饰。

思想来源 今天必须掌握的概念 落到 SSD / FTL 的判断
《软件设计哲学》 深模块通过简单接口隐藏大量复杂度。浅接口会把内部细节转嫁给调用者，导致每个调用点都理解同一套策略。 block_alloc_page_for_write() 应该隐藏 open block 选择、free pool 水位、reserve pool、wear hint 和 bad block 过滤。
《程序员修炼之道》 正交性要求分配策略变化不污染 GC、WriteBooster、Journal；DRY 要求 free pool 规则只有一个所有者。 GC 不应该复制“metadata reserve 不能用”的规则；它只提交 FTL_WRITE_GC_RELOCATE 意图。
《设计的设计》 概念完整性来自统一取舍：接口应该围绕“分配物理页”这个主概念，而不是把每个场景补成一个布尔开关。 一个请求结构可以稳定承载 origin、page role、temperature、reserve policy；这比不断扩展参数列表更能演进。
《领域驱动设计》 接口是通用语言的落点。领域服务和聚合边界必须通过参数类型、返回值和错误语义表达。 Block Manager 拥有 block/page 生命周期；调用者传入写入意图，不直接传 open block id 或 write pointer。

概念融合解释：好的接口让调用者少知道一点

FTL 的写入路径天然有多种来源：host write、GC relocation、SLC flush、metadata journal write。 它们都需要物理页，但它们不是同一种请求。坏设计会把这些差异压成 type、mode、flags； 好设计会把这些差异建模为写入意图，然后让 Block Manager 做策略决策。

统一判断： 调用者负责表达领域意图，Block Manager 负责维护空间不变量。接口越深，调用者越不需要知道 free pool、open block、erase count、reserve pool 和 bad block 的内部规则。

这里四本书互相制约：深模块要求接口少暴露内部知识；务实工程要求分配规则不要重复； 概念完整性要求接口围绕稳定主概念；DDD 要求参数类型表达领域意图。合起来就是： 不要让调用者通过参数组合选择内部策略，让调用者说清业务意图。

SSD / FTL 设计场景

Host Write

Host 写入需要低延迟和普通 user data page。调用者只传 FTL_WRITE_HOST，不直接选择 open block。

GC Relocation

GC 搬移有效页需要避免抢占 metadata reserve，并可能接受更高延迟。 它应通过 relocation origin 表达，而不是复用 host write flag。

Metadata Journal

Journal 写入需要保留空间和更严格的失败语义。 如果普通 user data 可以耗尽 journal reserve，掉电恢复会被破坏。

SLC Cache / WriteBooster

WriteBooster 的 SLC page 分配不应暴露 SLC open block 游标。 它提交加速写入意图，Block Manager 决定介质类别和 fallback。

Free Block Pool

低水位不是所有调用者都能绕过的条件。Block Manager 依据请求来源返回 -ENOSPC、触发 GC 建议或允许 reserve。

Bad Block Management

分配接口必须过滤 bad block candidate 和 retired block。 调用者不应该带着 block id 绕过坏块隔离边界。

反例代码：万能分配接口让调用者背锅

坏味道： 这个接口看起来通用，实际上要求 GC、Host Write、Journal 都理解 Block Manager 的内部规则。 一旦 free pool、reserve 或 wear leveling 策略变化，所有调用点都会被迫修改。
反例：alloc_page() 暴露内部策略 C 
#define ALLOC_FAST		0x01
#define ALLOC_META		0x02
#define ALLOC_GC		0x04
#define ALLOC_USE_RESERVE	0x08

int alloc_page(struct ftl_dev *ftl, int pool, int flags, int *ppn)
{
	int blk;

	if (flags & ALLOC_META)
		blk = ftl->bm.meta_open_blk;
	else if (flags & ALLOC_FAST)
		blk = ftl->bm.slc_open_blk;
	else
		blk = ftl->bm.user_open_blk;

	if (ftl->bm.free_cnt[pool] < 2 && !(flags & ALLOC_USE_RESERVE))
		return -ENOSPC;

	if (ftl->bm.blocks[blk].is_bad)
		return -EIO;

	*ppn = blk * ftl->geo.pages_per_blk +
	       ftl->bm.blocks[blk].write_ptr++;

	if (flags & ALLOC_GC)
		ftl->bm.gc_alloc_cnt++;

	return 0;
}

int gc_move_page(struct ftl_dev *ftl, int old_ppn, u64 lba)
{
	int new_ppn;
	int ret;

	ret = alloc_page(ftl, 0, ALLOC_GC, &new_ppn);
	if (ret == -ENOSPC) {
		ret = alloc_page(ftl, 1, ALLOC_GC | ALLOC_USE_RESERVE,
				 &new_ppn);
	}
	if (ret)
		return ret;

	return nand_copy_page(ftl->nand, old_ppn, new_ppn);
}
问题 代码表现 工程后果
模块边界泄漏 调用者知道 pool 编号、reserve flag、SLC/META open block 的隐含关系。 Block Manager 的内部策略变化会扩散到 GC 和写路径。
领域意图被压扁 GC relocation 和 metadata write 都只是 flags 的组合。 错误路径无法区分“可延迟搬移失败”和“元数据空间耗尽”。
不变量不可集中维护 调用者可以使用 ALLOC_USE_RESERVE 绕开低水位。 Power Loss Recovery 依赖的 metadata reserve 可能被普通路径消耗。
返回值语义模糊 -ENOSPC 不说明是 user pool、metadata reserve 还是 SLC fallback 失败。 上层只能补更多 if/else，错误路径继续散落。

改进代码：请求结构承载领域意图

改进后的接口不要求调用者知道 pool 编号或 reserve 规则。调用者只提交分配意图； Block Manager 返回 PPA、介质类别和必要的分配诊断。

block_manager.h：用请求结构定义接口契约 C 
#ifndef BLOCK_MANAGER_H
#define BLOCK_MANAGER_H

#include <linux/types.h>
#include <linux/errno.h>

enum ftl_write_origin {
	FTL_WRITE_HOST = 0,
	FTL_WRITE_SLC_FLUSH,
	FTL_WRITE_GC_RELOCATE,
	FTL_WRITE_METADATA,
};

enum ftl_page_role {
	FTL_PAGE_USER_DATA = 0,
	FTL_PAGE_MAPPING_META,
	FTL_PAGE_JOURNAL_META,
};

enum ftl_media_class {
	FTL_MEDIA_ANY = 0,
	FTL_MEDIA_SLC,
	FTL_MEDIA_TLC,
};

enum block_alloc_policy {
	BLOCK_ALLOC_NORMAL = 0,
	BLOCK_ALLOC_ALLOW_BACKGROUND_GC,
	BLOCK_ALLOC_METADATA_RESERVE,
};

struct ftl_ppa {
	u32 ch;
	u32 lun;
	u32 blk;
	u32 page;
};

struct block_alloc_req {
	u64 lba;
	enum ftl_write_origin origin;
	enum ftl_page_role page_role;
	enum ftl_media_class preferred_media;
	enum block_alloc_policy policy;
};

struct block_alloc_result {
	struct ftl_ppa ppa;
	enum ftl_media_class media;
	bool used_reserve;
	bool should_trigger_gc;
};

int block_alloc_page_for_write(struct block_manager *bm,
			       const struct block_alloc_req *req,
			       struct block_alloc_result *result);

#endif
block_manager.c：策略留在所有者模块内 C 
static int block_select_pool(struct block_manager *bm,
			     const struct block_alloc_req *req,
			     struct block_pool **pool)
{
	if (req->page_role == FTL_PAGE_JOURNAL_META) {
		*pool = &bm->journal_pool;
		return 0;
	}

	if (req->preferred_media == FTL_MEDIA_SLC &&
	    block_pool_available(&bm->slc_pool)) {
		*pool = &bm->slc_pool;
		return 0;
	}

	*pool = &bm->user_pool;
	return 0;
}

int block_alloc_page_for_write(struct block_manager *bm,
			       const struct block_alloc_req *req,
			       struct block_alloc_result *result)
{
	struct block_pool *pool;
	struct nand_block *blk;
	int ret;

	if (!bm || !req || !result)
		return -EINVAL;

	ret = block_select_pool(bm, req, &pool);
	if (ret)
		return ret;

	if (block_pool_below_low_watermark(pool)) {
		result->should_trigger_gc = true;
		if (req->policy != BLOCK_ALLOC_METADATA_RESERVE)
			return -ENOSPC;
	}

	ret = block_pool_get_open_block(bm, pool, req, &blk);
	if (ret)
		return ret;

	if (block_is_bad_or_retired(blk))
		return -EIO;

	ret = block_next_writable_ppa(bm, blk, &result->ppa);
	if (ret)
		return ret;

	result->media = pool->media;
	result->used_reserve = pool->is_reserve;

	return 0;
}
gc.c：调用者表达意图，不复制分配策略 C 
int gc_alloc_relocation_page(struct ftl_dev *ftl, u64 lba,
			     struct block_alloc_result *result)
{
	struct block_alloc_req req = {
		.lba = lba,
		.origin = FTL_WRITE_GC_RELOCATE,
		.page_role = FTL_PAGE_USER_DATA,
		.preferred_media = FTL_MEDIA_TLC,
		.policy = BLOCK_ALLOC_ALLOW_BACKGROUND_GC,
	};
	int ret;

	ret = block_alloc_page_for_write(ftl->bm, &req, result);
	if (ret == -ENOSPC && result->should_trigger_gc)
		return gc_request_emergency(ftl->gc, GC_CAUSE_FREE_POOL_LOW);

	return ret;
}

接口与责任表

浅接口与深接口对比

维度 浅接口 深接口 设计收益
参数 pool, flags, mode struct block_alloc_req 调用者表达意图，而不是选择内部策略。
错误语义 所有空间不足都返回 -ENOSPC 结果中带 should_trigger_gc、reserve 使用情况 上层可以做正确补偿，不需要猜测原因。
策略所有权 GC / Host Write / Journal 各自判断 reserve Block Manager 统一维护 free pool 和 reserve 规则 避免知识重复和策略漂移。
演进能力 新增介质类别需要修改所有调用点 新增字段或内部策略即可吸收变化 支持 SLC/TLC/QLC、hot/cold、wear hint 演进。

分配请求到 Block Manager 决策矩阵

请求来源 Page Role 可用池 低水位处理 典型返回
Host Write User Data SLC 优先或 TLC user pool 返回 -ENOSPC 并建议 foreground GC 0、-ENOSPC、-EIO
GC Relocation User Data TLC user pool，不使用 metadata reserve 跳过 victim 或请求 emergency GC 0、-ENOSPC、-ESTALE
Journal Write Journal Meta journal pool / metadata reserve 允许 reserve，失败后标记恢复风险 0、-ENOSPC、-EIO
SLC Flush User Data TLC user pool 保留 SLC 数据并触发后台 GC 0、-EAGAIN、-ENOSPC
Recovery Replay Mapping Meta / User Data 受恢复阶段限制 不触发普通 GC，进入恢复失败路径 0、-EILSEQ、-EIO

四本书思想落点表

书 思想 SSD / FTL 设计位置 约束了什么 帮助避免什么问题
《软件设计哲学》 深模块：简单接口隐藏复杂内部策略。 Block Manager 分配接口。 调用者不得直接选择 open block、reserve pool、write pointer。 避免空间管理细节泄漏到 FTL Core、GC、Journal。
《程序员修炼之道》 DRY 和正交性：策略知识只属于一个模块。 Free Block Pool、metadata reserve、bad block 过滤。 分配规则只能由 Block Manager 维护。 避免 GC 和 WriteBooster 复制空间水位判断。
《设计的设计》 概念完整性：接口围绕稳定主概念，而非临时开关。 block_alloc_page_for_write() 的请求/结果模型。 新增场景应扩展意图模型，不应堆布尔参数。 避免万能函数逐步腐化为策略垃圾场。
《领域驱动设计》 接口承载通用语言和上下文边界。 origin、page role、media class、alloc policy。 请求字段必须对应真实 FTL 领域词汇。 避免参数名看似通用但语义无法审查。
