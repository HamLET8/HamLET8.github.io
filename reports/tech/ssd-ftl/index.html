计划与已生成报告

15 天计划表 Ready

独立计划文件，不嵌入每日页面。
Day 1：模块边界先于函数实现 Ready

FTL Core 只编排领域动作，不吸收 NAND / Block / Journal 私有知识。
Day 2：建立 FTL 通用语言 Ready

LBA、PPA、open block、victim block、mapping commit、relocation 进入接口和记录格式。
Day 3：接口深度 Ready

Block Manager 用请求结构承载写入意图，避免万能 alloc 参数污染调用者。
Day 4：Host Write 曳光弹 Ready

用一条真实 Host Write 主路径固定 program、journal、mapping commit 和错误补偿边界。
Day 5：Mapping Table 的所有权 Ready

Mapping 拥有 L2P/P2L 当前事实，GC、Recovery、WriteBooster 只能提交条件变更。
Day 6：错误路径不是 return 语句 Ready

NAND 结果先映射为 FTL 决策，再进入 Block、Mapping、Journal、Recovery 的补偿路径。
Day 7：Block 与 Page 生命周期状态机 Ready

用状态机统一 block/page 生命周期，禁止各模块重复判断 valid/free/bad/open。
Day 8：Metadata Journal 的最小可恢复记录 Ready

Journal 记录领域变更意图，而不是转储内存结构；每条记录都应可重放、可幂等、可校验。
Day 9：Power Loss Recovery 的阶段边界 Ready

Recovery 分为扫描、重放、校验、接管四阶段，避免 mount 路径混入正常写路径细节。
Day 10：GC 是领域服务，不是后台线程 Ready

GC 编排空间回收流程，victim 选择、有效页判断、提交和擦除分别由对应模块负责。
Day 11：Victim Selection 与 Wear Leveling 策略分离 Ready

选择策略输出 score 和 reason，GC 流程不散落 erase count、valid count、hot/cold 判断。
Day 12：SLC Cache / WriteBooster 的边界 Ready

WriteBooster 是写入加速上下文，主 FTL 写路径不理解 SLC/TLC 迁移细节。
