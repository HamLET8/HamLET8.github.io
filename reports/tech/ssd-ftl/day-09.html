Day 9 · SSD / FTL 代码设计判断

Power Loss Recovery 的阶段边界：扫描、重放、校验、接管必须分开

今天的核心判断是：Recovery 应分为扫描、重放、校验、接管四阶段，避免 mount 路径混入正常写路径细节。 掉电恢复不是“重新执行一次写入流程”，而是把 NAND 上已经留下的事实收敛到一个可接管的 FTL 状态。

一句话判断 Recovery 是独立状态机，不是普通 IO 的旁路分支 涉及：FTL Recovery / Metadata Journal / Mapping / Block Manager / GC 关键词：scan · replay · verify · handoff · mount boundary · readonly fail-safe

今日设计判断

Recovery 阶段不能调用普通 ftl_write()、GC 主循环或在线分配路径。 它只能调用专门的 replay / verify / rebuild 接口，把持久化证据收敛为运行时状态。

今日涉及模块

Power Loss Recovery、Metadata Journal、Mapping Table、Block Manager、Free Block Pool、 Bad Block Management、GC Relocation、NAND Program / Erase / Read Flow。

今日阅读重点：恢复路径要比正常路径更有边界

掉电恢复处在系统最脆弱的位置：内存状态不存在，free pool 可能不可信，journal 可能被截断， block 状态可能停在中间。此时如果复用正常写路径，会把“在线策略”和“离线收敛”混在一起，错误会被放大。

思想来源 今天必须掌握的概念 落到 SSD / FTL 的判断
《软件设计哲学》 复杂度控制依赖清晰接口和信息隐藏。Recovery 不应知道在线写路径的 open block cursor、cache flush 细节。 把 Recovery 设计成深模块：输入是 checkpoint、journal、NAND metadata；输出是可接管的 Mapping、Block 状态和 free pool。
《程序员修炼之道》 务实工程要求失败可诊断、路径可重复。恢复过程必须可重入，不能依赖一次性副作用。 扫描、重放、校验、接管每个阶段都有明确错误码；失败时进入 read-only 或 rebuild，而不是继续 mount。
《设计的设计》 设计取舍要保持概念完整性。恢复阶段的目标是收敛事实，不是追求启动路径最短。 不要把 mount 写成一条大 if/else；阶段边界允许以后增加 parallel scan、journal compaction、metadata version 迁移。
《领域驱动设计》 恢复不是底层技巧，而是领域状态重新建模。每个阶段都对应领域语言：scan evidence、replay events、verify invariants、handoff runtime。 Recovery 只处理领域证据和领域事件，不直接解释 NAND status bit，也不直接制造普通写请求。

概念融合解释：Recovery 的输出必须是“可接管状态”

四本书在这里约束同一个工程事实：Recovery 的关键不是把所有数据扫一遍，而是建立可靠边界。 扫描阶段只收集证据，不改变 mapping；重放阶段只应用可校验、可幂等的 journal 事件； 校验阶段确认 mapping、block 状态、valid count、free pool 彼此一致；接管阶段才把系统交给在线 FTL。

统一判断： Recovery 不是 mount 过程中的临时补丁，而是一个独立状态机。它必须明确哪些接口允许在恢复期调用， 哪些在线路径必须禁止，哪些错误会导致只读接管或停止挂载。

SSD / FTL 设计场景

Journal Torn Write

扫描阶段发现 record crc 不通过时停止在最后完整 seq。不能继续猜下一个 record，也不能进入在线写路径补洞。

Mapping Replay

重放阶段对 map delta 做幂等提交：已应用则跳过，未应用则检查 old PPA 和 expected seq 后提交。

Block State Rebuild

Block Manager 根据 journal 和 page metadata 重建 OPEN、CLOSED、VICTIM、ERASING、RETIRED 状态，不复用在线分配器。

Free Block Pool

Free pool 在校验阶段从 block 状态重建。只有 FREE 且 erase 校验通过的 block 才能进入 pool。

GC Relocation

掉电发生在 relocation 中间时，Recovery replay relocation delta，而不是重新启动 GC victim selection 策略。

Bad Block After Erase Fail

如果 erase fail 的 retire 事件完整，Recovery 必须把 block 保持 RETIRED；如果证据不足，进入隔离或只读策略。

反例代码：mount 路径混入正常写路径

坏味道： 这段代码在 mount 时边扫描、边修改 mapping、边调用普通写路径和 GC。 它没有阶段边界，错误发生后不知道系统已经改变了哪些运行时结构，也无法安全重入。
反例：Recovery 复用在线路径，状态边界不清 C 
int ftl_mount_bad(struct ftl_dev *ftl)
{
	struct journal_record rec;
	struct ftl_ppa ppa;
	int ret;

	ret = scan_checkpoint(ftl);
	if (ret)
		return ret;

	while (journal_read_next(ftl->journal, &rec) == 0) {
		if (rec.type == REC_MAP_UPDATE) {
			ftl->l2p[rec.lba] = rec.new_ppa;
			block_mark_valid(ftl->bm, &rec.new_ppa);
			block_mark_invalid(ftl->bm, &rec.old_ppa);
		}

		if (rec.type == REC_GC_MOVE) {
			ret = ftl_write(ftl, rec.lba, rec.data);
			if (ret)
				return ret;
		}

		if (rec.type == REC_ERASE_BLOCK) {
			ppa = rec.block_ppa;
			ret = block_erase(ftl->bm, &ppa);
			if (ret)
				ftl->bad_blocks++;
		}
	}

	if (free_pool_low(ftl->bm))
		gc_run_once(ftl);

	ftl->mounted = true;
	return 0;
}
设计问题 代码表现 真实后果
阶段混杂 扫描 journal 时直接修改 mapping、block 和 bad block 计数。 任意一步失败都会留下半恢复状态，下次 mount 无法判断进度。
复用在线写路径 Recovery 中调用 ftl_write()。 可能分配新 open block、追加新 journal、触发缓存策略，污染原始证据。
策略和恢复混合 mount 期间根据 free pool 触发 gc_run_once()。 victim selection 依赖尚未校验的 valid count，可能搬迁错误数据。
错误路径不可判定 erase fail 只增加计数，没有写入 RETIRED 状态。 坏块可能在后续重建 free pool 时重新进入可分配集合。

改进代码：Recovery 作为独立阶段状态机

改进版明确恢复上下文和阶段接口。扫描只构造证据集，重放只调用幂等 replay 接口， 校验重建派生结构，接管才允许在线 IO。任何阶段失败都能报告准确边界。

ftl_recovery.h：恢复期接口边界 C 
#ifndef FTL_RECOVERY_H
#define FTL_RECOVERY_H

#include <linux/types.h>
#include <linux/errno.h>

enum ftl_recovery_phase {
	FTL_RECOVERY_SCAN = 0,
	FTL_RECOVERY_REPLAY,
	FTL_RECOVERY_VERIFY,
	FTL_RECOVERY_HANDOFF,
};

enum ftl_recovery_result {
	FTL_RECOVERY_CLEAN = 0,
	FTL_RECOVERY_REPLAYED,
	FTL_RECOVERY_READONLY,
	FTL_RECOVERY_FAILED,
};

struct ftl_recovery_report {
	enum ftl_recovery_phase failed_phase;
	enum ftl_recovery_result result;
	u64 checkpoint_seq;
	u64 last_valid_journal_seq;
	u32 replayed_records;
	u32 orphan_pages;
	int errno_value;
};

struct ftl_recovery_ctx {
	struct ftl_dev *ftl;
	struct ftl_recovery_report report;
	bool allow_online_io;
	bool require_readonly;
};

int ftl_recover(struct ftl_dev *ftl, struct ftl_recovery_report *report);
int recovery_scan_metadata(struct ftl_recovery_ctx *ctx);
int recovery_replay_journal(struct ftl_recovery_ctx *ctx);
int recovery_verify_invariants(struct ftl_recovery_ctx *ctx);
int recovery_handoff_runtime(struct ftl_recovery_ctx *ctx);

#endif
ftl_recovery.c：四阶段编排 C 
int ftl_recover(struct ftl_dev *ftl, struct ftl_recovery_report *report)
{
	struct ftl_recovery_ctx ctx = {
		.ftl = ftl,
		.allow_online_io = false,
	};
	int ret;

	ctx.report.failed_phase = FTL_RECOVERY_SCAN;
	ret = recovery_scan_metadata(&ctx);
	if (ret)
		goto fail;

	ctx.report.failed_phase = FTL_RECOVERY_REPLAY;
	ret = recovery_replay_journal(&ctx);
	if (ret)
		goto fail;

	ctx.report.failed_phase = FTL_RECOVERY_VERIFY;
	ret = recovery_verify_invariants(&ctx);
	if (ret)
		goto fail;

	ctx.report.failed_phase = FTL_RECOVERY_HANDOFF;
	ret = recovery_handoff_runtime(&ctx);
	if (ret)
		goto fail;

	ctx.report.result = ctx.report.replayed_records ?
		FTL_RECOVERY_REPLAYED : FTL_RECOVERY_CLEAN;
	*report = ctx.report;
	return 0;

fail:
	ctx.report.errno_value = ret;
	ctx.report.result = ctx.require_readonly ?
		FTL_RECOVERY_READONLY : FTL_RECOVERY_FAILED;
	*report = ctx.report;
	return ret;
}

int recovery_replay_journal(struct ftl_recovery_ctx *ctx)
{
	struct journal_record_header hdr;
	u8 payload[JOURNAL_MAX_PAYLOAD];
	int ret;

	while ((ret = journal_read_next_valid(ctx->ftl->journal,
					      &hdr, payload)) == 0) {
		ret = journal_replay_record(ctx, &hdr, payload);
		if (ret == -ESTALE && recovery_record_already_applied(ctx, &hdr))
			ret = 0;
		if (ret)
			return ret;

		ctx->report.last_valid_journal_seq = hdr.seq;
		ctx->report.replayed_records++;
	}

	if (ret == -ENODATA)
		return 0;
	if (ret == -EILSEQ) {
		ctx->require_readonly = true;
		return ret;
	}

	return ret;
}

int recovery_verify_invariants(struct ftl_recovery_ctx *ctx)
{
	int ret;

	ret = map_verify_l2p_p2l_consistency(ctx->ftl->map);
	if (ret)
		return ret;

	ret = block_verify_page_counts(ctx->ftl->bm);
	if (ret)
		return ret;

	ret = free_pool_rebuild_from_block_states(ctx->ftl->bm);
	if (ret)
		return ret;

	return bad_block_verify_retired_set(ctx->ftl->bad_blocks,
					    ctx->ftl->bm);
}

int recovery_handoff_runtime(struct ftl_recovery_ctx *ctx)
{
	if (ctx->require_readonly)
		return -EROFS;

	ctx->allow_online_io = true;
	return ftl_runtime_accept_recovered_state(ctx->ftl,
						  ctx->report.last_valid_journal_seq);
}

阶段表与思想落点表

Power Loss Recovery 阶段责任表

阶段 输入 允许动作 禁止动作 输出
扫描 Scan checkpoint、journal 区间、block/page metadata。 识别最后完整 journal seq，收集 orphan 候选和 block 状态证据。 修改 mapping、分配 page、触发 GC。 recovery evidence set。
重放 Replay 完整 journal record、checkpoint seq。 调用 Mapping / Block Manager 的幂等 replay 接口。 调用普通 write path、普通 GC、在线 wear leveling。 已收敛的核心领域状态。
校验 Verify Mapping、Block 状态、P2L、valid count、retired block set。 检查不变量，重建 free pool，标记只读风险。 接受 host IO 或继续写新 journal。 可接管或不可接管判断。
接管 Handoff 已验证状态、last journal seq、free pool、open block 策略。 初始化在线游标，开启 host IO，恢复 GC 调度。 跳过校验直接 mount。 Online FTL runtime。

恢复错误处理矩阵

错误 发生阶段 领域含义 决策 返回语义
Journal torn record Scan 最后一条记录不完整，不能 replay。 截断到 last valid seq；必要时只读接管。 -EILSEQ 或 0
Map delta stale Replay 事件可能已应用，或 old PPA 与当前事实冲突。 已应用则跳过；冲突则停止恢复。 -ESTALE
Valid count mismatch Verify Mapping 与 Block Manager 派生计数不一致。 从 P2L 重建计数；无法重建则只读。 -EUCLEAN
Free pool contains non-free block Verify 空间池不可信，可能包含 VICTIM、ERASING 或 RETIRED block。 丢弃旧 pool，从 block state 重建。 -EINVAL
Retired block has live mapping Verify 坏块隔离与映射事实冲突。 尝试副本恢复；失败则停止 mount。 -EIO
Readonly required Handoff 核心数据可读，但不能安全写入。 以只读方式接管或拒绝 host write。 -EROFS

四本书思想落点表

书 思想 SSD / FTL 设计位置 它约束了什么 它帮助避免什么问题
《软件设计哲学》 通过深模块和清晰接口隐藏复杂度。 Recovery 编排器、阶段函数、recovery report。 mount 调用者只看到恢复结果，不理解 journal scan 和 block 重建细节。 避免恢复细节泄漏到普通 FTL 读写路径。
《程序员修炼之道》 失败可诊断，路径可重复，状态可验证。 阶段化错误码、只读接管、幂等 replay。 每个阶段都必须能独立失败并报告边界。 避免 mount 失败后留下不可解释的半恢复状态。
《设计的设计》 设计取舍服务概念完整性和未来演进。 Scan / Replay / Verify / Handoff 四阶段架构。 新增 parallel scan、metadata migration、scrub 都必须挂在清晰阶段上。 避免启动流程被临时 if/else 侵蚀成不可维护脚本。
《领域驱动设计》 用领域语言表达状态收敛，而不是硬件动作堆叠。 recovery evidence、journal event、invariant、runtime handoff。 Recovery 处理领域事实，不直接制造普通 host write。 避免恢复路径和在线路径互相污染。
