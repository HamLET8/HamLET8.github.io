Technical Deep Dive

F2FS cp_wait / write_end_io UAF：一个计数归零点如何变成对象生命周期边界

这份报告分析 patch f2fs: avoid cp_wait use-after-free in f2fs_write_end_io()。它表面上只是调整 F2FS_WB_CP_DATA 完成计数和 cp_wait 唤醒方式，实质上是在修复一个更深的边界问题：当 writeback completion 把 checkpoint data 的页面计数降到 0 时，等待 unmount 的线程可能立刻继续并释放 sbi，而 end_io 回调自己还没有停止访问 sbi->cp_wait。

0. Executive Summary

结论先行：真正危险的是“最后一个完成者”

这条 patch 的核心不是普通的 UAF，也不是简单地多加一个锁。它修的是 F2FS 中一个很典型的生命周期错位：f2fs_write_end_io() 在处理 DATA writeback completion 时，会递减 sbi->nr_pages[F2FS_WB_CP_DATA]。如果这次 completion 正好把计数降到 0，等待 f2fs_wait_on_all_pages(sbi, F2FS_WB_CP_DATA) 的线程就可能被唤醒。这个等待者可能是 unmount 路径；它一旦继续，就可能进入 f2fs_put_super() 并释放 sbi。但旧代码在计数归零后还要检查 sbi->cp_wait 是否有 sleeper，并调用 wakeup。也就是说，end_io 自己制造了“别人可以释放我还要访问的对象”的窗口。

patch 采用两个配套动作来修复。第一，在 completion 侧，对 F2FS_WB_CP_DATA 使用 atomic_dec_and_lock_irqsave()，把计数从 1 到 0 的转变和 cp_wait 唤醒放在同一把 waitqueue lock 下。第二，在等待侧，引入 f2fs_prepare_cp_wait()，让 waiter 先进入等待队列，再在 cp_wait.lock 下重新读取计数，避免错过刚刚发生的 wakeup。换句话说，这个 patch 同时修 UAF 窗口和 missed wakeup 窗口。

关键不变量：当某个计数从非零变为零会释放对象生命周期时，最后一个递减者不能在递减之后继续无保护地访问该对象。这个“归零点”必须被当作 release boundary。

1. Patch Facts

事实层：patch 改了两个函数

这条 patch 是 v2，当前 Patchwork 状态为 new。它修改两个位置：fs/f2fs/checkpoint.c 中的 f2fs_wait_on_all_pages()，以及 fs/f2fs/data.c 中的 f2fs_write_end_io()。提交信息明确指出问题链路：f2fs_write_end_io() 先递减 writeback page counter，再唤醒 sbi->cp_wait；如果递减动作让 F2FS_WB_CP_DATA 计数变为 0，等待 unmount 的路径可能继续执行并释放 sbi，而 end_io 还要访问 sbi->cp_wait。

patch 的 Fixes 指向 ce2739e482bc，即早前用于避免 f2fs_write_end_io() UAF 的修复。提交说明还提到另一个历史修复 2d9c4a4ed4ee：那个修复把 warm-node-list 相关的 sbi 访问移动到 dec_page_count() 之前。这说明维护者已经意识到一个原则：当 dec_page_count() 可能让等待者释放 sbi 时，递减之后继续访问 sbi 就不安全。这次 patch 是把同一原则补完整。

位置旧语义新语义意义
f2fs_write_end_io()先普通递减 nr_pages[type]，再检查 cp_wait 并唤醒。对 F2FS_WB_CP_DATA 使用原子递减加 waitqueue lock，归零时锁内唤醒。把归零和唤醒做成一个同步事件，避免归零后继续无保护访问 sbi。
f2fs_wait_on_all_pages()先读计数，后加入等待队列，再睡眠。准备等待后，在 cp_wait.lock 下重新读取计数。避免 waiter 在检查计数和进入睡眠之间错过 wakeup。
解释边界：以上来自 patch 本身和本地 Linux 源码。下面的等待图、调试建议和 UFS 映射是基于这些事实做的工程解释。

2. CP Data Writeback Model

先理解 F2FS_WB_CP_DATA 是什么

F2FS 用 nr_pages[] 统计多类页面状态。F2FS_WB_DATA 是普通 data writeback；F2FS_WB_CP_DATA 是 checkpoint 需要保证的数据写回。二者的区别不是“写到哪里”，而是“这个 writeback 是否被 checkpoint 一致性边界依赖”。在 WB_DATA_TYPE() 宏里，如果 folio 被认为需要 checkpoint guarantee，就会归入 F2FS_WB_CP_DATA。

checkpoint 和 unmount 路径不能只看 dirty data 是否已经提交，它还要确认那些被 checkpoint 语义保护的数据 writeback 已经完成。否则 checkpoint pack、node/meta 状态和 data block 的可见性可能不一致。所以 f2fs_wait_on_all_pages(sbi, F2FS_WB_CP_DATA) 是一个很重的同步边界：它等待的是“所有 checkpoint 相关 data writeback 都完成”。

CP data writeback lifecycle
dirty data普通写入或 fsync/checkpoint 相关数据。
writeback startF2FS 准备 bio，folio 进入 writeback。
count++F2FS_WB_CP_DATA 计数增加。
bio completionf2fs_write_end_io() 被调用。
count--最后一个完成者可能把计数降到 0。
wake cp_waitcheckpoint/unmount 等待者可以继续。

这里的危险在于最后两步。递减计数不是单纯更新统计值，它可能释放等待者；等待者释放后又可能释放 sbi。因此 count-- 不是一个普通原子操作，而是对象生命周期上的边界。

3. Old Flow

旧路径的问题：计数、唤醒、对象释放不是一个原子关系

旧代码在 f2fs_write_end_io() 中对所有类型都调用 dec_page_count(sbi, type)。随后如果类型是 F2FS_WB_CP_DATA，并且计数已经为 0，而且 cp_wait 上有等待者，就调用 wake_up(&sbi->cp_wait)。这个写法直观上没问题：最后一个 I/O 完成后唤醒等待者。但它忽略了一个事实：递减到 0 和访问 waitqueue 之间没有生命周期保护。

// 解释性伪代码，表示旧顺序
end_io(bio):
  for each folio:
    type = WB_DATA_TYPE(folio)
    dec_page_count(sbi, type)

    if type == F2FS_WB_CP_DATA and get_pages(sbi, type) == 0:
      if cp_wait has sleeper:
        wake_up(cp_wait)

    folio_end_writeback(folio)

如果没有 unmount 并发，这种顺序大概率可以工作。等待者被唤醒，end_io 继续收尾，folio writeback 被清掉。但 unmount 改变了语义：unmount 本身就是对象生命周期收尾路径，它等待 CP data writeback 完成的目的，就是确保可以销毁文件系统内部对象。因此，一旦最后一个 completion 把计数降到 0，另一个 CPU 上的 unmount 线程就有理由继续释放 sbi。

The unsafe gap
CPU0 end_io递减 nr_pages[F2FS_WB_CP_DATA]。
计数变 0等待条件满足。
CPU1 unmount等待路径返回，继续释放 superblock 私有数据。
CPU0 继续访问仍要检查和唤醒 sbi->cp_wait。

4. Failure Timeline

UAF 形成过程：不是“没加锁”，而是“锁保护的对象错了”

这类 bug 容易被误解为简单的 waitqueue lock 使用问题。更准确的说法是：旧代码没有把 nr_pages 的零转变和 cp_wait 的唤醒合并成一个受保护的临界区。nr_pages 是一个原子计数，但原子计数只能保护自己的数值变化，不能自动保护“数值变化导致的对象释放”。

时间CPU0: writeback end_ioCPU1: unmount/checkpoint wait状态
T0进入 f2fs_write_end_io()，处理 CP data folio。已经在 f2fs_wait_on_all_pages() 等待。F2FS_WB_CP_DATA == 1
T1调用 dec_page_count()。等待条件即将满足。计数从 1 变 0。
T2还没执行 waitqueue 检查。被唤醒或超时重检后继续 unmount。sbi 生命周期开始结束。
T3继续访问 sbi->cp_wait。可能已经经过 f2fs_put_super()。UAF 窗口。

这个时间线还有另一个伴生风险：missed wakeup。等待侧旧逻辑先检查计数，再加入等待队列。如果 completion 恰好在这两个动作之间发生并完成唤醒，那么等待者可能在条件已经满足后才睡下，只能等默认 timeout。这个问题未必导致 UAF，但会制造不必要的长尾延迟。

5. Fix Logic

修复策略：把“归零”和“唤醒”变成同一个同步事件

patch 在 completion 侧使用 atomic_dec_and_lock_irqsave()。这个 API 的语义正好适合这里：只有当原子变量递减后变成 0 时，才获取指定 spinlock 并返回 true。F2FS 用的锁是 sbi->cp_wait.lock。因此“最后一个 CP data completion”会在持有 waitqueue lock 的情况下执行 wakeup；非最后一个 completion 不需要拿这个锁。

// 解释性伪代码，表示新顺序
if type == F2FS_WB_CP_DATA:
  if atomic_dec_and_lock(nr_pages[type], cp_wait.lock):
    wake_up_locked(cp_wait)
    unlock(cp_wait.lock)
else:
  dec_page_count(sbi, type)

等待侧新增 f2fs_prepare_cp_wait()，先调用 prepare_to_wait()，再拿 cp_wait.lock 读取计数。这样，waiter 不会在“检查条件”和“正式入队睡眠”之间留下可见性空洞。如果 completion 已经把计数降到 0，waiter 会在锁内看到条件已满足并直接跳出；如果 completion 之后才发生，completion 会在同一把锁下唤醒 waiter。

Fixed synchronization
waiter加入 cp_wait，拿 waitqueue lock 重检计数。
↔
same lockcp_wait.lock 序列化 wait condition 和 wakeup。
↔
completion最后一个 completion 归零时锁内 wake。

这个修复没有改变 F2FS 的 checkpoint 策略，也没有改变 data writeback 的提交方式。它只重新定义了同步边界：F2FS_WB_CP_DATA 的最后一个完成者必须持有 cp_wait.lock 完成归零和 wakeup，等待者必须在同一把锁下验证自己是否真的还需要睡眠。

6. Waitqueue Semantics

为什么这不是普通 wakeup，而是 release/acquire 风格的协议

等待队列的正确用法通常包含三个动作：把当前任务放到等待队列，检查条件，条件不满足再睡眠。这里难点在于条件变量不只是一个普通布尔值，而是 nr_pages[F2FS_WB_CP_DATA]。这个计数的零值会让 unmount 这种释放路径继续推进。也就是说，等待条件和对象生命周期绑定在一起。

因此，completion 侧的 wake_up_locked() 不只是为了让等待者更快醒来；它还表达了“在我释放这批 CP data writeback 之前，等待者已经能以一致方式观察到计数为 0”。等待侧的重检也不是多余，它防止在睡眠前错过已经发生的最后 completion。

普通计数

递增和递减只影响统计值，读者下一次读到即可。

等待条件

计数为 0 会唤醒睡眠线程，需要防 missed wakeup。

生命周期边界

计数为 0 还可能让对象释放路径继续，这时最后递减者必须更保守。

这也是为什么旧注释“应该在 folio_end_writeback() 前访问 sbi”并不够。真正危险的不是 folio_end_writeback() 本身，而是 dec_page_count() 已经可以让等待者推进。patch 把这个更早的 release boundary 显式化了。

7. Source Walkthrough

逐函数导读：这条 patch 应该怎样沿源码读

读这条 patch 时，不建议只盯着 f2fs_write_end_io() 的 diff。正确路径应该从三个方向同时读：谁增加 F2FS_WB_CP_DATA 计数，谁等待这个计数归零，谁在归零后可能释放 sbi。这三个方向合起来，才能解释为什么一个 waitqueue wakeup 会变成 UAF。

源码位置要看什么和本 patch 的关系
fs/f2fs/f2fs.hWB_DATA_TYPE()、enum count_type、nr_pages[]、cp_wait。定义了 CP data writeback 的计数对象和等待队列对象。
fs/f2fs/data.cf2fs_write_end_io() 如何处理 bio 中每个 folio。completion 侧在这里递减计数、处理错误、清 gcing 标志、结束 writeback。
fs/f2fs/checkpoint.cf2fs_wait_on_all_pages() 如何等待 dirty meta 或 CP data writeback。waiter 侧在这里提交 merged DATA write、入队等待、睡眠和重检。
fs/f2fs/super.cunmount 路径如何 flush merged writes 并等待 F2FS_WB_CP_DATA。说明等待返回后为什么可能进入对象释放路径。
mm/filemap.cfolio_end_writeback() 如何清 writeback 并唤醒等待者。帮助区分 folio writeback 完成和 F2FS 内部计数归零这两个边界。

这里最容易混淆的是两个“完成”。第一个完成是设备 I/O 完成，block layer 调用 bio end_io；第二个完成是文件系统和 page cache 收尾完成，F2FS 递减内部计数并调用 folio_end_writeback()。这条 patch 修的是第二个完成中的顺序。设备已经把 I/O 完成上报了，但文件系统 completion 回调还没有安全结束。

另一个容易混淆的点是 F2FS_WB_CP_DATA 和 F2FS_WB_DATA。普通 data writeback 完成后，等待者通常不会因此释放整个 sbi；CP data writeback 完成则可能是 checkpoint/unmount 的最后一道门。它的归零语义更强，所以 patch 只特殊处理 F2FS_WB_CP_DATA，而不是把所有 data writeback 都改成 waitqueue lock 下递减。

checkpoint.c找到 waiter 如何睡眠。
super.c找到 unmount 如何释放对象。

8. State Machine

把问题画成状态机：关键是从 count=1 到 count=0

这条 patch 的状态机很小，但非常严格。F2FS_WB_CP_DATA 大于 0 时，checkpoint/unmount 不能继续释放相关对象；等于 0 时，等待者可以继续。于是状态机的危险边就是 1 -> 0。这条边由最后一个 writeback completion 触发。如果这个 completion 触发了等待者继续运行，那么它自己在这条边之后就不能再随意访问 sbi。

CP data writeback state machine
Activenr_pages[CP_DATA] > 0，还有 CP data writeback 未完成。
Last completionend_io 处理最后一个 folio。
Zero transition计数变 0，等待条件满足。
Release allowedcheckpoint/unmount 可以继续。

旧代码的问题是把 Zero transition 当作普通原子递减。新代码把它提升为同步事件：只有最后一个 completion 才拿 cp_wait.lock，并在锁内唤醒等待者。这相当于把状态机的边从“无保护数值变化”改成“受 waitqueue lock 保护的发布事件”。

等待侧的状态机也要配套。waiter 不能先读计数再入队，因为这样会有一个窗口：completion 已经完成 wakeup，但 waiter 还没进入队列，随后 waiter 继续睡眠。新代码先准备等待，再在锁内读条件。如果条件已经满足，就不睡；如果条件未满足，后续 completion 会在同一把锁下 wake。这个模式是典型的条件等待协议。

安全状态机约束：
1. waiter 入队和条件重检必须同锁排序。
2. 最后 completion 的归零和 wakeup 必须同锁排序。
3. 归零之后 completion 侧不能再无保护访问可能被释放的 sbi 字段。
4. 非 CP data writeback 不应无谓进入这个慢路径。

9. Failure Modes

这类问题的表现不只有 UAF，也可能是超时和长尾

提交信息的主线是 UAF，但同一类顺序错误也会制造 missed wakeup。missed wakeup 的表现更隐蔽：没有崩溃，没有明确的 use-after-free，只是某个等待者睡到 DEFAULT_SCHEDULE_TIMEOUT 才醒。用户看到的是 unmount、关机、fsync 或 checkpoint 偶发慢；设备侧可能没有异常，因为 I/O 已经完成了，问题发生在 completion 收尾之后。

这对排查很重要。很多存储长尾不是单纯设备延迟，也不是单纯文件系统慢，而是等待条件已经满足但等待者没有及时观察到。这个 patch 通过把 waiter 重检和 completion wakeup 放到同一把 waitqueue lock 下，减少了这种“条件已满足但任务还睡着”的窗口。

失败形态外部表现应该抓的证据
UAFKASAN 报告，位置可能靠近 cp_wait 或 sbi。end_io 递减到 0 后，unmount 进入释放路径，completion 仍访问 sbi。
missed wakeupcheckpoint/unmount 偶发睡满 timeout。计数已经为 0，但 waiter 没有及时被唤醒。
假性设备长尾用户感知存储慢，但 UFS command latency 正常。设备 completion 到 F2FS wait 返回之间出现异常间隔。

7. Review Checklist

读这条 patch 时应该检查什么

第一，检查所有 F2FS_WB_CP_DATA 的递增路径是否仍然和这个递减路径匹配。F2FS 在写 bio 构造时增加计数，在 completion 中减少计数；这次 patch 只特殊处理 CP data 类型，所以普通 F2FS_WB_DATA 不应该受影响。

第二，检查等待侧是否有双重入队问题。patch 的等待循环里会在进入 sleep 前和循环开头都调用准备函数，因此必须保证 finish_wait() 的位置正确，不会让同一个 wait entry 留在队列里，也不会在错误路径泄漏等待状态。

第三，检查锁顺序。cp_wait.lock 是 waitqueue 自带锁；completion 侧在 bio end_io 上下文中运行，不能引入可能睡眠的锁。当前 patch 用 spinlock/irqsave 语义，符合 completion 路径约束。等待侧在准备睡眠和重检条件时短暂持锁，也没有在锁内提交 I/O 或睡眠。

检查项应该成立为什么重要
最后递减者只有 F2FS_WB_CP_DATA 归零时拿 cp_wait.lock 并 wake。避免普通 data writeback 被无意义加锁拖慢。
等待者重检入队后在同一 waitqueue lock 下检查 get_pages()。避免 missed wakeup 和 timeout 长尾。
错误路径CP error 或 close 场景能正确跳出并 finish_wait()。避免等待队列状态泄漏。
completion 约束不在 end_io 中睡眠，不持锁执行复杂逻辑。避免把修复变成新的 completion path 风险。

8. Debugging

如果要复现或验证，应该抓哪些信号

这个 bug 的复现条件比较苛刻，因为需要 CP data writeback completion、checkpoint/unmount 等待和对象释放在很窄的窗口里交错。比起直接写一个简单 workload，更现实的方法是用 fault injection、KASAN、tracepoint 和延迟注入扩大窗口。

Trace plan

end_io记录 f2fs_write_end_io() 进入、递减、wakeup、退出。
unmount记录 f2fs_wait_on_all_pages() 返回和 f2fs_put_super() 进入。
KASAN/KCSAN观察 UAF 或竞态告警。
建议观察点：
- f2fs_write_end_io(): type, nr_pages before/after, bio status
- f2fs_wait_on_all_pages(): type, wait start/end, timeout count
- f2fs_put_super(): start/end and sbi release timing
- block completion latency: DATA write bio completion distribution
- KASAN/KCSAN: UAF or race report around cp_wait / sbi

性能上也要看 timeout 长尾。如果旧路径发生 missed wakeup，不一定马上 UAF，但可能让 waiter 睡到 DEFAULT_SCHEDULE_TIMEOUT。在移动设备上，这种 timeout 可能表现为卸载、关机、重启、应用数据同步过程中的偶发长尾。

9. UFS Implication

对 UFS 固件/性能分析的启发

这条 patch 不是 UFS patch，但它对 UFS 性能分析很有价值。用户看到的“存储卡住”不一定发生在设备请求执行阶段。这里的问题发生在 bio completion 之后、filesystem waitqueue 和 superblock lifetime 之间。设备已经完成 I/O，block layer 已经回调，真正出错的是上层 completion 收尾和等待者释放路径的顺序。

做 UFS 固件分析时，不能只看 command latency。如果某个场景出现关机、卸载、fsync 或 checkpoint 长尾，而 UFS 侧命令延迟并不高，需要继续向上看 F2FS 的 writeback counters、checkpoint wait、folio writeback/end_io。尤其是最后一个 I/O completion，它在设备侧看起来只是普通 completion，在文件系统里却可能是释放整个等待链的事件。

设备侧视角

UFS command 正常完成，CQ/interrupt/completion 都没有明显异常。

文件系统视角

completion 清理顺序错误，等待队列和对象生命周期交错，导致 UAF 或 timeout 长尾。

因此，端到端 trace 应该把 UFS completion timestamp 和 F2FS f2fs_write_end_io() timestamp 对齐。只有这样才能区分“设备完成慢”和“设备完成后上层收尾慢”。
