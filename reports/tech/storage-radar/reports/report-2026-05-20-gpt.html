#1UFSnew · v7 · needs reviewraceplatformufs-direct

[v7,5/5] scsi: ufs: ufs-qcom: Remove NULL check from devm_of_qcom_ice_get()

主要内容

这条是 qcom ICE race 修复系列里的 UFS consumer 侧清理。系列已经把 devm_of_qcom_ice_get() 的返回语义收紧为不会返回 NULL，因此 ufs-qcom 里继续保留 NULL 分支会误导错误路径；patch 删除 NULL 检查，保留正常的错误处理。

为什么值得看

它看起来只是 cleanup，但真正值得看的是 provider/consumer probe 顺序。UFS host 依赖 Qualcomm ICE inline crypto 资源，如果 provider 尚未 ready，consumer 必须得到可区分的错误状态，而不是把 NULL 当成“没有 ICE”继续跑。对固件/平台 bring-up 来说，这类 race 会表现成偶发初始化失败或 crypto 能力不一致。

预备知识：provider/consumer probe race

Linux 平台驱动经常通过 devm_* helper 获取由另一个驱动发布的资源。关键点不是 helper 名字，而是返回协议：成功对象、ERR_PTR、EPROBE_DEFER、或 NULL。返回协议一旦变更，所有调用方的分支都要跟着收敛，否则错误路径会掩盖真实初始化问题。

阅读检查点

  • 确认 devm_of_qcom_ice_get() 的所有调用方是否都按新返回语义处理。
  • 重点看 v7 之前 reviewer 是否质疑过 EPROBE_DEFER、optional ICE 或 OP-TEE/TZ 场景。
  • 检查删除 NULL 分支后，非 ICE 平台和 ICE probe 延迟平台是否仍有明确路径。

#3UFSnew · v3 · needs reviewperformancetracepointufs-core

[v3,08/11] scsi: ufs: Use trace_call__##name() at guarded tracepoint call sites

主要内容

这条把已经由 trace_foo_enabled() 保护的 UFS tracepoint 调用，从 trace_foo() 改为 trace_call__foo()，避免再次执行 static branch 判断。

为什么值得看

这是典型的热路径微优化：单次收益很小，但位置在 UFS core 的 trace 调用点，可能覆盖 request/completion/error 等频繁路径。它还反映了现代内核 review 的一个重点：可观测性不能在未启用时给热路径留下额外成本。

预备知识：tracepoint static branch

内核 tracepoint 通过 static key/static branch 降低未启用时开销。如果调用点外层已经判断 enabled，再调用普通 trace 宏会重复判断。trace_call__* 适合只在已经确认 enabled 的 guarded call site 使用。

阅读检查点

  • 逐个确认替换点外层确实有 trace_*_enabled() guard。
  • 确认 trace_call__* 不改变参数求值、RCU 或 preempt 相关语义。
  • 关注 reviewer 是否要求把同类改动保持一致，而不是只改 UFS 局部。

#4F2FSnew · v1 · needs reviewlockingwritebackgc

[f2fs-dev] f2fs: fix potential deadlock in gc_merge path of f2fs_balance_fs()

主要内容

这条修复 gc_merge 挂载选项下 f2fs_balance_fs() 的潜在死锁：写回路径缓存 data bio 后唤醒 foreground GC 并等待，而 truncation 路径等待 folio writeback 清除，GC 又可能卡在同一个 folio lock 上。patch 的方向是在 GC_MERGE 路径也提交 cached bios，和 NOGC_MERGE 路径保持一致。

为什么值得看

这是今天最值得看的非 UFS patch。它把 writeback、folio lock、F2FS GC、waitqueue 连成一个具体死锁环，适合训练“不要只看单个函数，要把等待关系画出来”的能力。对移动存储性能调优也有意义，因为 GC/writeback 死锁或长等待经常被用户感知为卡顿。

预备知识：F2FS GC 与 writeback 等待环

F2FS 在空间紧张时会触发 GC。GC_MERGE 试图把 foreground GC 合并到后台线程，减少重复工作，但它也引入了等待队列。如果调用方持有或间接依赖 folio writeback 状态，就要避免“等待 GC，而 GC 又等待当前 folio”的环。

阅读检查点

  • 把 patch 描述中的三条执行路径画成 wait-for graph。
  • 确认提交 cached bios 是否会改变原有 write ordering 或错误处理。
  • 注意 Cc stable 和 Fixes tag，判断这是 bug fix 还是行为收敛。

[f2fs-dev] f2fs: don't BUG on node footer mismatch in f2fs_write_end_io

主要内容

这条处理 f2fs_write_end_io() 中 node footer mismatch 时继续 BUG 的问题。此前 sanity helper 已经能标记 SBI_NEED_FSCK 并报警，但 write_end_io 后面仍保留 f2fs_bug_on()，导致 fuzzed on-disk image 仍能触发 kernel BUG。patch 的实际目标是把可恢复/可标记的坏盘状态从 panic 路径移出来。

为什么值得看

这条值得看，因为它体现了 syzbot 类问题的常见修复原则：不是所有一致性错误都应该 BUG；如果错误来自可疑介质或 fuzzed image，内核更好的行为通常是标记需要 fsck、隔离错误并返回。这个思路对存储固件也有参考价值：错误分类比单纯 assert 更重要。

预备知识：end_io 错误分类

end_io 是 I/O 完成路径，常在 softirq/kworker 上下文执行。这里的 BUG 会把一次坏数据或坏元数据直接升级成系统级崩溃。文件系统通常要区分“内核状态不可能发生”与“介质/镜像损坏导致的不一致”。

阅读检查点

  • 确认 helper 返回错误后，调用方是否真正停止后续危险处理。
  • 看 reviewer 是否要求保留调试信息但降低为 warning。
  • 关注 syzbot reproducer 是否覆盖 write_end_io 和 read_end_io 两条路径。

iomap: don't make REQ_POLLED imply REQ_NOWAIT

主要内容

这条把 iomap direct I/O 的 REQ_POLLED 与 REQ_NOWAIT 语义拆开：轮询完成不等于提交过程不能阻塞。patch 删除 bio_set_polled helper，改为只设置 REQ_POLLED。

为什么值得看

这是性能调优必须关注的语义边界。UFS 或 block 性能实验里经常会用 polling、direct I/O、NOWAIT 组合做延迟测试，如果把 polling 误解为 nowait，会把文件系统写路径带到不支持的重试模型里。该 patch 明确指出文件系统写不是简单幂等操作，不能只重试 bio 而不重试整个操作。

预备知识：REQ_POLLED vs REQ_NOWAIT

REQ_POLLED 表示调用方希望通过轮询等待完成，减少中断路径影响；REQ_NOWAIT 表示提交路径不应睡眠。二者优化目标不同。direct I/O 写入还涉及页锁、块映射、元数据和错误回滚，不具备随意 bio 级重试的简单语义。

阅读检查点

  • 确认该改动和 block 层同类修复保持一致。
  • 看 iomap direct write 是否还有其它地方隐式依赖 REQ_NOWAIT。
  • 做 UFS 性能测试时，把 polling、hipri、nowait 三个概念分开记录。
