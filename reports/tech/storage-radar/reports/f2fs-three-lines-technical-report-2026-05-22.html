0. Executive Summary

先给结论：F2FS 卡顿通常不是单条路径的问题

如果你只看 F2FS 的 GC，就会把卡顿理解成“空间回收慢”；如果只看 writeback，就会把卡顿理解成“脏页写回慢”；如果只看锁，就会把卡顿理解成“某个锁竞争”。这些解释都可能局部正确，但不够完整。真实系统里，空间压力会触发 GC，GC 会读取 SIT/NAT/SSA 并锁 data folio 迁移有效块，writeback 会把 dirty folio 转成 merged bio，checkpoint 会要求数据、节点、元数据达到一致边界，truncate/invalidate 会锁 page cache 并等待 writeback。卡顿发生时，往往是这三条线交汇，而不是某一条线单独失效。

这份报告的核心判断是：F2FS 的“进展性”来自三个边界同时成立。第一，空间回收边界：文件系统必须能从 dirty segment 中找出可回收 victim，并把有效数据迁走，最终释放 section。第二，writeback 边界：已经进入 writeback 的 folio 必须有可达的 bio submit 和 end_io 完成路径。第三，并发等待边界：持有 folio lock、gc_lock、i_gc_rwsem、cp_rwsem 的路径不能形成互相等待的闭环。任何一个边界断掉，用户看到的都是“存储卡住”。

阅读目标：这不是 F2FS 函数百科，而是一套定位框架。读完后你应该能把一个移动端存储卡顿拆成：是否空间不足、是否 GC 正在迁移、是否 bio 已提交、是否 checkpoint 等待、是否 truncate/invalidate 持锁、是否 UFS 设备真的收到请求。

1. Three-line Map

三条线不是并列孤岛，而是互相调用、互相等待。空间回收线负责制造 free section；writeback 线负责把 page cache 的脏数据变成设备 I/O；并发等待线负责保证 page cache、inode、checkpoint 和 GC 迁移的一致性。理解 F2FS 卡顿时，要同时标记每条线上的“对象”和“等待点”。

2. Space Reclaim Line

空间回收线：F2FS 为什么必须 GC

F2FS 是面向闪存的 log-structured 文件系统。它倾向于把新数据写到新位置，而不是频繁原地覆盖。这样有利于顺序写，也更符合 NAND flash 和 FTL 的写入特点；代价是旧位置会积累 invalid blocks。F2FS 需要周期性选择包含较多 invalid blocks 的 segment 或 section，把其中仍然有效的数据迁走，然后把整段空间回收出来。

官方 F2FS 文档给出的磁盘布局是理解 GC 的入口：整个卷被划分成 segment，多个 segment 组成 section，多个 section 组成 zone；主要区域包括 CP、SIT、NAT、SSA 和 Main Area。对 GC 来说，Main Area 是被回收的对象，SIT 告诉它每个 segment 有多少有效块和哪些块有效，NAT 帮它从 node id 找到 node block 地址，SSA 帮它从物理块反查所属对象。

Space reclaim object model
Main Areadata/node block 所在区域，也是 GC victim 来源。
SITsegment 的 valid block count 和 validity bitmap。
SSAsummary entry，记录 block owner 信息。
NATnode id 到 node block 地址的映射。
CP一致性边界，决定哪些 NAT/SIT 副本有效。
Free Section写路径真正需要的可分配空间。

空间回收线最容易被误解成“后台整理”。实际上 foreground GC 直接进入写路径的关键路径。当 f2fs_balance_fs() 发现 free section 不足时，写入路径就被迫进入空间回收语境。此时 GC 不再是可有可无的后台工作，而是当前写入能否继续前进的前置条件。

源码中 f2fs_gc() 会先判断是否已经没有足够 free sections，如果空间不足，会提升为 foreground GC，并且在有 prefree segments 时先尝试 checkpoint，让 prefree 变成真正可用的 free segments。这个细节很重要：GC 和 checkpoint 不是完全分离的两个世界。GC 迁移可以制造 prefree，checkpoint 才能把某些状态固化，让空间真正释放。

3. Victim Selection

Victim selection：不是随便找一段搬走

F2FS 的 victim selection 是空间回收线的第一关。它要在 dirty segment 或 dirty section 中选择一个最值得回收的对象。这里有几个维度：GC 类型是 foreground 还是 background，分配模式是 LFS 还是 SSR/AT_SSR，GC 策略是 greedy、cost-benefit 还是 age-threshold，搜索范围是否受 max_victim_search 限制，是否是 large section，是否存在 background GC 预选的 victim。

主线源码里的 select_policy() 会根据分配模式和 GC 类型设置搜索 bitmap、搜索范围和 cost 模式。foreground GC 和 urgent GC 会扩大候选范围，因为它们的目标是保证当前系统前进；background GC 更倾向于限制搜索，避免频繁打扰系统。这个差异解释了为什么同样是 GC，前台卡顿和后台整理的行为完全不同。

Victim selection flow

→

→

一个关键点是：victim selection 的目标不是找“最脏”的 segment，而是找“搬迁成本和释放收益最合适”的 section。greedy 更直接看有效块数量，cost-benefit 会考虑 age 与有效块比例，age-threshold 模式还会把时间因素引入选择。对于移动设备，age 很重要，因为刚写入的数据很可能很快又被更新或删除；过早 GC 它们会增加写放大。

foreground GC 还有一个特殊逻辑：它可以复用 background GC 之前标记过的 victim section，因为这些 section 通常已经被认为有效块较少。这个机制说明 BG_GC 不只是独立后台任务，它还会为 FG_GC 留下候选信息。但 foreground GC 也要跳过 pinned section，否则可能在不可迁移数据上浪费时间。

策略/条件目的卡顿分析意义
FG_GC保证当前写路径能继续前进。用户可感知延迟；如果 victim 难找，写入直接卡。
BG_GC空闲时提前整理 dirty segments。可能减少未来 FG_GC，但也可能与业务 I/O 竞争。
Greedy倾向选择有效块少的 segment。释放空间快，但不一定最优写放大。
Cost-benefit / age考虑有效块比例与冷热时间。更接近长期效率，但选择过程更复杂。
Pinned / busy / CP disabled filter跳过不能安全回收的 section。空间足够少时，这些过滤会放大 foreground GC 长尾。

4. SIT / NAT / SSA

SIT、NAT、SSA 如何参与 GC

要理解 GC 数据迁移，必须把 SIT、NAT、SSA 的角色分清。SIT 是 segment 级的状态表，记录有效块数量和有效位图；SSA 是 summary area，记录每个物理 block 的 owner 信息，例如 nid 和 ofs_in_node；NAT 是 node address table，用来把 node id 映射到 node block 的物理地址。GC 要从物理 victim block 反推出逻辑对象，必须靠这些元数据协作。

在 do_garbage_collect() 中，F2FS 会读取 victim segment 对应的 summary page。对于 data segment，gc_data_segment() 会分阶段处理：先根据 summary 中的 nid 预读 NAT，再预读 node page，再确认 inode，再预读 data folio，最后迁移 data page。这个多阶段流程不是为了复杂而复杂，而是为了减少随机读和锁持有时间，同时保证迁移的数据仍然有效。

SIT / SSA / NAT data migration
Victim block物理位置：segno + offset
→
SIT checkvalid map 判断这个 block 是否仍有效
→
SSA summary读取 nid、ofs_in_node、owner 信息
NAT lookup根据 nid 找 node block，验证父节点仍有效
→
inode / bidx得到 inode 和文件逻辑块号
→

这条线对并发分析非常关键：GC 不是直接搬物理块，它会回到 inode/page cache 层确认数据。也就是说，GC 会碰到普通写入、truncate、direct I/O、page cache lock、i_gc_rwsem。上一篇 gc_merge 死锁中，GC thread 卡在锁 data folio 上，根本原因就是 data GC 必须回到 page cache 语境。

另一个关键点是有效性检查。GC 扫描到一个 summary entry 时，不能盲目信任它，因为普通写入可能已经让这个 block 失效。源码中 check_valid_map()、is_alive()、NAT/node 验证共同保证“要搬的数据确实还是当前有效数据”。这就是为什么 GC 可能花很多时间做元数据读取与验证，而设备侧未必立即看到大量连续写请求。

对 UFS 性能分析来说，这解释了一个常见现象：应用层卡在删除或写入，F2FS 处于 foreground GC，但 UFS 侧 queue depth 不高。原因可能不是设备不忙，而是文件系统还在 victim 验证、NAT/SSA 读取、folio lock、inode 锁、checkpoint 等上层阶段，尚未形成设备请求。

5. Foreground / Background GC

foreground GC 和 background GC 的切换

background GC 是系统试图在空闲时提前整理空间；foreground GC 是写路径已经没有足够 free section，必须同步回收空间才能继续。两者的区别不是“一个在后台，一个在前台”这么简单，而是目标函数不同：BG_GC 尽量低干扰、低优先级、可跳过；FG_GC 追求进展性，必要时牺牲延迟。

gc_thread_func() 中的逻辑体现了这个差异。后台线程会根据 sleep time、IO idle、dirty segments、urgent mode、zoned 设备空间百分比等条件决定是否运行。启用 GC_MERGE 且 foreground waitqueue 有等待者时，它会把本轮工作视为 foreground 相关，并设置 no_bg_gc 和 nr_free_secs，让 f2fs_gc() 为当前等待者释放至少需要的空间。

FG/BG GC switching
BG_GC空闲检测、trylock gc_lock、可跳过、受 max search 限制。
FG_GC空间不足时强制进入，目标是释放当前写路径所需 section。
GC_MERGEforeground 请求交给 GC thread，调用者等待 fggc_wq。
Urgent GC更短 sleep、更激进，用于高压力或特殊模式。

这解释了为什么 gc_merge 既有价值又有风险。价值在于它把 foreground GC 的执行集中到 GC thread，可以避免多个前台调用者重复执行 GC，也能统一调度。风险在于它把“当前线程自己执行 GC”变成“当前线程等待另一个线程”。一旦当前线程进入等待前还保留着完成条件，例如未提交的 DATA bio，就会形成上一篇报告中的死锁结构。

foreground GC 还可能触发 checkpoint。源码中当 free section 不足且存在 prefree segments 时，f2fs_gc() 会先尝试 f2fs_write_checkpoint()，因为 checkpoint 可以让 prefree segments 真正变成可用空间。这意味着卡顿分析中，GC 和 checkpoint 不是两个独立阶段：foreground GC 可能先卡在 checkpoint，checkpoint 又可能卡在 dirty node、dirty dentry、dirty meta 或 CP data writeback 上。

6. Zoned Devices

zoned 设备为什么会改变 GC 行为

zoned 设备给 F2FS 增加了额外约束。普通块设备可以把 segment/section 看成比较均匀的可写区域；zoned 设备存在 zone size、zone capacity、写指针和顺序写约束。源码中针对 zoned 设备设置了不同的 GC sleep time、no-zoned-GC percent、boost-zoned-GC percent，并且在 large section 场景下根据 zone capacity 调整可回收 segment 范围。

在 do_garbage_collect() 中，注释明确说明 zone capacity 可能小于 zone size，因此可 GC 的 end segno 要扣掉不可用 segment。这个细节对性能分析很重要：你以为一个 zone/section 有完整可回收空间，但 F2FS 实际可用部分更小；GC 的迁移窗口、boost 策略和 one-time GC 都会因此变化。

Zoned GC constraints
Zone size设备公布的 zone 范围。
Zone capacity真正可写容量，可能小于 zone size。
Migration windowBG_GC / one_time 可能限制一次迁移窗口。
Boost空间不足时放大迁移粒度，减少未来 FG_GC。

对移动设备来说，zoned 不是每台设备都会遇到，但这个分支能训练你看 F2FS 的“设备形态敏感性”。同样的文件系统路径，在不同块设备约束下可能选择不同的 victim、迁移窗口和提交节奏。UFS 设备通常不是 host-managed SMR 那样的 zoned block device，但 UFS 固件内部也有 zone-like erase block、SLC cache、FTL GC、thermal throttling 等隐藏结构。F2FS 的 zoned 分支提醒你：上层文件系统策略经常会被设备几何或顺序写约束改变。

7. Writeback Line

writeback 线：从 dirty throttling 到 folio_end_writeback

writeback 线从用户或内核把 page cache 弄脏开始。普通写入会产生 dirty folio；dirty 页面数量增长后，MM 层的 dirty throttling 会限制继续弄脏页面的速度，并唤醒 background writeback。随后 VFS/MM 通过 writeback_control 把写回要求传给文件系统，F2FS 的 f2fs_write_cache_pages() 扫描 dirty folio，锁 folio，清 dirty 标志，调用 f2fs_write_single_data_page() 形成 DATA write。

dirty throttling 的关键不是“写入变慢”，而是“脏页生产速度必须和后端写回能力平衡”。balance_dirty_pages() 会根据全局和 memcg 的 dirty limits、writeback 带宽、pos_ratio 等计算当前任务是否需要暂停。如果系统脏页太多，会启动 background writeback，并让 dirtier 线程 sleep。移动端上，这个阶段可能被误判成设备慢，但实际上还没进入 F2FS 或 UFS 的具体 I/O 路径。

Writeback lifecycle
Dirty folio用户写入或 mmap 修改 page cache。
→
Dirty throttlingbalance_dirty_pages 控制脏页生产速度。
→
writeback_controlWB_SYNC_NONE / WB_SYNC_ALL、range、nr_to_write。
F2FS writepages扫描 PAGECACHE_TAG_DIRTY，锁 folio，清 dirty。
→
merged bioOPU/IPU DATA bio 被合并缓存。
→
end_io设备完成后 f2fs_write_end_io 清理并 folio_end_writeback。

writeback_control 是这条线的控制面。WB_SYNC_NONE 更像后台写回，遇到正在 writeback 的 folio 可以跳过；WB_SYNC_ALL 则是同步语义，常见于 fsync 或 checkpoint，需要等待或重试。F2FS 还会用 wb_sync_req 区分同步写回和普通后台写回，避免混合 IO 拆分或潜在死锁。

F2FS 的 writeback 还有合并策略。f2fs_write_cache_pages() 写完一批 folio 后，会提交和当前 inode 相关的 merged DATA write；如果存在 IPU bio，也会提交 merged IPU write。这个逻辑解释了上一篇 patch 为什么要在 f2fs_balance_fs() 的 GC 等待前提交 cached DATA/IPU bios：正常写回函数尾部会提交，但如果中途进入等待，尾部提交就到不了。

8. Bio Submit Boundary

bio 提交边界：构造请求不等于设备收到请求

writeback 线最关键的工程边界是 bio submit。F2FS 会先构造 bio，并可能把它放在 sbi->write_io[type][temp] 对应的 merge slot 中。真正进入 block layer 的动作发生在 __submit_merged_bio()，最终调用 blk_crypto_submit_bio()。只有跨过这个边界，UFS host、SCSI、block queue、设备固件才有机会看到请求。

这条边界非常容易在分析中被忽略。应用层说“写卡住”，文件系统说“folio writeback”，设备侧却没有请求，这三件事可以同时成立：folio 已经被标记为 writeback，但 bio 还在 F2FS 内部 merge cache。上一篇报告的死锁就是这个边界放错位置：进入 GC thread wait 之前，没有先跨过 DATA bio submit 边界。

阶段状态谁能观察到诊断含义
Dirtyfolio dirty，尚未 writeback。MM、F2FS dirty inode/page counters。设备还没有理由看到请求。
Writeback startedfolio writeback 标志已置位。truncate、fsync、filemap_fdatawait、checkpoint。等待者可能出现，但 bio 未必提交。

f2fs_write_end_io() 是 writeback 线的收口。它会处理 I/O error、checkpoint data write failure、node footer sanity、fsync node entry，然后减少 page count，必要时唤醒 cp_wait，最后调用 folio_end_writeback()。MM 层的 folio_end_writeback() 会唤醒等待 PG_writeback 清除的线程。也就是说，truncate 和 fsync 等待的完成事件，最终落在 end_io 路径上。

因此，在移动端排查时要把 writeback 延迟拆成两个部分：submit 前延迟和 submit 后延迟。submit 前延迟通常是 F2FS/MM/锁/GC/checkpoint 问题；submit 后延迟才是 block/SCSI/UFS/设备问题。如果不区分这两个阶段，就会把大量文件系统等待误判成 UFS 固件长尾。

9. Fsync / Checkpoint

fsync 和 checkpoint 如何要求一致性

F2FS 的一致性不是只靠 data writeback。它采用 checkpoint 机制维护文件系统一致边界。checkpoint 会阻塞或冻结部分 FS operations，刷新 dirty dentry、dirty inode meta、dirty node，提交 merged writes，刷新 NAT/SIT，写 checkpoint pack，等待 meta 和 CP data writeback，必要时 flush device cache，并通过 barrier/FUA 让关键元数据达到设备。

这条路径解释了为什么 fsync 或 checkpoint 卡顿经常比普通写入复杂。普通 writeback 可能只关心 data folio；checkpoint 关心 data、node、meta、summary、NAT、SIT、orphan、checkpoint version 和 device cache。任何一个环节等待，都可能让用户感知到 “fsync 慢” 或 “系统卡”。

Checkpoint consistency pipeline
block_operations锁住关键 FS 操作，刷新 quota、dentry、inode meta、node。
→
flush merged writes提交 data/node/meta merged bios，避免悬挂状态。
→
NAT / SIT flush把地址映射和 segment 状态写入对应区域。
do_checkpoint更新 checkpoint pack、summary、orphan、bitmap。
→
wait writeback等待 dirty meta 和 CP data writeback。
→
device flush / barrier确保一致性边界真正落到设备。

f2fs_wait_on_all_pages() 展示了 checkpoint 与 writeback 的直接关系：如果等待的是 F2FS_WB_CP_DATA，它会提交 DATA merged write，然后睡在 cp_wait 上。f2fs_write_end_io() 在 CP data writeback 完成时会唤醒 cp_wait。这和上一篇死锁报告中的原则一致：等待前必须确保完成路径已启动。

Fsync 不是本报告逐函数展开的重点，但它与 checkpoint 共享核心语义：数据和元数据必须达到可恢复的一致状态。F2FS 的 fsync 可能通过 fsync node、roll-forward recovery、checkpointed data 判断、node writeback 等机制减少全量 checkpoint，但当条件不满足时，仍可能把压力传导到 checkpoint 或 data/node writeback。排查 fsync 慢时，要先判断它是 data writeback 慢、node/meta 慢、checkpoint 慢，还是 device flush 慢。

10. Concurrency Line

并发等待线：truncate / invalidate / GC / checkpoint 怎样互相影响

并发等待线是最容易导致“看起来每个函数都没错，但系统卡住”的部分。truncate 要移除 page cache，需要锁 folio 并等待 writeback；invalidate 也可能锁 folio、等待 writeback、调用 filesystem 的 release/invalidate 钩子；GC 迁移 data block 需要锁 data folio 和 inode 的 i_gc_rwsem；checkpoint 要通过 cp_rwsem 或全局 checkpoint 锁阻塞部分 FS 操作；GC thread 还要拿 gc_lock。这些等待点组合起来，才是 F2FS 长尾卡顿的主要来源。

MM 层的 truncate_inode_pages_range() 明确分两轮：第一轮非阻塞，尽量移除容易处理的 folio；第二轮会等待。对部分 folio，truncate 会通过 __filemap_get_folio(..., FGP_LOCK) 拿锁，然后调用 truncate_inode_partial_folio()，其中会等待 writeback。对整页范围，第二轮也会锁 folio、等待 writeback、再从 page cache 移除。

Concurrency wait objects
folio locktruncate、invalidate、GC data migration 都可能争用。
PG_writebackfsync、truncate、checkpoint、filemap_fdatawait 等待清除。
i_gc_rwsemGC 与普通读写/迁移保护 inode 级数据路径。
cp_rwsem / gc_lockcheckpoint 和 GC 的全局协调点。

GC 的 data migration 也有多重等待。gc_data_segment() 在 phase 3 会尝试拿 i_gc_rwsem[WRITE] 做预读和收集 inode，在 phase 4 对 regular file 同时尝试拿 i_gc_rwsem[WRITE] 和 i_gc_rwsem[READ]，并等待 inflight DIO。之后 move_data_page() 会锁 data folio；foreground GC 还会等待该 folio 原有 writeback 完成，再把数据写到新位置。

这些等待都是为了正确性，不是多余开销。问题发生在等待顺序组合错误时。一个通用判断标准是：如果某线程持有 A 并等待 B，而另一个线程持有或负责 B 的完成事件但等待 C，第三个线程又等待 A，就要立刻画等待图。F2FS 的复杂点在于 A/B/C 可能不是同一种对象：A 可能是 folio lock，B 可能是 writeback completion，C 可能是 GC waitqueue。

11. Locks and Wait Points

关键锁和等待点逐一解释

对象保护什么常见等待者风险
folio lockpage cache 中单个 folio 的状态、截断、invalidate、GC 迁移。truncate、invalidate、GC data path、writeback。持锁等待 writeback 时，会阻塞 GC 锁同一 folio。
PG_writebackfolio 正在写回，完成由 end_io 清除。filemap_fdatawait、truncate、checkpoint、foreground GC。如果 bio 未提交，等待 completion 不可达。
gc_lock全局 GC 执行串行化。GC thread、foreground GC、gc_merge。FG/BG GC 切换时影响前台写入延迟。

一个细节值得单独强调：MM 层的 folio_end_writeback() 注释指出 writeback 不持有自己的 folio 引用，而依赖 truncation 等待 PG_writeback 清除。这说明 truncate 等 writeback 不是偶然实现，而是核心语义。文件系统不能让 PG_writeback 长时间没有完成路径，否则会把 MM 的合理等待变成系统级卡顿。

另一个细节是 invalidate_mapping_pages() 与 truncate_inode_pages_range() 的差异。invalidate 更像“尽量丢弃 clean/unlocked cache”，不会像 truncate 那样强制等待所有状态；如果需要移除所有 page cache，文档和源码都指向 truncate。排查卡顿时，要区分调用方使用的是轻量 invalidate 还是强语义 truncate/hole punch。

12. Diagnosis Framework

三线合并：移动端“文件系统卡住”的诊断框架

当你遇到移动设备上的存储卡顿，建议按三线框架逐层排除，而不是直接从 UFS 设备侧开始。第一问：空间回收线是否激活？看 free sections、foreground GC、victim selection、pinned section、zoned/usable capacity、prefree/checkpoint。第二问：writeback 线是否跨过 bio submit 边界？看 dirty folio、writeback_control、merged bio、block trace、end_io。第三问：并发等待线是否形成闭环？看 truncate/invalidate、folio lock、PG_writeback、GC thread、checkpoint wait、i_gc_rwsem。

设备侧无请求

优先怀疑 submit 前阶段：dirty throttling、F2FS merged bio 未提交、GC 元数据验证、folio lock、checkpoint block_operations。不要先调 UFS firmware。

设备侧有请求且长尾

进入 block/SCSI/UFS 分析：queue depth、request type、flush/discard、runtime PM、MCQ、error handling、device thermal 或 firmware GC。

设备完成但上层仍卡

看 completion 后处理：f2fs_write_end_io、folio_end_writeback、cp_wait、mapping error、node/meta writeback、fsync node entry。

推荐 trace 视角

一次高质量抓取至少要包含四类事件。第一，F2FS GC：trace_f2fs_gc_begin/end、victim selection、foreground/background、skipped_gc_rwsem。第二，F2FS writeback：writepages、submit_merged_write、write_end_io、folio writeback start/end。第三，MM/page cache：truncate/invalidate、folio lock wait、filemap_fdatawait。第四，block/UFS：bio submit、request issue、completion、flush/discard、UFS command latency。

// 分层时间线模板
T0 dirty folio created
T1 dirty throttling / background writeback starts
T2 f2fs_write_cache_pages locks folio and starts writeback
T3 DATA bio cached or submitted?
T4 foreground GC / checkpoint / truncate starts
T5 block request issued to UFS?
T6 device completion
T7 f2fs_write_end_io -> folio_end_writeback -> waiters wake

这个模板能把“卡住”拆成可验证的阶段。如果 T2 到 T3 很长，问题在 F2FS/MM；如果 T3 到 T6 很长，问题才进入 block/UFS/设备；如果 T6 到 T7 很长，问题在 completion 之后的文件系统清理或等待唤醒。这样的分层能避免把所有延迟都归因于设备。

和上一篇 patch 的连接

上一篇 gc_merge patch 正好是三线框架的一个例子：空间回收线进入 foreground GC；writeback 线已经设置 writeback 但没有提交 DATA bio；并发等待线中 truncate 持有 folio lock，GC thread 等 folio lock，Kworker 等 GC thread。修复是把 DATA bio submit 提升到进入 GC_MERGE 等待之前。也就是说，它修复的是三线交汇处的进展性边界。

读完这篇之后，你应该能把类似问题转化成一句话：哪个状态已经发布，哪个完成事件还没启动，哪个线程已经进入等待，哪个锁或 waitqueue 把它们连成环。只要这句话能写出来，修复点通常就不会太远。

13. Function Walkthrough

逐函数导读：把三条线放回源码执行路径

这一节不是列函数名，而是帮你建立“从现象回到源码”的路线。以后你看到 F2FS 卡顿，不能只打开一个函数。正确做法是先判断卡顿落在哪条线，再沿着那条线找边界点。如果是空间压力，先看 GC 入口和 victim；如果是写回，先看 writeback_control、merged bio 和 end_io；如果是等待，先看 folio lock、PG_writeback、gc_lock、i_gc_rwsem 和 checkpoint 锁。

空间回收线的函数顺序

空间回收线可以按这个顺序读：f2fs_balance_fs() 判断 free sections 是否足够；如果不够，可能通过 GC_MERGE 唤醒 GC thread，也可能直接构造 f2fs_gc_control 调用 f2fs_gc()。进入 f2fs_gc() 后，代码会再次判断空间是否不足；如果存在 prefree segments，可能先调用 checkpoint，把这些预释放段变成可用空间。然后 __get_victim() 调用 f2fs_get_victim()，后者根据策略从 dirty bitmap 中选择 victim。最后 do_garbage_collect() 读取 summary page，根据 segment 类型走 node GC 或 data GC。

这条顺序的关键是：f2fs_balance_fs() 不是单纯后台维护函数，它可能从写路径进入，并且决定当前写入是否必须等待空间回收。f2fs_gc() 不是单纯搬数据，它还可能触发 checkpoint。f2fs_get_victim() 不是简单找最脏 segment，它要考虑搜索范围、dirty bitmap、pinned section、CP disabled、age、valid block cost。do_garbage_collect() 不是只写新块，它要先读 SSA，然后用 SIT/NAT 验证数据是否仍然有效。

Space reclaim function chain
balance_fs写路径触发空间平衡。
gc_thread_funcBG/FG/GC_MERGE 切换。
f2fs_gc必要时升级 FG_GC 或 checkpoint。
get_victim从 dirty bitmap 中选 section。
do_gc读 SSA，区分 data/node。
move data锁 folio，迁移有效块。

writeback 线的函数顺序

writeback 线可以按这个顺序读：用户写入产生 dirty folio；MM 的 balance_dirty_pages_ratelimited() 周期性检查 dirty 状态，必要时启动 background writeback 或让当前任务睡眠；VFS 使用 writeback_control 调用 do_writepages()；F2FS 进入 __f2fs_write_data_pages()，再进入 f2fs_write_cache_pages()；该函数按 dirty tag 找 folio，锁 folio，等待旧 writeback，清 dirty，调用 f2fs_write_single_data_page()；最后提交 merged DATA/IPU bio。设备完成后，f2fs_write_end_io() 清理状态并调用 folio_end_writeback()。

这条线最重要的是不要把“写回开始”和“设备开始写”混为一谈。folio_start_writeback() 或 writeback 标志说明 folio 进入写回生命周期；f2fs_submit_merged_write() 才说明 F2FS 准备把对应 bio 推给 block layer；blk_crypto_submit_bio() 之后，block/SCSI/UFS 才有机会看到请求；f2fs_write_end_io() 和 folio_end_writeback() 才是等待者真正被唤醒的收口。

上一篇 patch 的关键就在这里：正常情况下 f2fs_write_cache_pages() 尾部会提交 merged writes，但如果 writeback 中途进入 f2fs_balance_fs() 并在 GC_MERGE 路径睡眠，函数尾部就到不了。于是 folio 已经 writeback，bio 还没 submit。这个窗口不是性能细节，而是进展性边界。

并发等待线的函数顺序

并发等待线可以按这个顺序读：truncate 或 hole punch 进入 truncate_inode_pages_range()；第一轮尽量非阻塞移除 page cache；第二轮会锁 folio 并调用 folio_wait_writeback()；invalidate 路径也可能锁 folio、等待 writeback、调用 filesystem 的 release/invalidate 钩子。GC data path 在 gc_data_segment() 的后期阶段会尝试拿 inode 的 i_gc_rwsem，等待 DIO，之后调用 move_data_page() 锁 data folio。checkpoint 则通过 block_operations()、cp_global_sem、cp_rwsem、cp_wait 把一致性边界和 writeback 绑定起来。

这里最容易误判的是“等待对象类型不一致”。一个死锁环不一定都是 mutex。它可能是 folio lock、PG_writeback、waitqueue、bio completion、gc_lock 的混合。例如 Kworker 等 fggc_wq，GC thread 等 folio lock，Truncator 等 PG_writeback，PG_writeback 又等 Kworker 提交 bio。这种环不会总是被传统 lockdep 直接表达出来，所以你必须手工把等待条件画成图。

14. Three Intersection Cases

三个交汇案例：如何判断卡顿来自哪条线

案例一：删除大文件时系统卡住

用户删除大文件或应用清理缓存时，表面动作是 unlink/truncate，但实际可能同时触发三条线。truncate 需要移除 page cache，可能锁 folio 并等待 writeback；文件删除会让旧 data blocks 变 invalid，空间回收线未来可以受益；如果空间已经紧张，foreground GC 可能正在迁移同一 inode 的 data block；writeback 线可能还在提交或等待同一文件的 DATA bio。这个场景下，设备侧不一定立刻有大量请求，因为系统可能卡在 folio lock 或 writeback wait 上。

诊断时先看线程栈：是否有 truncate 线程在 folio_wait_writeback()，是否有 GC thread 在 f2fs_get_lock_data_folio() 或 move_data_page()，是否有 kworker 在 F2FS writeback 或 f2fs_balance_fs()。然后看 block trace：对应 DATA bio 是否已经提交。如果没有请求，问题在 submit 前；如果请求已提交但 completion 慢，再进入 UFS 侧分析。

案例二：fsync 长尾很高

fsync 慢不等于单个 data write 慢。F2FS 可能需要写 data、写 node、记录 fsync node、等待 filemap writeback，某些条件下还可能触发 checkpoint。checkpoint 会 flush dirty dentry、dirty inode meta、dirty node、NAT、SIT、summary、checkpoint pack，并等待 device cache flush。任何一环慢，用户看到的都是 fsync 慢。

诊断时要拆出 fsync 的阶段。如果 data writeback 未提交，先看 merged bio 边界；如果 data 已完成但 node/meta 未完成，看 dirty node 与 checkpoint；如果 checkpoint 到了 device flush，看 flush/FUA 在 UFS 上的 latency；如果设备完成后仍卡，看 cp_wait、mapping error、folio_end_writeback 和唤醒路径。不要用一个总 fsync latency 去推断设备固件慢。

案例三：写入吞吐突然下降但 UFS queue depth 不高

这类现象很常见：应用写入变慢，UFS 侧 queue depth 却不高。可能原因是 dirty throttling 已经让 dirtier 线程睡眠；也可能是 F2FS 在 foreground GC 中做 victim selection、NAT/SSA 读取或 folio lock 等待；还可能是 checkpoint 正在 block operations 中刷新元数据；也可能是 merged bio 没到 submit 边界。设备侧 queue depth 不高，只能说明请求没有大量进入设备，不能说明上层没有压力。

这种场景要看三个计数：dirty pages 是否高，foreground GC 是否高，checkpoint 或 CP data writeback 是否高。再看 block trace 中普通 write、flush、discard 的比例。如果普通 write 少但 flush/discard 多，可能是 checkpoint 或清理路径影响；如果 block trace 也少，优先看 F2FS 和 MM 的 submit 前等待。

Intersection diagnosis
卡在 submit 前dirty throttling、GC metadata、folio lock、checkpoint locks、merged bio 未提交。
卡在设备中block request 已 issue，UFS command latency、flush、discard、error recovery 长尾。
卡在 completion 后end_io、folio_end_writeback、cp_wait、mapping error、fsync node 清理。

15. UFS Mapping

映射到 UFS 工作：哪些现象不能直接归因于设备

你做 UFS 固件开发时，最容易被误导的是“用户说存储卡，所以下意识看设备”。但 F2FS 三条线告诉我们，用户态卡顿可以发生在设备请求产生之前。dirty throttling 会让用户线程主动睡眠；foreground GC 会在文件系统里读 NAT/SSA、锁 folio、等待 DIO；checkpoint 会等待 dirty node/meta/CP data；truncate 会持有 folio lock 等 writeback。所有这些阶段都可能让用户感觉存储慢，但 UFS 侧没有对应 command。

因此，做 UFS 性能调优时，trace 必须分层。第一层记录应用或 VFS 事件：write、fsync、truncate、unlink 的开始结束。第二层记录 MM/F2FS：dirty throttling、writepages、GC begin/end、checkpoint begin/end、submit_merged_write、write_end_io。第三层记录 block/SCSI/UFS：bio submit、request issue、UFS command send、completion、timeout、flush/discard。只有这样才能判断延迟在哪一层产生。

用户现象先检查 F2FS/MM再检查 UFS
删除文件卡truncate 是否等 writeback；GC 是否等 folio lock；checkpoint 是否进行。如果已有 discard/flush/write request，再看命令延迟。
fsync 慢data/node/meta writeback、checkpoint、cp_wait、device flush 前后。flush/FUA latency、cache flush 语义、UFS power state。
吞吐掉速dirty throttling、foreground GC、writeback_control、merged bio submit。queue depth、MCQ、thermal、firmware GC、SLC cache。
周期性长尾checkpoint 周期、background GC、discard、segment cleaner。flush/discard 长尾、background operations、error recovery。

一个实用规则是：如果 UFS command trace 中没有对应请求，就不要先讨论设备延迟；如果 block request 已经 issue 但 UFS completion 慢，再进入固件和 host 分析；如果 UFS completion 已经返回但上层仍然卡，回头看 end_io、folio writeback 清除和等待者唤醒。这个三段式能减少大量无效排查。

16. Study Plan

后续阅读路线：按三条线逐步深入

后续建议按“先对象、再路径、再等待”的顺序读。第一周读空间回收线：重点看 F2FS layout、SIT/NAT/SSA、victim selection、gc_data_segment() 的 phase 机制、foreground/background GC 切换。目标是能解释一个物理 victim block 如何被验证为有效数据，并被迁移到新位置。

第二周读 writeback 线：重点看 MM dirty throttling、writeback_control、f2fs_write_cache_pages()、merged bio、IPU/OPU、f2fs_write_end_io()、folio_end_writeback()。目标是能把一次写入拆成 dirty、writeback start、bio cached、bio submitted、device completion、writeback cleared 六个阶段。

第三周读并发等待线：重点看 truncate_inode_pages_range()、invalidate_inode_pages2_range()、folio lock、PG_writeback、i_gc_rwsem、gc_lock、checkpoint 的 block_operations 和 cp_wait。目标是看到任意卡顿栈时，能快速写出等待图。

最后把三条线合并到一个真实 workload：空间接近满时，一边写数据库，一边删除文件，一边触发 fsync。你需要记录每层事件，并判断延迟是在 submit 前、设备中还是 completion 后。这个练习比单独读任何一个函数都更接近真实工作。
