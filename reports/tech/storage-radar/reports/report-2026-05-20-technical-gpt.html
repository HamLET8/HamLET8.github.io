1. UFS QCOM ICE：真正的问题不是 NULL，而是 probe 时序

问题拆解

patch 标题是删除 devm_of_qcom_ice_get() 的 NULL 检查，但这只是表层。真正的变化是 helper 的返回协议已经收敛：它不再用 NULL 表达“没有拿到对象”，而是用明确的错误值表达失败或延迟。UFS consumer 侧继续保留 NULL 分支，会让读代码的人误以为 NULL 仍是合法状态，也可能让错误路径的语义变得不一致。

这类问题对 UFS 固件开发者有实际意义。Inline crypto、PHY、clock、power domain 等资源经常由另一个 platform driver 发布。consumer 太早取资源时，正确行为通常不是继续跑，而是推迟 probe 或返回明确错误。否则你看到的可能不是稳定复现的 probe failure，而是某些板子、某些启动顺序下的偶发能力缺失。

阅读目标

不要只看删除了几行。要追 helper 定义、所有调用方、返回值约定，以及 reviewer 是否确认 optional ICE 与 OP-TEE/TZ 场景。

2. F2FS gc_merge：先画等待关系，再看修复

这条 patch 的价值很高，因为它不是单点 bug，而是一个跨路径等待环。写回路径把 data bio 缓存在内部队列里，然后进入 f2fs_balance_fs() 唤醒 GC 并等待；truncation 路径锁 folio 并等待 writeback 清除；GC 线程又试图锁同一个 folio。结果是三个路径互相等待。

Wait-for graph

Kworker writeback

folio_start_writeback 后缓存 bio，进入 f2fs_balance_fs()，等待 GC 线程完成。

GC thread

foreground GC 被唤醒，move_data_page 时需要锁目标 folio。

Truncator

持有或等待 folio lock，并等待 writeback 标志被清除。

Kworker → waits for GC thread
GC thread → waits for folio lock
Truncator → waits for writeback completion
writeback completion → stuck behind cached bio not submitted

修复逻辑

patch 让 GC_MERGE 路径像 NOGC_MERGE 路径一样，在等待前提交 cached data bios 和 IPU writes。也就是说，修复不是“加锁”，而是打破等待环：先把会释放 writeback 条件的 I/O 推出去，再等待 GC。

阅读检查点
  • 确认提交 cached bio 的时机不会破坏 write ordering。
  • 确认 GC_MERGE 和 NOGC_MERGE 行为收敛是否合理。
  • 注意 Cc stable 和 Fixes tag，它说明这是稳定性修复，不只是优化。
fs/f2fs/segment.c

3. iomap：REQ_POLLED 不是 REQ_NOWAIT

这条 patch 适合作为性能调优基础知识。REQ_POLLED 表示调用方希望通过轮询完成 I/O，减少中断路径影响；REQ_NOWAIT 表示提交路径不能睡眠。二者目标不同，不能因为都和低延迟相关就绑定在一起。

概念含义读 patch 时的判断
REQ_POLLEDcompletion 侧通过 polling 等待结果。关注完成路径、CPU 消耗、中断规避和尾延迟。
REQ_NOWAITsubmit 侧不允许阻塞。关注页锁、块映射、元数据分配失败时如何返回。
iomap direct write文件系统写入不是简单 bio 操作。不能只重试 bio；错误恢复可能需要重试更大的文件系统操作。
Two different axes
Submit pathmay need mapping, locks, metadata
Completion pathpolling can reduce interrupt overhead
Retry modelfile writes are not simple bio-level retries

对你的 UFS 性能实验来说，这条提醒很实用：记录 hipri/polling/nowait 时要分开。否则你可能把“轮询完成带来的收益”误判成“非阻塞提交语义”的收益。

4. UFS tracepoint：可观测性也要守住热路径成本

这条 patch 把已经被 trace_foo_enabled() 包住的调用点改成 trace_call__foo()，避免进入 trace 宏后再次执行 static branch 判断。它不是功能修复，但展示了内核在热路径上的标准：未启用 tracing 时，观测设施应尽量接近零成本。

阅读检查点

逐个确认替换点外层确实有 enabled guard；确认参数求值和上下文没有变化；关注 reviewer 是否要求这类改动在其他子系统保持一致。
