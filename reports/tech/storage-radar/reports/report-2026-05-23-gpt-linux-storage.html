1. UFS TX Equalization：高速链路参数不再只是 PHY 默认值

snapshot fact 本地事实

  • 系列条目包括：新增 UFS_HS_G6 / UFS_HS_GEAR_MAX、UFS core TX Equalization support、debugfs 参数、debugfs retrain、QCOM vendor hooks 和最终 ufs-qcom: Enable TX Equalization。
  • 另有 2026-05-01 的新系列 Add static TX Equalization settings support from DT，包含 DT binding 和 UFS core 支持，状态 new / needs review。

GPT interpretation 背景知识

UFS 进入更高 gear 后，链路稳定性不再只取决于“能否 link startup”。TX EQ 会影响眼图、反射、串扰和 BER。Linux 这类 patch 的重点不是把一个 magic number 写进 PHY，而是把 power mode negotiation、vendor vops、debugfs observability 和 retrain 组织成可 review 的内核语义。

UFS TX Equalization control loop

问题怎样形成

如果 TX EQ 只在 vendor driver 里作为一次性初始化，core 无法知道它与 gear、lane、PAM/NRZ 模式和 power mode 的关系。等线上出现间歇 CRC、link recovery、hibern8 resume 后不稳，日志只能看到 UIC error，而看不到参数来源和 retrain 行为。

Review 时看什么

  • core enum 和 gear 上限是否改变所有判断分支，而不是只让新 gear 编译通过。
  • debugfs 是否只是实验接口，还是会影响生产路径状态机。
  • retrain 前是否暂停 command processing，避免链路参数改变时仍有 SCSI command 在飞。
  • DT static settings 是否是硬件描述，不应变成 board policy 或调参笔记。

2. classic polling：polling 路径不能重新依赖中断唤醒

snapshot fact 本地事实

  • 条目：blk: optimization for classic polling，文件为 fs/block_dev.c、fs/direct-io.c、fs/iomap.c，风险标签 performance，signal 为 needs review。
  • snapshot excerpt 说明：当 need_resched() 为真时，把 task state 设为 TASK_RUNNING；旧行为可能让 polling task sleep，并依赖 interrupt wakeup。
  • source caveat：该 snapshot 中此条日期为 2083-05-30，明显是源数据异常。本节只采用 excerpt 中保存的语义，不把日期当作事实。

GPT interpretation Linux storage 路径

classic polling 常见于 direct I/O 或 iomap 路径里等待 request completion。它的目标是用 CPU 主动轮询换取更低 tail latency。如果等待者在调度点把自己放进 sleepable 状态，又没有可靠的 completion wakeup，polling 就会退化成“本来想不等中断，最后又等中断”。NVMe interrupt coalescing 时，这种尾延迟会被放大。

Patch 的修复逻辑

从 excerpt 看，修复不是让 polling 永不让出 CPU，而是在必须 reschedule 时确保 task 仍是 runnable 语义，不把完成依赖转交给中断唤醒。review 的关键是确认这不会制造 busy-spin starvation，也不会破坏已有 completion/wakeup 契约。

对 UFS/固件的启发

UFS 没有 NVMe poll queue 的同构路径，但同样有“host 以为自己在低延迟等待，设备侧却因为中断、clock gating、runtime PM 或 completion aggregation 拉长尾巴”的问题。固件侧性能 debug 不应只看平均 latency，要把 host 等待状态、IRQ coalescing、doorbell/completion 可见性放在同一张时间线上。

3. bio_map_user_iov：错误路径要按 ownership 回滚，而不是按“正常路径印象”回滚

snapshot fact 本地事实

  • 条目：block: fix pages array leak in bio_map_user_iov error path，影响 block/blk-map.c，signal 为 check: fail。
  • excerpt 说明：当 nr_vecs > UIO_FASTIOV 导致 pages == NULL 时，want_pages_array() 会通过 kvmalloc_array() 分配 pages array。
  • 随后 iov_iter_extract_pages() 如果因为 pin_user_pages_fast() 失败返回 bytes <= 0，代码跳到 out_unmap，遗漏释放动态 pages array，kmemleak 可见。
  • 触发路径来自用户态在 bsg 设备上发 SG_IO v4，大 din_xfer_len 加无效 din_xferp。

GPT interpretation 背景知识

bio_map_user_iov() 是用户态 buffer 进入 block I/O 的危险边界：它既要 pin 用户页，又要构造 BIO vector，还要处理 fast array 和动态 array 两种存储。最容易出错的不是成功路径，而是“对象已经分配，但页 pin 还没成功”的半初始化状态。

4. ublk：用户态 block driver 的 bug 本质是 command lifetime bug

snapshot fact 本地事实

  • ublk: fix use-after-free in ublk_cancel_cmd()：ublk_reset_ch_dev() 清 io->cmd 与 ublk_cancel_cmd() 并发，后者可读到 stale pointer 并传给 io_uring_cmd_done()，signal 为 check: fail，风险 critical。
  • 修复摘要：ublk_cancel_cmd() 在 cancel_lock 下读取并清空 io->cmd，在锁外完成；ublk_reset_ch_dev() 持锁执行 ublk_queue_reinit()。
  • ublk: reject max_sectors smaller than PAGE_SECTORS：queue_limits 转换后丢失旧的 clamp 行为，导致 blk_validate_limits() WARN_ON_ONCE；修复为参数验证阶段直接拒绝。

GPT interpretation 问题怎样形成

ublk 把 block request 的执行委托给用户态进程，内核对象、io_uring command 和用户态 daemon 共同持有“一个 I/O”的生命周期。reset、cancel、reinit、completion 这些路径如果没有单一 owner 规则，就会出现 stale command pointer 或重复 completion。

Review 时应该看什么

  • cancel_lock 是否覆盖所有写 io->cmd/io->flags 的路径。
  • 锁外调用 io_uring_cmd_done() 是否避免锁内 completion 的反向调用风险。
  • reset 后 request 是否一定被完成或取消，不留下用户态永远等待的 command。
  • queue_limits 参数验证是否和 block layer invariant 对齐，而不是依赖旧 helper 的隐式 clamp。

对 UFS 稳定性的启发

UFS host driver 里的 tag、LUN reset、abort task、MCQ completion 也有同样的 lifetime 问题：清 slot、完成 request、通知上层和恢复队列必须有明确 owner。任何“先清全局指针，再让另一路径拿旧指针完成”的设计，都应该被当作潜在 UAF 审查。

GPT interpretation 为什么 UFS 开发者也该读

MMC/SDHCI 的 tuning 语义比 UFS TX EQ 更显性：调电压、换 pinctrl、开时钟、扫 delay window、选择最佳采样点。这是把模拟信号问题抽象成内核状态机的清晰案例。UFS 的 PHY calibration、PA parameters、TX EQ、hibern8 resume 稳定性，本质上也要回答同样的问题：何时调、调什么、失败怎么回退、日志如何证明窗口存在。
