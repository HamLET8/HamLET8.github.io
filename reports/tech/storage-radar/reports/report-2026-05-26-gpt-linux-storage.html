1. UFS QCOM PHY：把 init、power 和 gear 切换从隐含顺序改成显式状态

snapshot fact 本地事实

  • snapshot 记录了 Vladimir Oltean 的 Split Generic PHY consumer and provider API v8 系列，其中 UFS 相关条目状态为 not-applicable，说明本地 snapshot 不把它们当作当前待合入 UFS patch。
  • UFS 相关标题包括：ufs-qcom: call phy_init() before phy_power_on()、ufs-qcom: make use of QMP PHY dynamic gear switching ability、ufs-qcom: keep separate track of PHY power state、ufs: exynos: use dedicated API for updating PHY bus width。
  • 风险标签集中在 platform 和 reset；snapshot 没有保存完整 diff 或 context excerpt。

GPT interpretation 背景知识

UFS host driver 与 PHY provider 的关系很容易被写成“probe 里按经验调用几个函数”。但高速链路下，PHY init、power on、lane/bus width、gear 切换、runtime suspend/resume 和 reset recovery 都是状态机边。如果内核只靠调用顺序隐含状态，某个 error path 或 PM path 很容易重复 power-on、漏掉 init，或者在 PHY 已断电时做 gear 切换。

UFS PHY state ownership after API split

Patch 的修复逻辑

GPT interpretation 从标题看，这组 patch 的价值不是某个 SoC 多支持一个 helper，而是把 Generic PHY consumer/provider API 的职责切开：consumer 不应直接假设 provider 的内部上电状态；provider 能表达 dynamic gear switching 能力；UFS QCOM driver 必须在 phy_power_on() 前完成 phy_init()；driver 还要单独记录 PHY power state，避免 link state 和 power state 被混成一个布尔值。

Review 时看什么

  • 所有 PM 和 reset path 是否遵循同一条 init → power_on → configure gear → link startup 序列。
  • dynamic gear switching 是否只在 provider 明确支持时使用，失败后是否回到可诊断状态。
  • 新增状态变量是否真正防止重复 power-on/off，而不是制造另一个可能失真的 shadow state。
  • Exynos/QCOM 等 host 的 bus width 或 gear API 是否保持同一层级的抽象，不把 provider 私有寄存器泄漏给 core。
对 UFS 固件开发的启发对应检查点
固件看到的 link startup failure 可能源自 host PHY 状态顺序错误。抓日志时同时记录 ref clock、PHY power、PA gear/lane、hibern8 状态，而不是只看 UIC error code。
动态 gear 切换不是性能开关，它改变 PHY/provider 与 host/core 的契约。确认 gear switch 前后是否 quiesce I/O，是否有失败回滚和 retrain 证据。

2. iSCSI target ECDB AHS：命令解析器必须先验证长度，再产生 CDB 语义

snapshot fact 本地事实

  • 条目：scsi: target: iscsi: validate ECDB AHS length，文件为 drivers/target/iscsi/iscsi_target.c，状态 new，signal 为 needs review。
  • snapshot excerpt 指出 iscsit_setup_scsi_cmd() 处理 Extended-CDB Additional Header Segment 时没有先约束 AHSLength，尽管附近已有 long-standing FIXME。
  • 恶意 Command PDU 在 login 后携带 hlength=1、ahstype=ISCSI_AHSTYPE_CDB、ahslength=0，会走到 kmalloc(0 + 15)、随后 16 字节 memcpy 到 15 字节声明目标，非 FORTIFY 情况下还可能进入 size_t 下溢的第二次 memcpy。
  • excerpt 明确说触发者是完成 login 的 initiator；作者没有声称 RCE、LPE 或 controlled write。

GPT interpretation Linux storage 路径

这是网络协议输入进入 SCSI command 的边界。iSCSI target 收到 PDU 后要生成内核内部的 CDB 表示，再交给 target core。问题不在 SCSI 命令本身复杂，而在 Additional Header Segment 的长度字段同时影响 allocation size、copy size 和指针偏移。一旦这些单位没有先闭合，后面的 target/SCSI 语义都已经建立在坏内存对象上。

Patch 的修复逻辑

snapshot fact excerpt 说修复是在任何 dereference 和 allocation 之前验证：socket 中收到的 AHS 区域至少包含 4 字节 iscsi_ecdb_ahdr，ECDB AHS 的 AHSLength 至少为 1，并且后续长度换算不会造成短分配或下溢。GPT interpretation 这就是把 parser 分成两阶段：先证明输入 buffer 结构完整，再把它转成内核 CDB 对象。

Review 时看什么

  • hlength、ahslength、字节数和 32-bit word 数是否统一换算，避免 off-by-one。
  • 验证是否发生在所有 dereference、allocation、memcpy 之前。
  • 错误返回是否清理半初始化 command，不把坏 CDB 继续交给 target core。
  • FORTIFY 触发只是症状；非 FORTIFY 下的第二次 memcpy 下溢才是必须一起覆盖的语义漏洞。
对 UFS/固件的启发可迁移规则
UFS 也有 query request、descriptor、RPMB frame、vendor command 等外部输入。先验证长度字段和对象边界，再生成内部命令对象；不要让 copy helper 同时承担验证和构造。
固件 parser 的“只在 debug/厂内可达”经常会变成 host/tooling 可达。把可达性、认证状态、最短长度、最大长度和单位换算写成表，而不是散在 memcpy 旁边。

3. SCSI host state：锁上下文不是注释，是状态机的一部分

snapshot fact 本地事实

  • snapshot 记录 Bart Van Assche 的 Enable lock context analysis for the SCSI subsystem v2 系列，至少包括 [55/56] scsi: core: Enable lock context analysis 和 [56/56] scsi: core: Protect host state changes with the host lock。
  • 这两条状态为 new，风险标签 locking，signal 为 needs review；snapshot 没有保存具体 diff 或 context excerpt。
  • 同一系列还覆盖具体 low-level driver，例如 sym53c8xx_2 的 lock context analysis。

GPT interpretation 背景知识

SCSI host state 影响 scan、queue command、error handling、device removal、runtime PM 和 transport recovery。它不是一个普通 enum 字段。如果 state change 可以在不同锁上下文中发生，review 很难证明 queue 是否仍能提交命令、error handler 是否能推进、remove 是否会和 timeout/reset 交错。

Wait-for and mutation graph around host state

Patch 的修复逻辑

GPT interpretation 从标题看，系列先让 SCSI subsystem 能表达并检查 lock context，再把 core host state change 收敛到 host lock 下。这个顺序很重要：如果只改某个赋值点，reviewer 不知道别的路径是否还裸写 state；先启用分析再保护核心状态，才像是在建立全局不变量。

Review 时看什么

  • 所有 host state writer 是否都在相同锁下，尤其是 error handler、remove、scan 和 PM path。
  • 新增 lockdep/context annotation 是否覆盖 driver callback，而不只是 SCSI core 文件。
  • 锁顺序是否与 request queue freeze、device lock、transport lock 形成循环等待。
  • state reader 是否需要 lock、RCU、READ_ONCE 或已经由上层 serialization 保证。
对 UFS 稳定性的启发Review 落点
UFS 有 ufshcd_state、link state、pm state、eh flags、MCQ enable state 等多套状态。每套状态都要能回答：谁能写、持什么锁写、读者是否允许看到旧值、状态变化是否唤醒等待者。
error handler 常常是“看起来能修复所有状态”的路径。真正要审的是 error handler 和正常 I/O path 是否共享同一套可证明的锁顺序。

4. blk-mq：fast path 的小优化只有在 request 和 queue 状态不变量成立时才安全

snapshot fact 本地事实

  • [PATCHv2] blk-mq: check for stale cached request in blk_mq_submit_bio，提交者 Keith Busch，状态 new，signal 为 check: success。
  • [v6,1/1] block/blk-mq: use atomic_t for quiesce_depth to avoid lock contention on RT，提交者 Ionut Nechita，风险标签 locking，signal 为 check: success。
  • snapshot 没有保存这两条的完整 diff 或 context excerpt，因此本节将具体路径解释标记为 GPT 推理。

GPT interpretation 背景知识

blk-mq 的性能来自两个方向：submit path 尽量少分配、少锁、复用上下文；control path 则用 freeze/quiesce 等机制让 queue 在 resize、reset、limits change 或 driver teardown 时停下来。这两者天然紧张：fast path 想缓存 request，control path 想证明没有旧 request 混入新 queue 状态。

Patch 的修复逻辑

GPT interpretation stale cached request patch 的核心应是：复用 request 前先证明它没有被 queue reconfiguration、context switch、tagset change 或 completion path 变成“属于旧世界的对象”。quiesce_depth patch 则是在不改变 quiesce 嵌套计数语义的前提下，用 atomic_t 降低 RT 内核上的锁竞争。前者保护对象身份，后者保护状态计数。

Review 时看什么

  • stale 判断依据是否足够强：queue、hardware context、tag、rq flags 和 bio merge 状态是否都在同一代。
  • 缓存命中失败后是否回到普通 allocation path，而不是静默提交错误 request。
  • quiesce_depth 改成 atomic 后，等待者是否仍有正确 barrier 和 wakeup 语义。
  • RT 场景的优化是否改变非 RT 内核行为，尤其是嵌套 quiesce 和并发 unquiesce。
对 UFS/性能调试的启发观察方式
UFS MCQ 和 blk-mq 都高度依赖 tag/request 身份。看超时或乱序完成时，先确认 request 是否属于当前 queue generation，而不是只看 device response。
RT latency 问题可能来自控制路径锁，而不是设备慢。把 quiesce/freeze/reset 计数和 submit/completion latency 放在同一时间轴。

5. smartpqi reset recovery：恢复路径要等待协议重新收敛，而不是只等待固定延时

snapshot fact 本地事实

  • 系列：scsi: smartpqi: fix PCIe hot reset recovery，两条状态 new，signal 为 needs review。
  • 第一条：smartpqi: add pci_error_handlers for bus reset recovery，风险标签 reset。
  • 第二条：smartpqi: increase SIS ctrl ready resume timeout to 180s，风险标签 timeout、reset、power-management。
  • snapshot 没有保存完整 diff 或 context excerpt。

GPT interpretation 背景知识

smartpqi 是 SCSI RAID/HBA 类驱动。PCIe hot reset 后，内核侧要经历 PCI error recovery、控制器重新 ready、队列恢复、SCSI host 重新接收命令等多个阶段。把 timeout 从较短值增加到 180s 不是简单“等更久”，而是承认某些控制器在 reset 后需要更长时间完成 firmware/adapter 侧初始化。

Patch 的修复逻辑

GPT interpretation 第一条通过 pci_error_handlers 把 smartpqi 接入标准 PCI error recovery 框架，让 bus reset 不再只是外部事件。第二条增加 SIS controller ready 的 resume timeout，使 driver 在恢复阶段等待控制器真正 ready，而不是过早宣告失败或让上层 SCSI 看到半恢复设备。

Review 时看什么

  • error handler 回调是否正确 freeze/stop I/O，避免 reset 中还有 command 进入硬件。
  • 180s timeout 是否只作用在 reset/resume recovery，不拖慢正常 probe 的失败反馈。
  • ready wait 是否检查明确硬件状态位，而不是盲 sleep。
  • 超时失败路径是否清理队列、tag、SCSI device state，并向上层返回可恢复错误。
对 UFS 固件/稳定性的启发迁移视角
UFS reset、power collapse resume、hibern8 exit 后也有“设备还活着但协议未收敛”的窗口。timeout 不是越短越好，也不是越长越稳；要对应可观测状态位、event log 和恢复阶段。
host 过早恢复队列会把固件初始化噪声放大成随机 I/O timeout。在恢复流程中明确区分：链路 up、device ready、LUN ready、queue ready。
