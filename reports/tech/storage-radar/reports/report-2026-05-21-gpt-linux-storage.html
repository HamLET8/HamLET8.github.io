1. UFS hibern8 notify：失败路径也必须进入状态机

snapshot fact 本地事实

  • 条目：[v5] scsi: ufs: core: call hibern8 notify when hibern8 cmd failed。
  • 模块：UFS，状态：new，信号：needs review，风险标签：power-management。
  • 来源：Patchwork linux-scsi，提交者 Fang Hongjie，时间 2026-05-02。

GPT interpretation 为什么值得读

标题说明修复点在 hibern8 命令失败后的 notify 行为。UFS core 的 hibern8 不是单纯发一条 UIC command，它牵涉 runtime PM、link state、host controller vendor hooks 和可能的 PHY/firmware side effect。成功路径通知平台很自然，失败路径更容易被遗漏，但失败路径恰恰决定 recovery 能否把 host、device、PHY 和固件状态重新对齐。

UFS hibern8 state machine, expected notify symmetry

Review 时应该看什么

  • notify 回调是否在 success 和 failure 两条路径上保持语义对称。
  • failure notify 是“命令失败后的清理”还是“状态改变通知”，两者不能混用。
  • 是否会改变 runtime PM、clock gating、hibern8 enter/exit retry 的已有顺序。
  • v5 说明该 patch 已多轮迭代，优先看每版变化和 reviewer 关心的边界。

对 UFS 固件开发的启发

固件接口也要把失败状态当作一等状态。只在成功路径更新影子状态，容易造成 host 以为 link/PHY 仍处于旧状态，而固件已经进入 abort/recovery 分支。调试 hibern8 问题时，日志要能区分 command issue、device response、host recovery 和平台 notify 四个阶段。

GPT interpretation 背景知识

3. loop_queue_rq 竞态：block freeze 不会自动覆盖 driver 的私有异步队列

snapshot fact 本地事实

  • 条目：loop: Fix NULL pointer dereference by synchronizing lo_release and loop_queue_rq。
  • 影响文件：drivers/block/loop.c，状态 new，信号 check: success。
  • snapshot excerpt 说明 crash 发生在 lo_rw_aio() 访问已被 __loop_clr_fd() 清空的 lo->lo_backing_file。
  • excerpt 说明修复引入 SRCU-based synchronization，并在 device release 时显式 drain workqueue。

GPT interpretation 问题怎样形成

block layer 的 freeze 主要约束 request queue 的可见状态，但 loop driver 在 queue_rq 之后还把请求交给自己的 workqueue。只要一个 request 已经通过 loop_queue_rq() 的状态检查，还没真正排进或执行私有 work，就存在 teardown 先清 lo_backing_file、异步 work 后访问的窗口。

Race window across block queue and loop private workqueue
submit bioblock layer dispatches request
→
loop_queue_rqstate check passed, request not fully owned by workqueue yet
→
loop workqueuelo_rw_aio later dereferences lo_backing_file
release pathlo_release / __loop_clr_fd starts teardown
gapfreeze cannot see every private async handoff
fix logicSRCU guards handoff and drain waits before freeing backing state

Review 时应该看什么

  • SRCU read-side critical section 是否覆盖从 state check 到 work enqueue 的窗口。
  • release path 的 drain 是否等待所有已越过边界的 work，而不仅仅是 queue freeze。
  • teardown 对 lo_backing_file 清空、引用释放和 request completion 的顺序是否可证明。

对 UFS/固件稳定性的启发

任何“主队列到私有队列”的交接都是竞态高发区。UFS 中类似场景包括 error handler、async scan、reset work、runtime PM work 和 completion work。仅靠上层状态位或 freeze 不够，必须明确谁拥有对象引用，release 等待的是哪个生命周期边界。

4. mpi3mr BSG SGL 越界：用户态长度和 kernel buffer 长度必须在同一个语义域

snapshot fact 本地事实

  • 条目：scsi: mpi3mr: fix out-of-bounds write in mpi3mr_bsg_build_sgl()。
  • 影响文件：drivers/scsi/mpi3mr/mpi3mr_app.c，风险标签 critical，信号 needs review。
  • snapshot excerpt 说明 RAIDMGMT_CMD path 将 kern_buf_len 设为 chain buffer size，但 bsg_buf_len 仍保留用户传入值。
  • 当 bsg_buf_len > kern_buf_len 时，mpi3mr_bsg_build_sgl() 中 unsigned subtraction 使 available_sges underflow，导致越界写；修复是将 bsg_buf_len clamp 到 kern_buf_len。

GPT interpretation 背景知识

BSG 是用户态管理工具向 SCSI HBA/RAID 控制器发送 vendor 或 management command 的入口。它看起来不是普通数据 I/O，但本质仍会构造 scatter-gather 描述符交给控制器或 firmware。这里的安全边界是：用户态声明的长度、kernel 预分配的 chain buffer、firmware 可消费的 SGE 数量，三者必须收敛到同一个上限。

Length mismatch creates SGL overwrite
User BSG requestbsg_buf_len can be larger than real chain buffer
RAIDMGMT_CMD pathkern_buf_len set to chain buffer size
SGL builderunsigned subtraction underflows available_sges
Fixclamp bsg_buf_len to kern_buf_len before SGL construction
边界错误模式review 检查点
用户态 ABI用户长度被误当作已验证的 kernel 内部长度。所有从 ioctl/bsg 进入的 length 都需要最早归一化。
kernel chain buffer预分配 buffer 比用户请求短，SGL 计算仍按用户请求走。对称检查 CMD 和 RESP 两条路径，避免只有一边 clamp。
firmware command错误 SGL 可能把管理命令变成 kernel memory overwrite。对固件命令路径应用和数据 I/O 同等级的边界校验。

对 UFS 固件开发的启发：UFS vendor command、RPMB、FFU、query request 等管理路径也常被当作“控制面”。控制面同样需要严格的 host buffer 长度、device payload 长度和 descriptor 数量一致性校验。

5. block-as-NVMEM：把 eMMC 分区当配置 EEPROM 时，lifetime 是第一问题

snapshot fact 本地事实

  • 系列：Support for block device NVMEM providers。
  • 核心 patch：[v2,4/8] block: implement NVMEM provider，影响 block/blk-nvmem.c、drivers/nvmem/core.c 和 include/linux/nvmem-consumer.h。
  • snapshot excerpt 说明嵌入式设备常用 eMMC 分区存 MAC 地址和 Wi-Fi calibration EEPROM data。
  • excerpt 还说明新增 nvmem_dev() accessor，让 provider private data 的释放延后到 consumers release cells 之后。

GPT interpretation 问题怎样形成

这不是“从 block device 读几个字节”那么简单。NVMEM consumer 可能在 Wi-Fi、Bluetooth 或网络驱动里长期持有 cell 引用，而 block provider 可能随设备移除、分区重扫、devm teardown 被释放。private data 如果跟 provider device 生命周期绑定得太短，consumer 释放 cell 时就可能访问已释放对象。

Patch directionregister devm action on nvmem device lifetime
Review pointensure block teardown and NVMEM cell release order is explicit

Review 时应该看什么

  • cell read 路径是否处理 block device disappearance、I/O error、partition size mismatch。
  • DT 中引用分区的语义是否稳定，避免把易变分区布局当硬 ABI。
  • NVMEM provider private data 的释放是否跟 nvmem device 而不是 parent block device 粗略绑定。

对 UFS/固件开发的启发

UFS 设备也可能承载出厂校准、板级配置或 provisioning 数据。把这些数据暴露给内核其他子系统时，最重要的不是读取 API 有多薄，而是断电、热移除、分区变更、consumer late release 时是否仍有清晰 lifetime 规则。
