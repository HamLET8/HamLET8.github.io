Technical Deep Dive

F2FS large folio read：为什么“错误返回前提交 pending bio”是进展性要求

这份报告分析 patch f2fs: fix missing read bio submission on large folio error。它修复的是一个 read path 进展性问题：f2fs_read_data_large_folio() 可以让一个 read bio 跨多个 readahead folio 积累请求；如果后续 folio 在加入 bio 之前出错，旧路径可能直接返回，导致前面已经加入 bio 的 folio 永远没有 I/O completion，读者一直等 locked folio。

0. Executive Summary

结论先行：folio 已经交给 bio 后，函数就背上了提交责任

这条 patch 的关键是一个很朴素的不变量：只要某些 folio 已经被加入 read bio，函数在返回前就必须保证这个 bio 被提交，或者用等价方式完成这些 folio。旧代码在 large folio readahead 路径中没有维持这个不变量。它可能在处理后续 folio 时遇到错误，此时当前 folio 还没加入 bio，folio_in_bio 为 false，于是错误路径会结束当前 folio 并直接返回。问题是，前面 folio 对应的 bio 可能已经存在，只是还没有到提交点。直接返回会让这个 pending bio 丢在函数局部变量里，前面 folio 没有 completion，也就不会被 folio_end_read() 解锁。

修复方式是把错误路径重新导向统一出口：当前 folio 如果没有进 bio，就可以直接 folio_end_read()；但如果有 pending bio，必须走到 out，调用 f2fs_submit_read_bio()。同时，出错后不再继续消耗更多 readahead folio；只有当前 folio 确实加入了 bio 时，才在错误返回前等待并清理它的 uptodate 状态。

关键不变量：read path 中的 locked folio 必须有明确完成路径。完成路径可以是同步 folio_end_read()，也可以是 bio completion 调用 folio_end_read()。不能存在“folio 已经交给 bio，但 bio 没提交”的中间状态逃出函数。
Problem in one screen

→
bio pendingbio 还在函数局部变量里，没有提交。
→
folio B 出错尚未加入 bio，错误路径触发。
旧路径直接返回结束 folio B，但没有提交 folio A 的 bio。
→
folio A 无 completion没有设备 I/O，也没有 end_read。
→
读者长期等待看起来像 read hang 或 page cache 卡住。

1. Patch Facts

事实层：这个 patch 只改一个函数，但修的是路径不变量

patch 是 v2，当前 Patchwork 状态为 new。它只修改 fs/f2fs/data.c 中的 f2fs_read_data_large_folio()。diff 很小：初始化 folio_in_bio，调整错误路径，让 ret 情况跳到统一出口提交 pending read bio，并且只有当前 folio 真的在 bio 中时才执行等待和清 uptodate 的收尾逻辑。

提交信息指出触发条件：f2fs_read_data_large_folio() 可能把一个 read bio 跨多个 readahead folio 保留下来。如果后续 folio 在任何 block 加入 bio 之前出错，旧错误路径会在结束当前 folio 后直接返回。这样前面已经加入 bio 的 folio 没有提交到 block layer，也就不会收到 read completion。

变化点旧行为新行为意义
folio_in_bio局部变量未在声明处初始化，依赖每轮处理重置。声明时初始化为 false，并在每个 folio 开始时重置。让错误路径始终有明确状态。
错误返回当前 folio 未进 bio 且 ret 非零时直接返回。当前 folio可结束，但错误要跳到统一 out。确保 pending bio 被提交。
readahead某些错误路径可能继续取下一个 readahead folio。出错后停止消耗更多 folio。避免把错误扩大成更多锁定 folio。
错误清理只要 ret 就锁当前 folio 并清 uptodate。只有 ret && folio_in_bio 才做。避免处理一个已经同步结束、未交给 bio 的 folio。
解释边界：以上来自 patch 和本地源码。下面关于 large folio、readahead 和 UFS 的部分是工程解释，用来帮助建立可迁移的分析模型。

2. Read Path Background

先把 Linux read 路径和 F2FS read bio 放在一起

普通 buffered read 从 VFS 进入 page cache。如果目标 folio 不在 page cache，或者还不是 uptodate，MM 会通过文件系统的 read_folio/readahead 方法发起读。F2FS 的读路径会把逻辑 block 映射成物理 block，构造 read bio，然后提交给 block layer。bio completion 之后，F2FS 在 read end_io 中递减读计数，并在所有相关子页完成后调用 folio_end_read()，把 folio 标为 uptodate 或失败，并唤醒等待这个 folio lock 的读者。

large folio 让这条路径更复杂。传统小页时代，一个 folio 通常对应一个 page；large folio 可能覆盖多个 page-sized blocks。F2FS 需要跟踪这个 large folio 中有多少子页读请求尚未完成。因此源码里有 f2fs_folio_state，其中 read_pages_pending 记录这个 large folio 尚未完成的 read page 数。只有 pending 归零，end_io 才能真正结束整个 folio。

Buffered read to F2FS read bio
filemap_read发现 page cache 缺页或 folio 未 uptodate。
readahead批量取多个 folio，交给文件系统。
f2fs map逻辑 block 映射到物理 block。
read bio多个 block/folio 可以合并。
bio submit请求进入 block/UFS 路径。
end_readcompletion 唤醒等待 folio 的读者。

这条 patch 修的不是映射错误，也不是设备读失败，而是“bio 构造阶段出错时是否仍然履行已接收 folio 的 completion 责任”。它发生在设备请求生成之前，属于 submit 前进展性问题。

3. Large Folio Model

large folio 让“当前 folio”和“pending bio”不再是一一对应

在 f2fs_read_data_large_folio() 中，函数可能处理单个 folio，也可能处理 readahead 控制器提供的一批 folio。每个 folio 内部又可能包含多个 page-sized block。函数会逐 block 调用 f2fs_map_blocks()，验证物理地址，再把对应范围加入 read bio。为了减少 I/O 数量，它不会每添加一个 block 就立即提交 bio，而是尽量让相邻、可合并的读请求进入同一个 bio。

这正是 bug 的基础：pending bio 是函数级状态，不是当前 folio 私有状态。当前 folio 出错时，folio_in_bio 只说明“当前 folio 是否已经加入 bio”，并不能说明“函数里是否有之前 folio 留下的 pending bio”。如果错误路径只根据当前 folio 状态决定是否直接返回，就可能遗忘前面的 pending bio。

Current folio vs pending bio
bio函数局部状态，可能包含多个 folio 的 block。
folio_in_bio只表示当前 folio 是否加入 bio。
readahead folios函数循环逐个消费，可能跨 folio 合并 bio。
completion duty只要 folio 进 bio，必须提交 bio 或完成 folio。

所以这条 patch 的核心不是 large folio 自身，而是 large folio/readahead 让状态维度从“一个 folio”变成“当前 folio + pending bio + readahead 队列”。错误处理必须同时考虑这三个维度。

4. Old Error Flow

旧错误路径：当前 folio 被处理了，前面的 bio 被忘了

旧代码在 err_out 中，如果当前 folio 没有加入 bio，会调用 folio_end_read(folio, !ret)。如果 ret 非零，则直接返回错误。这个逻辑对单 folio 情况看起来合理：当前 folio 没交给 bio，就同步结束它并返回。但对 readahead 批量场景不够，因为函数里可能有一个已经包含前面 folio 的 bio。

// 解释性伪代码，表示旧错误路径
for each readahead folio:
  map blocks
  if mapped:
    add current folio blocks to bio
    folio_in_bio = true

err_out:
  if current folio not in bio:
    folio_end_read(current)
    if ret:
      return ret   // pending bio for previous folios is not submitted

这会产生一个非常隐蔽的等待问题。前面的 folio 已经被加入 bio，所以它们不会在当前函数中直接 folio_end_read()。它们依赖 bio completion 来完成。但是 bio 还没有提交，completion 不可能发生。于是读者卡在 folio lock 或 uptodate 等待上，看起来像底层 I/O 没返回，实际上请求根本没进入设备。

步骤状态为什么危险
处理 folio AA 的 block 被加入 read bio。A 的完成责任转移给 bio completion。
bio 尚未提交为了合并后续读，bio 保留在局部变量。这是合法优化，但要求返回前提交。
处理 folio BB 在加入 bio 前遇到映射或校验错误。folio_in_bio 对 B 为 false。
旧路径返回B 被同步结束，函数返回错误。A 的 bio 丢失提交机会，A 无法 completion。

5. Bio Submit Boundary

bio 提交边界：从“构造了请求”到“请求能完成”

这条 patch 和之前 GC merge 报告有一个共同主题：不能把“构造了 bio”和“提交了 bio”混为一谈。F2FS 出于性能考虑，会延迟提交 bio 以获得更好的合并效果。这个设计本身没有问题，但它带来一个协议要求：一旦函数决定返回，就必须检查有没有 pending bio。否则 page cache 层看到的是 folio 仍然 locked，block 层看到的是没有请求，设备层更看不到任何命令。

Submit boundary
folio lockedMM 等待读完成。
bio_add_foliofolio 的 block 被挂进 bio。
bio pending还只是函数内对象。
submit_bio进入 block layer。
folio_end_readcompletion 唤醒等待者。

从调试角度看，这类问题的典型特征是：上层读任务等待 page cache folio，但 block/UFS trace 没有对应请求。此时不能先怀疑设备 timeout；应该先检查文件系统是否把 pending bio 提交出去了。

6. Source Walkthrough

逐函数导读：从 filemap_read() 读到 f2fs_read_data_large_folio()

读这条 patch，建议从 Linux buffered read 的正常路径开始，而不是直接进入 diff。用户态 read 进入 filemap_read() 后，如果 page cache 中没有 uptodate folio，会触发同步读或 readahead。readahead 路径把一组 folio 交给文件系统，文件系统负责为这些 folio 建立 I/O，并在 I/O 完成后调用 folio_end_read()。这个调用非常关键，因为它会解锁 folio 并唤醒等待者。

F2FS 的 f2fs_mpage_readpages() 会根据 mapping 是否支持 large folio 选择进入 f2fs_read_data_large_folio()。large folio 路径要处理两层循环：外层逐个 readahead folio，内层逐个 page-sized block。每个 block 先通过 f2fs_map_blocks() 找物理地址，再验证 block address，最后加入 read bio。如果当前 bio 与新 block 不可合并，就先提交旧 bio 再分配新 bio。

源码位置要看什么和本 patch 的关系
mm/filemap.cfilemap_read() 如何触发 readahead 和 folio update。说明读者为什么会等待 folio 变 uptodate。
include/linux/pagemap.hreadahead_folio() 如何把 folio 从 readahead control 中取出。说明文件系统拿到 folio 后必须负责完成或提交它。
fs/f2fs/data.cf2fs_read_data_large_folio() 的外层 folio 循环和内层 block 循环。bug 就发生在这个函数的错误出口。
fs/f2fs/data.cf2fs_read_end_io() 如何减少 pending 计数并调用 folio_end_read()。说明 pending bio 不提交时，folio 为什么不会被唤醒。
include/linux/bio.hbio_add_folio() 的职责边界。加入 bio 不等于提交 bio；这个边界是 patch 的核心。

这里要特别注意 f2fs_read_end_io()。对于 large folio，end_io 会根据 bio 中完成的长度扣减 read_pages_pending，只有所有子页完成后才调用 folio_end_read()。所以如果 bio 没有提交，end_io 永远不会运行，pending 计数也不会减少，folio lock 也不会被释放。

Source reading route
filemap谁在等 folio uptodate。
readahead谁把 folio 交给 filesystem。
F2FS map谁把 logical block 映射成 physical block。
bio add/submit谁负责跨过 submit boundary。
end_io谁调用 folio_end_read()。

7. State Machine

把 read folio 看成状态机：每个锁定状态必须有出口

这个 bug 用状态机看会非常清楚。一个 folio 从 readahead control 中被取出时，通常处于 locked 状态；文件系统接手后，必须让它进入以下两个完成路径之一：如果没有加入 bio，则同步调用 folio_end_read()；如果加入 bio，则提交 bio，等待 bio completion 调用 folio_end_read()。错误路径也不能逃出这个规则。

Read folio completion state machine
Locked folioMM 把 folio 交给 F2FS。
No bio pathhole 或早期错误，F2FS 直接 end_read。
Bio pathfolio 被加入 bio，completion 负责 end_read。
Donefolio unlock，等待者继续。

旧代码的问题是制造了第三条非法路径：folio A 已经进入 bio path，但 bio 没有提交；随后 folio B 走了 no bio error path 并直接返回。这个返回把函数局部 bio 对象丢掉，使 folio A 卡在“已经交给 bio，但 bio 不会 completion”的状态。状态机没有出口。

patch 的新逻辑把非法路径折回合法路径。即使当前 folio 没进 bio，只要 ret 非零，也不能直接返回，而是先到统一出口提交 pending bio。这样前面 folio 仍然能沿 bio path 完成；当前 folio则沿 no bio path 同步完成。

状态机约束：
1. 从 readahead_folio() 取得的 folio，必须被 end_read 或 bio completion 完成。
2. bio_add_folio() 之后，folio 的完成责任转移到 bio completion。
3. 函数返回前，所有 pending bio 必须提交或被显式撤销。
4. 错误路径不能只看 current folio，也要处理 previous folios 留下的 pending bio。

6. Fix Logic

修复逻辑：错误路径也必须经过统一出口

patch 把 ret 错误从“直接返回”改成“跳到 out”。out 会调用 f2fs_submit_read_bio()，这保证了之前已经积累的 bio 能进入 block layer。当前 folio 如果没有加入 bio，仍然会被同步 folio_end_read()；但函数不会因此忘掉之前 folio 的 pending bio。

// 解释性伪代码，表示新错误路径
err_out:
  if current folio not in bio:
    folio_end_read(current)

  if ret:
    goto out

  if readahead has more:
    continue

out:
  submit pending read bio

  if ret and current folio was in bio:
    wait/clear current folio state

另一个重要变化是出错后停止继续消耗 readahead folio。readahead 是预测性优化，不是必须完成的用户请求集合。当前已经遇到错误时，继续从 rac 中取 folio 只会扩大锁定范围和清理复杂度。停止 readahead，让已经接受的 work 正确完成，是更稳的选择。

当前 folio未进bio

可以同步结束当前 folio，因为没有设备 completion 会负责它。

之前 folio已进bio

必须提交 pending bio，否则之前 folio 永远等不到 completion。

当前 folio已进bio且出错

提交后需要等待并清理当前 folio 的 uptodate 状态。

7. Key Scenarios

三个场景帮你判断修复是否完整

第一个场景是单 folio read 出错，且没有 pending bio。此时新旧路径差别不大，当前 folio 没进 bio，直接 folio_end_read() 并返回错误即可。这个场景不是 bug 的重点。

第二个场景是 readahead 中前一个 folio 已经加入 bio，当前 folio 在加入 bio 前出错。这是 patch 的核心场景。新路径必须提交 pending bio，让前一个 folio 通过 completion 正常结束；当前 folio则同步结束并返回错误。

第三个场景是当前 folio 已经部分加入 bio 后出错。这时当前 folio 的完成责任已经转移到 bio completion，不能简单同步结束。patch 通过 ret && folio_in_bio 保护清理路径，只在当前 folio确实进了 bio 时执行等待和清 uptodate。

场景pending biocurrent folio in bio正确动作
单 folio 映射失败无否同步结束当前 folio，返回错误。
readahead 后续 folio 失败有，来自前面 folio否结束当前 folio，提交 pending bio，返回错误。
当前 folio 部分加入后失败有是提交 bio，等待/清理当前 folio 状态。

8. Review Checklist

读这条 patch 时应该检查什么

第一，检查 folio_in_bio 的生命周期。它必须在每个 folio 开始时重置，不能继承上一个 folio 的状态。patch 同时在声明处初始化和每轮重置，目的是让所有错误路径都能依赖这个变量。

第二，检查 bio 是否可能为 NULL。统一出口调用提交函数时，需要确认 f2fs_submit_read_bio() 能处理空 bio，或者路径上只有存在 bio 时才会提交。本地源码中这个提交函数会检查 bio 为空并直接返回，因此统一出口是安全的。

第三，检查 readahead 消费语义。出错后不继续消费 readahead folio 是合理的，但要确认已经从 readahead control 中拿出的当前 folio被正确结束。patch 在当前 folio未加入 bio 时调用 folio_end_read()，在当前 folio加入 bio 时让 completion 路径负责它。

第四，检查 large folio 的 pending 计数。对 large folio 来说，end_io 会根据 read_pages_pending 判断是否所有子页读完成。错误路径不能绕过这个机制去重复结束 folio，也不能让 pending 计数永远不归零。

重点 review 问题：这条 patch 的风险不在 diff 大小，而在状态组合。必须按“current folio state + pending bio state + readahead state”三维验证。

9. Debugging

如果要复现或验证，应该抓哪些信号

复现需要让 readahead 中前面的 folio 已经进入 bio，但后面的 folio 在加入 bio 前出错。错误来源可能是 f2fs_map_blocks() 返回错误，或者物理块地址校验失败。为了扩大窗口，可以用 fault injection 或构造损坏映射。验证时重点不是看设备是否返回慢，而是看 pending bio 是否被提交。

Trace plan
read path记录进入 f2fs_read_data_large_folio() 的 folio index。
bio state记录 bio_add_folio 和 f2fs_submit_read_bio。
folio state记录 folio_end_read 和 large folio pending。
block/UFS确认是否真的产生 read request。
建议观察点：
- f2fs_read_data_large_folio(): folio index, ret, folio_in_bio, bio != NULL
- f2fs_submit_read_bio(): submit count, bio sector, folio count
- f2fs_read_end_io(): read_pages_pending before/after
- filemap_read(): folio lock wait / uptodate retry
- block trace: 是否出现对应 read bio/request

如果旧路径触发，你会看到上层等待 folio，但 block trace 中没有对应的 read request。修复后，即使后续 folio 出错，之前 pending bio 也应该被提交，前面 folio最终会 completion。

10. UFS Implication

对 UFS 固件/性能分析的启发

这条 patch 是典型的 submit 前问题。用户态读卡住，未必意味着 UFS command timeout；可能是文件系统已经把 folio 放进 bio，但 bio 因错误路径没有提交。对 UFS 侧来说，这种卡顿非常迷惑，因为设备没有收到请求，自然也没有异常 completion。

做端到端性能分析时，建议把 read 卡顿分成三段：第一段是 page cache/MM/F2FS 构造 bio 之前；第二段是 bio 已提交但还在 block/UFS 队列中；第三段是设备 completion 后上层 end_io 收尾。这个 patch 属于第一段到第二段之间的断点：请求在文件系统局部变量里存在，但没有跨过 submit boundary。

UFS 看不到请求

不能据此判断设备空闲或正常，只能说明请求没有到设备层。

F2FS 需要补 trace

必须记录 bio 构造、bio 提交和 folio completion，才能解释 read hang。

这对你后续读 patch 很重要：任何涉及 readahead、large folio、bio merge、error path 的改动，都要问一个问题：错误返回前，所有已经接受的 folio 是否都有完成路径？
